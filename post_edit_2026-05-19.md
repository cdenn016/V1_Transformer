## F4.10 — Causal-mask wasted compute (opt-in toggle `causal_lower_triangle`)

**Goal:** when the caller passes a strict lower-triangular causal mask, the fused E-step kernel `_fused_attention_and_vfe_gradients_block_diag` allocates and computes (B, N, N, K)/(B, N, N, d, d) per-pair tensors for all N² pairs even though softmax masks the upper half to β=0. Restructure the per-block loop to compute only the lower-triangle pairs (j ≤ i, M = N(N+1)/2 pairs) and scatter into the existing dense accumulators.

**Investigation:** 5 parallel expert agents (performance-engineer, debugger, refactoring-specialist, python-pro, code-reviewer) + 1 verifier (general-purpose). Verifier-locked plan:

- **Representation: Candidate C (dense-zeroed accumulators)**. Packed-everywhere (B, M, K) was ruled out as BLOCKING by two independent constraints: (1) the `unsqueeze(2)` broadcast at lines 1359/1390 requires (B, N, N, K) shape for `avg_grad - grad_kl_per_pair`; (2) `_un_apply_rope_pair_outer` at `transport_ops.py:252` hardcodes `B, N_i, N_j, K = grad.shape`. The chosen representation packs Omega and per-pair intermediates inside the block loop and scatters into the dense (B, N, N, K) accumulators via `torch.tril_indices`.
- **Mask + softmax stay UNCHANGED** at vfe_gradients.py:1318-1336. Under the triangle path, `kl_values[..., j>i] = 0` (zero-init never written), not `-inf` — the existing `masked_fill(mask==0, -inf)` is still what produces β=0 there.
- **Bit-identity achievable** (`torch.equal`) for β, grad_mu, grad_sigma — verified across `use_rope ∈ {False, True}`, `mask_self_attention ∈ {False, True}`, `alpha_div ∈ {1.0, 0.5}`. `kl_matrix` differs in upper triangle only (real KL vs zero).
- **Diagnostic correction needed** at lines 1404-1410: `kl_values.mean()`, `.max()`, `.quantile(0.95)`, `.float().mean()` shift semantically under triangle (j>i zeros dilute the mean). Reduce over `kl_values[:, _tril_i, _tril_j]` when the toggle is on.
- **Validation hoisted to once-per-`forward()`** at the VFEEStep call site, not per kernel call (would sync per E-step inner iteration).

# Wave 6: Diagnostic — why `use_autograd_mu_sigma=True` produces worse training results

## Trigger

> deploy multiple expert agents and investigate why 'use_autograd_mu_sigma' = True produces worse results than when it is false. deploy a verifier to verify any findings. ultrathink

## Process

Four parallel investigators followed by one independent verifier:

- `vfe-codebase-auditor` — theoretical: is total-derivative the correct E-step gradient under mean-field VMP?
- `debugger` — code-level bug hunt: 12 targets in `_compute_mu_sigma_grad_autograd` vs the analytic kernel.
- `general-purpose` — empirical gradient-magnitude probe (saved to `scripts/diagnostic_autograd_gradient_scale_probe.py`).
- `vfe-experiment-analyst` — scan of `transformer/vfe/vfe_runs/` for matched pairs.

Verifier (`general-purpose`) re-read every cited line and ran the probe. Six of seven claims CONFIRMED, one PARTIALLY-CONFIRMED on line-numbering nit; no claim invalidates the conclusion.

## Root cause

The analytic kernel (`use_autograd_mu_sigma=False`, default) computes the *query-side partial* ∂F/∂(μ_i,σ_i) — the canonical mean-field VMP gradient (Beal 2003 Ch. 2; Winn & Bishop 2005; Friston 2010). The autograd path (`=True`) computes the *total derivative* via `torch.autograd.grad(F_total, [μ_g, σ_g])`, which double-counts asymmetric couplings under parallel q-updates and adds a key-side σ term with magnitude O(1/σ²) from `KL(q_j‖Ω_ji q_i)` for every j ≠ i.

Empirical probe at matched (μ,σ,φ) state with the production config (K=20, GL(20), λ_align=4, σ_init=0.4):

| iter | μ RMS ratio (au/an) | σ RMS ratio (au/an) | σ mean(grad_au − grad_an) |
|---|---|---|---|
| 0 | 1.82× | 19.6× | −5.65 |
| 5 | 1.26× | **144.6×** | **−15.58** |

β was identical between paths (verified `max|β_an − β_au| = 0.0`), so the gap lives entirely in the (μ,σ) kernel. Effective σ-step under `e_sigma_lr=0.015` reaches 0.26 by iter 5 (vs 0.002 analytic) — 124× past what the LR was tuned for, blowing past σ_init=0.4. Per-token σ cosine drops to −0.3 in the tail: the autograd σ-gradient is partly *orthogonal* to the analytic one, not just rescaled. The F-monotone-monitor warning the analytic path produces is the expected non-monotonicity of parallel mean-field VMP, not a bug — the monitor was the original justification (audit-2026-05-17) for adding the autograd path.

## Recommendations

1. **Keep `use_autograd_mu_sigma=False` as the default.** The flag-on path is a *non-canonical* gradient, not a more principled one.
2. **Rewrite the e_step.py:1314-1338 docstring.** It currently calls the autograd path the "total-derivative path" and the analytic path the "query-side partial only (mean-field convention)" — implying the autograd is more correct. Reverse the framing: query-side partial *is* canonical mean-field VMP; total-derivative is a research-only variant whose F-monotonicity claim depends on a monitor that is not the right convergence criterion for parallel q-updates.
3. **Document the F-monitor caveat** in `_f_monotone_step` — parallel mean-field VMP is not expected to produce per-iteration monotone F decrease (Jordan et al. 1999 §6). Only fully sequential coordinate descent is.
4. **Defect** (separate, owner: trainer): `transformer/vfe/trainer.py` writes no per-run config snapshot (no `system_info.json`, no `dataclasses.asdict(cfg)`). This blocks all post-hoc "which flag did this" run analysis. Suggested fix: emit `<output_dir>/system_info.json` at trainer init with config, git SHA, torch/cuda versions, GPU, seed.
5. **Note** that the audit-2026-05-17 recommendation "users who want strict F-monotone descent should set `use_autograd_mu_sigma=True`" is now superseded by this analysis. The recommendation should be removed or inverted.
