## Update — F6.1 partial close (later 2026-05-19)

Per user request, added the per-block direct-sum Killing form as a **toggle**
rather than flipping the default. The ambient `'killing'` path is unchanged;
a new `'killing_per_block'` value selects the direct-sum metric.

**New code:**
- `transformer/core/gauge_preconditioner.py` — `build_killing_form_preconditioner_per_block(generators, irrep_dims, center_reg=None, center_only=True, return_both=False, block_mass_threshold=0.9)`.
  - Identifies each generator's owning block by Frobenius-mass dominance.
  - Builds `metric_h = 2·d_h·gram_h − 2·tr_h⊗tr_h` per block; assembles block-diagonally in the generator index.
  - Per-block center regularisation (each `gl(d_h)`'s center lifted independently).
  - Raises `ValueError` if any generator straddles blocks (direct-sum Killing form is undefined for those).
- `transformer/core/phi_evolution.py::precondition_phi_gradient` — `mode='killing_per_block'` routes through the same `apply_killing_form_natural_gradient` kernel; the semantic difference lives entirely in how the preconditioner was built upstream.

**Wiring:**

- `transformer/core/block_config.py` — `phi_natural_gradient` docstring lists the new option.
- `transformer/vfe/e_step.py` — `VFEEStep.__init__` builds the per-block preconditioner when `cfg.phi_preconditioner == 'killing_per_block'`.
- `transformer/core/variational_ffn.py` — legacy publication path (`VariationalFFNDynamic.__init__`) supports the new mode; raises if `irrep_dims` is not supplied.

**Tests added** (`tests/transformer/test_gauge_preconditioner.py::TestKillingFormPreconditionerPerBlock`, 10 tests):
- Shape, symmetry, block-diagonality in generator index, per-block scale `2·d_h` (vs ambient `2·K_full`), positive-definiteness, natural-gradient finiteness, natural-gradient does-not-couple-blocks, straddling-generator raises, irrep_dims-sum mismatch raises, per-block ≠ ambient on the same generator bank.

## F4.10 — Causal-mask wasted compute (opt-in toggle `causal_lower_triangle`)

**Goal:** when the caller passes a strict lower-triangular causal mask, the fused E-step kernel `_fused_attention_and_vfe_gradients_block_diag` allocates and computes (B, N, N, K)/(B, N, N, d, d) per-pair tensors for all N² pairs even though softmax masks the upper half to β=0. Restructure the per-block loop to compute only the lower-triangle pairs (j ≤ i, M = N(N+1)/2 pairs) and scatter into the existing dense accumulators.

**Investigation:** 5 parallel expert agents (performance-engineer, debugger, refactoring-specialist, python-pro, code-reviewer) + 1 verifier (general-purpose). Verifier-locked plan:

- **Representation: Candidate C (dense-zeroed accumulators)**. Packed-everywhere (B, M, K) was ruled out as BLOCKING by two independent constraints: (1) the `unsqueeze(2)` broadcast at lines 1359/1390 requires (B, N, N, K) shape for `avg_grad - grad_kl_per_pair`; (2) `_un_apply_rope_pair_outer` at `transport_ops.py:252` hardcodes `B, N_i, N_j, K = grad.shape`. The chosen representation packs Omega and per-pair intermediates inside the block loop and scatters into the dense (B, N, N, K) accumulators via `torch.tril_indices`.
- **Mask + softmax stay UNCHANGED** at vfe_gradients.py:1318-1336. Under the triangle path, `kl_values[..., j>i] = 0` (zero-init never written), not `-inf` — the existing `masked_fill(mask==0, -inf)` is still what produces β=0 there.
- **Bit-identity achievable** (`torch.equal`) for β, grad_mu, grad_sigma — verified across `use_rope ∈ {False, True}`, `mask_self_attention ∈ {False, True}`, `alpha_div ∈ {1.0, 0.5}`. `kl_matrix` differs in upper triangle only (real KL vs zero).
- **Diagnostic correction needed** at lines 1404-1410: `kl_values.mean()`, `.max()`, `.quantile(0.95)`, `.float().mean()` shift semantically under triangle (j>i zeros dilute the mean). Reduce over `kl_values[:, _tril_i, _tril_j]` when the toggle is on.
- **Validation hoisted to once-per-`forward()`** at the VFEEStep call site, not per kernel call (would sync per E-step inner iteration).

**Files modified:**
- `transformer/vfe/config.py` — added `causal_lower_triangle: bool = False` after `use_autograd_mu_sigma`. Default OFF (dense path, no behavioral change).
- `transformer/core/vfe_gradients.py` — added kwarg + pre-loop `torch.tril_indices` precompute + ~150-line triangle branch inside the per-block loop (handles all alpha_div / use_rope / compute_sigma_align_grad cases); dense path is byte-identical to the prior code. Diagnostics reduce over the lower triangle when the toggle is on.
- `transformer/vfe/e_step.py` — `VFEEStep.__init__` plumbs `cfg.causal_lower_triangle`; `forward()` validates ONCE per forward that the mask is strict lower-triangular when the toggle is on (one host sync per forward, not per inner iteration); kernel call site passes the kwarg.

**Tests added** (`tests/transformer/test_vfe_gradients.py::TestCausalLowerTriangleParity`, 10 tests):
- 8 parametrized parity tests across `use_rope × mask_self_attention × alpha_div` — assert β, grad_mu, grad_sigma match dense within fp32 tolerance, kl_matrix matches on the lower triangle, kl_matrix is exactly zero on the strict upper triangle.
- 1 row-stochastic test — β rows sum to 1; upper-triangle β is exactly zero.
- 1 finite-outputs test under `mask_self_attention=True`.

**Validation:**
- `pytest tests/transformer/test_vfe_gradients.py::TestCausalLowerTriangleParity -x` → 10/10 passed.
- `pytest tests/` (full suite) → 960 passed, 3 skipped. Baseline 950 + 10 new tests; no regressions.

**Realistic perf expectation:** the verifier flagged that performance-engineer's "97-98% inner-kernel cost" claim is INCONCLUSIVE without runtime profiling. The Omega construction at line 1154 (a `bikl,bjlm->bijkm` einsum producing (B,N,N,d,d)) is the largest single op per block and is halved under the toggle. End-to-end E-step iteration savings estimated at 25-40% (not the naive 50%, because the per-token precompute `fused_block_matrix_exp_pairs` at lines 1116-1124 and self-coupling work at lines 1063-1102 stay full-N). Real wall-clock gains require runtime measurement.

**Defaults:** unchanged. `causal_lower_triangle=False` remains the default; users opt into the fast path explicitly. The dense path is byte-identical to prior behavior. Phase 2 (flip default after empirical confirmation at scale) is OUT OF SCOPE for this PR.

**Out of scope (deferred):**
- Flipping the default to True after empirical baking.
