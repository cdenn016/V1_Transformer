## F4.10 ΓÇö Causal-mask wasted compute (opt-in toggle `causal_lower_triangle`)

**Goal:** when the caller passes a strict lower-triangular causal mask, the fused E-step kernel `_fused_attention_and_vfe_gradients_block_diag` allocates and computes (B, N, N, K)/(B, N, N, d, d) per-pair tensors for all N┬▓ pairs even though softmax masks the upper half to ╬▓=0. Restructure the per-block loop to compute only the lower-triangle pairs (j Γëñ i, M = N(N+1)/2 pairs) and scatter into the existing dense accumulators.

**Investigation:** 5 parallel expert agents (performance-engineer, debugger, refactoring-specialist, python-pro, code-reviewer) + 1 verifier (general-purpose). Verifier-locked plan:

- **Representation: Candidate C (dense-zeroed accumulators)**. Packed-everywhere (B, M, K) was ruled out as BLOCKING by two independent constraints: (1) the `unsqueeze(2)` broadcast at lines 1359/1390 requires (B, N, N, K) shape for `avg_grad - grad_kl_per_pair`; (2) `_un_apply_rope_pair_outer` at `transport_ops.py:252` hardcodes `B, N_i, N_j, K = grad.shape`. The chosen representation packs Omega and per-pair intermediates inside the block loop and scatters into the dense (B, N, N, K) accumulators via `torch.tril_indices`.
- **Mask + softmax stay UNCHANGED** at vfe_gradients.py:1318-1336. Under the triangle path, `kl_values[..., j>i] = 0` (zero-init never written), not `-inf` ΓÇö the existing `masked_fill(mask==0, -inf)` is still what produces ╬▓=0 there.
- **Bit-identity achievable** (`torch.equal`) for ╬▓, grad_mu, grad_sigma ΓÇö verified across `use_rope Γêê {False, True}`, `mask_self_attention Γêê {False, True}`, `alpha_div Γêê {1.0, 0.5}`. `kl_matrix` differs in upper triangle only (real KL vs zero).
- **Diagnostic correction needed** at lines 1404-1410: `kl_values.mean()`, `.max()`, `.quantile(0.95)`, `.float().mean()` shift semantically under triangle (j>i zeros dilute the mean). Reduce over `kl_values[:, _tril_i, _tril_j]` when the toggle is on.
- **Validation hoisted to once-per-`forward()`** at the VFEEStep call site, not per kernel call (would sync per E-step inner iteration).

# Wave 6: Diagnostic ΓÇö why `use_autograd_mu_sigma=True` produces worse training results

## Trigger

## Process

Four parallel investigators followed by one independent verifier:

- `debugger` ΓÇö code-level bug hunt: 12 targets in `_compute_mu_sigma_grad_autograd` vs the analytic kernel.

Verifier (`general-purpose`) re-read every cited line and ran the probe. Six of seven claims CONFIRMED, one PARTIALLY-CONFIRMED on line-numbering nit; no claim invalidates the conclusion.

## Root cause

The analytic kernel (`use_autograd_mu_sigma=False`, default) computes the *query-side partial* ΓêéF/Γêé(╬╝_i,╧â_i) ΓÇö the canonical mean-field VMP gradient (Beal 2003 Ch. 2; Winn & Bishop 2005; Friston 2010). The autograd path (`=True`) computes the *total derivative* via `torch.autograd.grad(F_total, [╬╝_g, ╧â_g])`, which double-counts asymmetric couplings under parallel q-updates and adds a key-side ╧â term with magnitude O(1/╧â┬▓) from `KL(q_jΓÇû╬⌐_ji q_i)` for every j Γëá i.

Empirical probe at matched (╬╝,╧â,╧å) state with the production config (K=20, GL(20), ╬╗_align=4, ╧â_init=0.4):

| iter | ╬╝ RMS ratio (au/an) | ╧â RMS ratio (au/an) | ╧â mean(grad_au ΓêÆ grad_an) |
|---|---|---|---|
| 0 | 1.82├ù | 19.6├ù | ΓêÆ5.65 |
| 5 | 1.26├ù | **144.6├ù** | **ΓêÆ15.58** |

╬▓ was identical between paths (verified `max|╬▓_an ΓêÆ ╬▓_au| = 0.0`), so the gap lives entirely in the (╬╝,╧â) kernel. Effective ╧â-step under `e_sigma_lr=0.015` reaches 0.26 by iter 5 (vs 0.002 analytic) ΓÇö 124├ù past what the LR was tuned for, blowing past ╧â_init=0.4. Per-token ╧â cosine drops to ΓêÆ0.3 in the tail: the autograd ╧â-gradient is partly *orthogonal* to the analytic one, not just rescaled. The F-monotone-monitor warning the analytic path produces is the expected non-monotonicity of parallel mean-field VMP, not a bug ΓÇö the monitor was the original justification (audit-2026-05-17) for adding the autograd path.

## Recommendations

1. **Keep `use_autograd_mu_sigma=False` as the default.** The flag-on path is a *non-canonical* gradient, not a more principled one.
2. **Rewrite the e_step.py:1314-1338 docstring.** It currently calls the autograd path the "total-derivative path" and the analytic path the "query-side partial only (mean-field convention)" ΓÇö implying the autograd is more correct. Reverse the framing: query-side partial *is* canonical mean-field VMP; total-derivative is a research-only variant whose F-monotonicity claim depends on a monitor that is not the right convergence criterion for parallel q-updates.
3. **Document the F-monitor caveat** in `_f_monotone_step` ΓÇö parallel mean-field VMP is not expected to produce per-iteration monotone F decrease (Jordan et al. 1999 ┬º6). Only fully sequential coordinate descent is.
4. **Defect** (separate, owner: trainer): `transformer/vfe/trainer.py` writes no per-run config snapshot (no `system_info.json`, no `dataclasses.asdict(cfg)`). This blocks all post-hoc "which flag did this" run analysis. Suggested fix: emit `<output_dir>/system_info.json` at trainer init with config, git SHA, torch/cuda versions, GPU, seed.
5. **Note** that the audit-2026-05-17 recommendation "users who want strict F-monotone descent should set `use_autograd_mu_sigma=True`" is now superseded by this analysis. The recommendation should be removed or inverted.
