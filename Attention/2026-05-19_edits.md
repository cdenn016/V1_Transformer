# Cross-head coupling ported to `/vfe`

User asked: "investigate whether cross_coupling feature is available in /vfe
implementation … build it out plug any metrics and visualizations". Investigation
confirmed the feature was legacy-only (`transformer/core/model.py` +
`transformer/core/block_equivariant_mixer.py`); shared math kernels lived in
`math_utils/generators.py` (`generate_glK_cross_head_generators`,
`merge_coupled_heads`, `reorder_cross_head_generators`, `_dedup_cross_couplings`).
This pass ports it into the `/vfe` package end-to-end, covering the full +
non-flat + full-cov scope per the user's selection.

## What `cross_couplings` is

Each `(a, b)` entry adds `d_head²` off-diagonal generators spanning head-a's
subspace into head-b's. The resulting `Ω = exp(φ·G)` becomes non-block-diagonal
across the coupled heads, so gauge transport itself moves information between
them — a different mechanism from the value-level `VFEHeadMixer`. Coupled heads
are merged via union-find into super-blocks; the K-axis is permuted so super-
blocks are contiguous.

## Files changed

`transformer/vfe/config.py`

- `cross_couplings: List[Tuple[int,int]]`, `auto_close_cross_head_basis: bool`,
  `validate_cross_head_closure: bool` fields added.
- `__post_init__` validates: GLK-only, single-type multihead, `n_heads >= 2`,
  rejects self-coupling, out-of-range head indices, malformed pairs; silently
  dedups exact duplicates (preserves directional `(a,b)` vs `(b,a)`).
- `n_gen` updated to add `|cross_couplings| · d_head²` to the diagonal count.
- New properties: `is_cross_coupled`, `super_block_dims`,
  `super_block_head_groups`, `effective_block_dims` (returns
  `super_block_dims` when cross-coupled, else `irrep_dims`). Lazily computed
  via `merge_coupled_heads`; results cached on the instance.

`transformer/vfe/model.py`

- `_build_generators` dispatches to `generate_glK_cross_head_generators` →
  `merge_coupled_heads` → `reorder_cross_head_generators` when
  `cross_couplings` is non-empty. Optional Lie-closure handling honored via
  `auto_close_cross_head_basis` / `validate_cross_head_closure`. Stashes
  `_cross_head_perm`, `_super_block_dims_cache`,
  `_super_block_head_groups_cache` on `cfg` for downstream consumers.
- Threading: `VFEPositionalEncoding` and `init_omega_from_phi` now receive
  `cfg.effective_block_dims` (super-block partition under cross-coupling,
  identity otherwise).

`transformer/vfe/e_step.py`

- `self.irrep_dims = cfg.effective_block_dims` so every block-iteration site
  (per-block softmax, per-block KL, per-block Ω, kappa_per_head dispatch,
  killing/cartan preconditioner, F-monitor diagnostic state) walks the
  correct gauge-block partition. `self._original_irrep_dims = cfg.irrep_dims`
  retained for diagnostics that need the original per-head granularity.
- `kappa_per_head` parameter sized to `len(cfg.effective_block_dims)`:
  one learnable κ per super-block, not per original head.
- `VFENonFlatConnection` constructor receives `head_sub_dims` so diagonal
  generators inside a merged super-block get a tight `d_head × d_head`
  mask while cross generators (which span multiple heads inside the super-
  block) fall back to the full super-block mask.

`transformer/vfe/prior_bank.py`

- `self.irrep_dims = cfg.effective_block_dims`. `_phi_trace_vec`,
  `phi_project_slk`, and per-block sandwich-transform iteration all walk
  the super-block partition; the trace constraint targets one determinant
  per super-block GL.

`transformer/vfe/head_mixer.py`

- `VFEHeadMixer.__init__` accepts `super_block_head_groups`. Heads inside a
  merged super-block are *excluded* from the mixer (each merged super-block
  has full GL(d_super) gauge with scalar Schur commutant — any additional
  value-level mixing of constituent heads breaks equivariance). Singleton
  super-blocks of the same type still get the standard free `n × n` mixer.

`transformer/vfe/block.py`

- Passes `cfg.super_block_head_groups` into `VFEHeadMixer`.

`transformer/vfe/non_flat.py`

- `VFENonFlatConnection.__init__` accepts `head_sub_dims` and builds finer
  block candidates so the auto-detection picks the tightest containing block
  per generator under cross-coupling.

`transformer/vfe/positional.py`, `transformer/vfe/omega_direct.py`

- No internal changes — both already iterate `irrep_dims` as the gauge-block
  partition; correctness under cross-coupling follows from passing
  `effective_block_dims` at the call sites above.

`transformer/vfe/cross_coupling_metrics.py` (new)

- `phi_energy_partition`: splits `||φ||²` between diagonal and cross
  generator groups.
- `omega_block_strength`: mean Frobenius mass of each (head_a, head_b)

- `cross_block_kl_share`: fraction of total per-super-block KL inside

- `per_super_block_attention_entropy`: row entropy of `β_ij` per super-block.
- `holonomy_norm_cross_blocks`: Frobenius norm of length-3 holonomies on

- `collect_cross_coupling_metrics`: convenience aggregator.

`transformer/vfe/cross_coupling_viz.py` (new)

- `plot_phi_energy_partition`: stacked bar of diag vs cross φ-energy per

- `plot_super_block_diagnostics`: side-by-side bar charts of per-super-

## AIF Build-Out — Compute Feasibility Analysis (Investigation Agent 4)

New planning document written at:
`docs/plans/2026-05-19-aif-transformer-buildout/04_compute_feasibility.md`

This is a read-only analysis document; no source code was modified.

Content summary:

- §1 establishes that the dominant cost per forward is the `(B,N,2K) × (V,2K)^T` decode matmul at `transformer/vfe/prior_bank.py:498-500`, costing 33 GFLOPs and 1.65 GB logit memory at training shape (B=64, N=128, V=50257, K=20). The E-step attention grid costs approximately 2–3 GFLOPs — about 15× cheaper than decode.

- §2 analyzes the depth-1 EFE candidate scoring loop (`vfe/efe.py:164-201`) and identifies a structural double-encode: the model forward at `efe.py:174` and the BALD encode at `efe.py:191` each perform a separate embedding lookup on the same token sequence. The beliefs from the first forward are not surfaced. This is a latent integration point for the build-out.

- §3 establishes the tree-search FLOP and wall-clock frontier. At B=1 generation shape, (D=2, b=8) requires 128 forwards per token (~0.5–2 s on A100) and is the practical demo target. (D=4, b≥16) and (D=5, b≥8) are demo-prohibitive. Full-covariance Σ at depth > 1 is structurally intractable.

- §4 shows the belief cache at (D=3, b=8) totals approximately 103 MB, well within 24 GB GPU. φ dominates the per-node cache at n_gen=200 vs K=20 (factor 10×). The shared decode prior cache (4 MB) must be preserved across tree siblings; the current `invalidate_cache()` call per forward would create 585 redundant recomputations at D=3 b=8.

- §5 concludes S=4 is acceptable for generation-time scoring with epistemic_weight=0.5. The streaming BALD pattern at `core/active_inference.py:277-296` must be carried through any depth-D recursion to avoid S×(B,N,V) logit stacking.

- §6 recommends demo defaults: D=2, b=4, S=4, top_k=50, B=1, diagonal Σ. Expected 0.5–3 s per token on 24 GB GPU under the current sequential forward loop.

- §7 lists OOM failure modes: training-shape tree search (1.65 GB × b² logits), full-cov at depth > 1 (128× E-step cost), non-streaming BALD in recursion, per-node decode cache recomputation, and gradient-tracked tree search (200–500× memory overhead).

- §8 compares beam, top-k, and nucleus sampling at expansion; sophisticated inference vs flat leaf scoring; and depth-1 vs depth-D EFE quality tradeoffs. Sophisticated inference (recursive EFE at every node) adds only a 2× cost at D=2 and <1.1× overhead at D=3.

- §9 covers training-time EFE cost. Regime A (observed trajectory as policy, no expansion): less than 1% overhead over standard CE. Regime B (tree expansion at training time): approximately 200–500× overhead, estimated 940 A100-hours for 15,000 steps — off-by-default, research-only.

### Math design choices (Phase 2)

- **Per-branch beam (not global beam).** Each node has exactly `beam_width` children; total leaves = `beam_width^D`. The full-cov runtime guard at D > 1 keeps the diagonal-Σ practical envelope reasonable on RTX 5090 (32 GB). The recommended demo preset (D=2, b=4) explores 16 leaves per commit step.
- **Mean back-propagation.** Root child V = mean over reachable leaf G_cum. This is the uniform-policy expectation of the canonical Friston-2021 recursion `E_{q(a'|s)}[G_{d+1}]`; the softmax-weighted form (canonical) is Phase 3. Mean-over-beam is a defensible Phase-2 default because (a) it has no extra hyperparameter and (b) it preserves the property that argmin over root V is the argmin path's G_cum at b=1.
- **Geometric discount across depth.** Hyperparameter `discount` (default 1.0, undiscounted). Each depth-d local G contributes `discount^d * G_local`. Discount is an engineering hyperparameter, not a canonical EFE quantity (per verifier §7.5).
- **AIF decode_tau independent of VFE decode_tau.** The AIF readout re-decodes the last position at `AIFConfig.decode_tau` so generation-time scoring can be tuned without retraining. Costs one extra (V, K) matmul per node — small fraction of the per-node E-step cost.

## AIF Build-Out — Phase 3: sophisticated-inference recursion

Canonical [Friston2021SophisticatedInference] recursion shipped. The
Phase 2 mean back-propagation was the uniform-child-posterior limit of
the canonical recursive form; Phase 3 replaces it with the softmax-
weighted expectation under :math:`q(a' \mid s_d) \propto \exp(-\gamma\, V(a'))`.

### Math

The canonical recursion per `01_canonical_efe_for_lm.md` §3:

.. math::
    V(\text{leaf}) = G_{\text{local}}(\text{leaf})

.. math::
    V(\text{internal}, d) = G_{\text{local}}(d) + \gamma_{\text{disc}}
        \sum_{a' \in \text{beam}(d+1)} q(a' \mid s_d) \, V(a').

The child posterior is dispatched on ``cfg.branching_strategy``:

- ``'beam'`` / ``'top_k'``: uniform ``q(a') = 1 / |beam|``, equivalent to
  the Phase 2 mean aggregation. This is the :math:`\gamma \to 0` limit
  of sophisticated inference.
- ``'sophisticated'``: ``q(a') ∝ exp(-gamma · V(a'))``. At
  :math:`\gamma \to 0` converges to uniform (Phase 2 beam); at
  :math:`\gamma \to \infty` collapses to argmin over the beam.

### Files modified

- ``transformer/aif/tree_search.py``: replaced the Phase 2 mean back-
  propagation with a parametrized recursive V computation. New helpers
  ``_make_aggregator(strategy, gamma)`` dispatches uniform vs softmax-
  weighted aggregation; ``_backprop_V(level_nodes, horizon_D, discount,
  aggregator)`` walks the tree bottom-up. ``V(leaf) = components.G_local``;
  ``V(internal_d) = components.G_local + discount · aggregator(child V's)``.
  Under uniform aggregation the recursion unrolls to mean-over-leaves of
  the geometric-discounted path-sum (matching Phase 2 bitwise modulo
  floating-point order). The ``sophisticated_expand`` Phase-1 stub is
  removed.
- ``transformer/aif/config.py``: removed the ``__post_init__`` rejection
  of ``branching_strategy='sophisticated'``. Updated the docstring on
  ``branching_strategy`` to document the three-way dispatch and the
  ``gamma → 0`` / ``gamma → ∞`` limits. Updated the ``gamma`` docstring
  to note its dual role: root-level action posterior AND recursive
  child-action posterior under sophisticated inference.
- ``tests/transformer/test_aif_package.py``: converted the Phase-1
  ``test_rejects_sophisticated_branching`` guard into a
  ``test_accepts_sophisticated_branching`` acceptance test. Added 14
  new Phase-3 tests:
  - ``TestMakeAggregator`` (7): beam is mean, top_k is mean, sophisticated
    at small gamma matches mean, sophisticated at large gamma matches
    argmin, intermediate gamma is between min and mean, empty input
    returns 0, unknown strategy raises.
  - ``TestSophisticatedExpand`` (4): D=1 matches beam at D=1 (no
    aggregation), small-gamma D=2 ≈ beam D=2, large-gamma D=2 runs
    end-to-end without numerical issues, AIFGenerator with
    sophisticated branching generates tokens.
  - ``TestBackpropV`` (3): hand-computed verification on a synthetic
    2-level tree — uniform recursion, discount halving, sophisticated
    argmin limit at large gamma.

### Design choices

- ``cfg.gamma`` is shared between the recursion's child posterior and
  the root-level action sampling. This is the canonical Friston 2021
  interpretation: one policy precision controls all softmax-weighted
  posteriors. Separating them is a research extension.
- ``_backprop_V`` keys nodes by ``id(node)`` (not by ``action_seq``)
  because two distinct nodes can share an action_seq under aggressive
  truncation; ``id()`` is safe by construction.
- Aggregator builds use fp64 internally for the softmax to preserve
  precision in the ``gamma → ∞`` argmin limit (the
  ``test_sophisticated_at_large_gamma_matches_argmin`` test would fail
  in fp32 at ``gamma=1e6``).

## AIF Build-Out — Phase 4: trajectory-as-policy training augmentation

Phase 4 (Regime A) ships the canonical training-time EFE augmentation.
The observed training sequence is treated as the agent's chosen policy;
per-position EFE is computed from the model's own forward outputs and
folded into the M-step objective as :math:`L_{\text{total}} =
L_{\text{CE}} + \lambda_{\text{AIF}}\, L_{\text{AIF}}`. No tree
expansion at training time (Regime B is structurally infeasible at
~200-500× CE cost per ``04_compute_feasibility.md`` §9).

### Math

For each batch position :math:`(b, t)`:

.. math::
    G_t = \text{pragmatic}_t + \text{ambiguity}_t
        - \lambda_{\text{epi}}\, \text{epistemic}_t,

with

- **Pragmatic**: :math:`\text{prag}_t = E_{q(o \mid a)}[-\log p^*(o)]`
  computed as cross-entropy of the predictive against the configured
  preference distribution. Cheap — no extra forward passes.
- **Ambiguity**: :math:`\text{amb}_t = E_{q(s_t)}[H[p(o \mid s_t)]]` via
  BALD MC sampling at every batch position. Gradient through the
  reparameterized :math:`z_s = \mu + \sqrt{\Sigma}\, \varepsilon`
  carries into both :math:`\mu` and :math:`\Sigma`.
- **Epistemic**: :math:`\text{epi}_t = I_{q(s_t, o)}(s_t; o) = H[\bar p]
  - E_s[H[p_s]]`. Shares the same MC sampling pass with ambiguity (no
  extra forward passes when both are enabled).

The augmentation is gated per term: ``train_include_pragmatic``,
``train_include_ambiguity``, ``train_include_epistemic`` flags let the
user drop the expensive BALD sampling (e.g. pragmatic-only ≈ <1%
overhead over standard CE per the verifier's Regime A estimate).

### Files added / modified

- **NEW** ``transformer/aif/training_loss.py``: standalone gradient-
  tracked loss function ``compute_training_efe_loss(logits, beliefs,
  preference, prior_bank, cfg) -> scalar tensor``. Helpers
  ``_pragmatic_per_position`` and
  ``_bald_ambiguity_and_epistemic_per_position`` keep the three terms
  modular. The reparameterized BALD sampling is the only stochastic
  component; both fields of beliefs (mu and sigma diagonal) receive
  gradients via the reparameterization.
- ``transformer/aif/config.py``:
  - Removed the Phase-1 NotImplementedError on
    ``training_objective='efe_augmented'``.
  - Added four new fields with documentation: ``aif_loss_weight``
    (multiplier, default 0.1), ``train_include_pragmatic``,
    ``train_include_ambiguity``, ``train_include_epistemic`` (all
    default True).
  - ``__post_init__`` now raises ValueError if
    ``training_objective='efe_augmented'`` is set with all three term
    flags False (the augmentation would be a no-op).
- ``transformer/aif/__init__.py``: exported ``compute_training_efe_loss``.
- ``tests/transformer/test_aif_package.py``: converted the Phase-1
  ``test_rejects_efe_augmented_training`` guard into a
  ``test_accepts_efe_augmented_training`` acceptance test plus a
  ``test_efe_augmented_with_no_terms_raises`` ValueError check. Added
  ``TestTrainingEFELoss`` with 6 new tests:
  - Returns a scalar tensor with finite value.
  - Pragmatic-only mode works (no BALD pass).
  - Gradient flows to at least one PriorBank parameter on backward.
  - Pragmatic value is lower when the preference is concentrated on the
    model's predicted argmax token (sign correctness).

  - Composes correctly with cross-entropy in a representative training
    step.

### Integration pattern for the user's training loop

The function is a drop-in addition to any training script — four lines
of integration:

```python
from transformer.aif import AIFConfig, build_preference, compute_training_efe_loss

aif_cfg = AIFConfig(training_objective='efe_augmented', preference_path=...)
preference = build_preference(aif_cfg.preference_type, aif_cfg.preference_path)
# ... inside the training step:
logits, beliefs = model.forward_with_beliefs(input_ids)
ce_loss = F.cross_entropy(logits.view(-1, V), targets.view(-1), ignore_index=-100)
aif_loss = compute_training_efe_loss(logits, beliefs, preference, model.prior_bank, aif_cfg)
total_loss = ce_loss + aif_cfg.aif_loss_weight * aif_loss
total_loss.backward()
```

Replaces the existing ``model.forward(input_ids, targets)`` call with
``forward_with_beliefs`` plus explicit CE; the AIF loss is added as a
single line. The user's existing ``VFETrainer`` can be extended this
way without subclassing.

  then adds ``self._loss_scale * aif_cfg.aif_loss_weight * L_AIF`` where
  ``L_AIF`` comes from
  :func:`transformer.aif.training_loss.compute_training_efe_loss`. The
  returned metrics dict carries a new ``aif_loss`` field for logging.
  Every other piece of training infrastructure — optimizer, schedulers,
  gradient clipping, CSV metrics, attention plots, eval loop — is
  inherited unchanged.
- **NEW** ``transformer/aif/train_aif_augmented.py``: click-to-run
  entry. Two config dicts at the top — ``vfe_config`` (mirrors
  ``train_vfe.py`` defaults) and ``aif_config`` (Phase-4 augmentation
  parameters). On first run it builds the empirical-marginal preference
  distribution from the training corpus and caches it to
  ``aif_runs/preferences/{DATASET}_empirical_marginal.pt`` (Laplace add-
  one smoothing for finite log-probs on unseen tokens). Subsequent runs
  reuse the cache. Output directory tag includes
  ``aif{aif_loss_weight}`` so multiple runs at different strengths
  don't collide.

### Tests added

``TestAIFAugmentedTrainer`` in ``tests/transformer/test_aif_package.py``
adds 4 smoke tests:

- ``test_construction``: builds an AIFAugmentedTrainer end-to-end with

  ``training_objective='standard_vfe'`` — they should use

### Usage

To run an EFE-augmented training experiment:

1. Edit ``transformer/aif/train_aif_augmented.py``: adjust
   ``vfe_config`` (model spec) and ``aif_config`` (augmentation
   strength + sampling) at the top of the file.
2. Press Run.
3. First run builds + caches the corpus marginal preference; subsequent
   runs reuse the cache.
4. Outputs land in ``aif_runs/{DATASET}_aif{lambda}_{timestamp}/``
   alongside CSV metrics, attention plots, and checkpoints from the
   standard ``VFETrainer`` infrastructure.

To recover standard VFE training from the same entry point: set
``aif_config['aif_loss_weight'] = 0.0`` (the AIF terms still compute
but contribute zero to backward), or set all three
``train_include_*`` flags to ``False`` to skip the augmentation
entirely.
