# 2026-05-16 Edits

## attention.py — fix two misleading log messages

User reported that `run_ablation_suite.py` emits

```
[GL(K) cross-head] super-blocks=[10, 10] d_head=10 (merged from 2 original heads; ...)
...
[GL(K) Attention] Single head, dim=20, n_generators=200  -> Full GL(20) transport on entire embedding space
```

for a config with `irrep_spec=[('fund', 2, 10)]` and `cross_couplings` commented out. This produced a false impression that (a) cross-coupling was active and (b) attention was collapsing the 2×GL(10) structure into single-head GL(20). Investigation (see commits and memory observations 4455, 4456, 4468 from 2026-05-16) showed both prints were stale or hardcoded — the model was in fact correctly built as two GL(10) heads (`self.glk_multihead=True`, `self.n_heads=2`, `self.irrep_dims=[10, 10]`, `200 = 2·10²` block-diagonal generators).

### Change 1 — `transformer/core/attention.py:1459-1474` (cross-head log)

Split the constructor log so that the "cross-head / super-blocks / merged from N original heads" wording only fires when `irrep_dims_override` actually differs from the plain per-head decomposition `[d_head] * n_heads`. The plain multi-head case (no `cross_couplings`, just `use_block_diagonal_kl=True`) now prints

```
[GL(K) multi-head] 2 heads x GL(10), block_dims=[10, 10] (no cross-coupling)
```

The true cross-coupled case (non-empty `cross_couplings` → `merge_coupled_heads` returns merged super-block dims) keeps the original "cross-head / super-blocks / merged" wording.

### Change 2 — `transformer/core/attention.py:1685-1697` (end-of-init "Print attention configuration" block)

Replaced the hardcoded `"Single head, dim={embed_dim}"` text — leftover from when `gauge_group='GLK'` was single-head only — with a branch on `self.glk_multihead`. Multi-head GLK now prints actual structure:

```
[GL(K) Attention] 2 head(s), irrep_dims=[10, 10], n_generators=200 (block-diagonal over heads)
  -> Per-head GL(d) transport; total embed_dim=20
```

Single-head GLK (`irrep_spec=[('full', 1, K)]`) keeps the existing single-head log.

### Not changed (intentionally)

- `scripts/run_ablation_suite.py` BASELINE_CONFIG and `transformer/train_publication.py` EM_CONFIG were left untouched per user request (active runs are using them).
- Underlying model construction is unchanged — these edits are diagnostic / cosmetic only.

## block_config / blocks — thread `include_attention_entropy` through to the FFN

Investigation confirmed that `'include_attention_entropy': True` in `train_publication.py:155` EM_CONFIG and `scripts/run_ablation_suite.py:95` BASELINE_CONFIG was being **silently dropped** at the `BlockConfig.from_config` boundary. `BlockConfig` had no field, `blocks.py` did not pass the kwarg, and `VariationalFFNDynamic.__init__` (`variational_ffn.py:221`) defaulted to `True` — so runtime behavior matched the dict only by accidental default coincidence. Setting `False` would have had zero effect with no warning.

The parallel `transformer/vfe/` package threads the flag correctly via `vfe/config.py:53` → `vfe/e_step.py:84`. The standalone `transformer/train.py:274,427` also wires it directly. Only the `BlockConfig`-mediated `transformer/core/` path used by `train_publication.py` (`mode='em'`) and `run_ablation_suite.py` was broken.

### Change 1 — `transformer/core/block_config.py:145-150` (new dataclass field)

Added `include_attention_entropy: bool = True` to the `BlockConfig` dataclass right after `E_lambda_softmax`. Default `True` matches the FFN constructor default at `variational_ffn.py:221` and the production EM_CONFIG / BASELINE_CONFIG dict values.

### Change 2 — `transformer/core/block_config.py:520` (from_config dispatch)

Added `include_attention_entropy=config.get('include_attention_entropy', True),` to the `from_config` dispatch, placed adjacent to `E_lambda_belief` and `E_lambda_softmax` since the flag gates the τ·Σβ·log(β/π) attention-entropy term that those two weights scale.

### Change 3 — `transformer/core/blocks.py:526` (FFN constructor call)

Added `include_attention_entropy=cfg.include_attention_entropy,` between `lambda_softmax=cfg.E_lambda_softmax,` and `kappa=cfg.ffn_kappa,` in the `VariationalFFNDynamic(...)` call inside `Block.__init__`.

### Verification

Smoke test (`python -c "from transformer.core.block_config import BlockConfig; ..."`):
- `'include_attention_entropy': False` in dict → `BlockConfig.include_attention_entropy == False`
- key absent → `BlockConfig.include_attention_entropy == True` (dataclass default)
- `'include_attention_entropy': True` in dict → `BlockConfig.include_attention_entropy == True`

Grep confirms the FFN receives the value via `cfg.include_attention_entropy` at `blocks.py:526` and stores it at `variational_ffn.py:513`, gating the τ·Σβ·log(β/π) term at the three call sites `:1050, :1249, :1288`.

### Not changed (intentionally)

- `transformer/core/variational_ffn.py` — the FFN already accepts the kwarg and gates the term correctly; no edits needed.
- `transformer/vfe/` package — already threads the flag correctly through `vfe/config.py:53`.
- `transformer/train.py` — already threads the flag directly into `compute_free_energy_loss` (`:274, :427`); unaffected by `BlockConfig` path.
- `EM_CONFIG` and `BASELINE_CONFIG` dict values — unchanged (both already set `True`, which now actually propagates).
