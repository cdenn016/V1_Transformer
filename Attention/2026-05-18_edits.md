## Per-step live status on the /vfe training console

User requested that the `/vfe` training loop show real-time iterations
per second every step, not only on the `log_interval`-gated verbose
line. The verbose every-200-step format

```
Step 200/2000 | Loss: 1.4183 | CE: 6.3428 | β: 0.0939 | PPL: 568.4
```

stays as-is for high-level captures; a per-step indicator was added
on the tqdm bar so the step counter ticks live with the metrics and
rate.

### Surgery — `transformer/vfe/trainer.py::VFETrainer.train`

Three localized edits inside the `train()` method.

**1. tqdm bar configuration (lines 899-906).** Renamed
`desc="Training"` to `desc="Step"` so the bar prefix matches the user's
preferred `Step N/total` wording, and dropped `mininterval` from
`0.5` to `0.1` so the bar redraws at ~10 Hz instead of every half
second — that change is what makes the per-step ticks feel real-time.

**2. Per-step postfix (lines 938-953, newly inserted).** Immediately
after `metrics = self.train_step(batch)` and the CSV/pub-tracker
hooks, the trainer now derives `it_per_sec = 1.0 / metrics['step_time']`
(guarded against zero) and, when a tqdm bar is active, calls
`pbar.set_postfix({'loss', 'ce', 'ppl', 'it/s'}, refresh=False)`. The
postfix update is cheap — it touches only fields already in the
`metrics` dict, which `train_step` populated synchronously, so no
extra GPU→CPU sync is triggered. `refresh=False` lets tqdm's own
`mininterval=0.1` rate-limit the redraw rather than forcing one per
step. `β` is deliberately excluded from the postfix because
`_attention_summary()` does a per-block `.tolist()` GPU sync (see
`trainer.py:230-`); calling it every step would add 2-3 `cudaSync`
points per iteration, defeating the purpose of a real-time rate
display.

**3. Verbose log line (lines 958-969).** Appended `| it/s: {it/s:.2f}`
to the every-`log_interval` message so the captured high-level lines
also carry the rate. β stays on this line — that single
`_attention_summary()` call every 200 steps was already paid for.

## Follow-up: full formatted line on the bar every step

User clarification: the per-step `set_postfix` (loss/ce/ppl/it/s only)
was still considered "only updating every log interval" because the
full `Step N/T | Loss | CE | β | PPL | it/s` line — the format the
user actually reads — was gated by `log_interval`. The request was to
have that exact line update live every step in the same manner
`experiment_runner.py` updates its bar (the second loop at
`experiment_runner.py:2861` uses `pbar.set_description(log_msg)`,
which surfaces the formatted line directly inside the bar prefix).

### Surgery — `transformer/vfe/trainer.py::VFETrainer.train` (lines 938-967)

The previous postfix-only block was replaced. The trainer now builds
the full `Step | Loss | CE | β | PPL | it/s` message every iteration
and pushes it onto the bar via `pbar.set_description_str(msg, refresh=False)`.
`set_description_str` (not `set_description`) suppresses the trailing
`": "` that tqdm would otherwise append, so the bar reads
`... it/s: 50.00 12%|...` instead of `... it/s: 50.00:  12%|...`.
`refresh=False` keeps the redraw rate-limited by tqdm's existing
`mininterval=0.1` (~10 Hz) instead of forcing a screen redraw on every
step. The `_write(msg)` at the `log_interval` boundary is preserved
unchanged for scrollback — it now prints exactly the same string that
the bar is showing, giving a permanent record every 200 steps without
duplicating the live status.

### Cost trade

The previous postfix path deliberately excluded β because
`_attention_summary()` does a `.tolist()` sync, and the original aim
was zero per-step extra syncs. Showing the full line live requires β
every step, so this trade is now accepted: one `.tolist()` per step on
the per-block stacked stats tensor (see `trainer.py:208-251`). For
n_layers=1 this is a single sync; the trainer already issues several
host-side reads per `train_step` (grad-norm `.tolist()`, CSV writes,
publication tracker), so the marginal cost is small relative to the
existing per-step work.

### Why `set_description` and not `tqdm.write`

`tqdm.write(msg)` would scroll a new line above the bar every step —
2000 lines for a 2000-step run, defeating tqdm. `set_description` puts
the message inline as the bar's prefix, so the user sees one line that
updates in place — the live status the user asked for. The
`log_interval`-gated `_write(msg)` then converts that live status into
a scrollback record at coarse intervals.

## Orphaned e_sigma_lr / e_phi_lr sweeps in vfe_ablation_suite.py

User observation: sweeping `e_sigma_lr` and `e_phi_lr` via
`vfe_ablation_suite.py` produced identical Val PPL across every value,
while `e_mu_lr` sweeps did vary as expected. The orphan warning added
in commit `39f19299` fires for the user's `BASELINE_CONFIG`, but the
sweep machinery did not act on it.

### Root cause (asymmetry between mu vs. sigma/phi)

Inside the E-step iteration body (`vfe/e_step.py:666-696`):
```
1. Compute beta, nat_grad_mu, nat_grad_sigma  (reads INPUT mu, sigma, phi)
2. mu  <- mu - e_mu_lr * nat_grad_mu          (retract mu)
3. sigma <- retract via e_sigma_lr             (retract sigma)
4. phi   <- retract via e_phi_lr               (retract phi)
```

After the loop exits with `n_e_steps=1`, the per-output downstream
consumers under BASELINE_CONFIG (`n_layers=1`, `use_prior_bank=False`,
`norm_type='layernorm'`) are:

| Output | Reaches loss via | Live? |
|---|---|---|
| mu | CE decoder: `logits = output_proj(mu)` | YES, always |
| sigma | next-iter `nat_grad_mu` / next-layer / `MahalanobisNorm` / `prior_bank.decode` | NO under all four gates |
| phi | next-iter attention / next-layer | NO under both gates |

So `e_mu_lr` changes a value the decoder reads (loss changes);
`e_sigma_lr` and `e_phi_lr` change values the decoder never reads
(autograd computes their effect on a discarded tensor; identical loss).
This is exactly the condition the `VFEConfig.__post_init__` orphan
warning detects (`config.py:368-393`).

### Fix — per-sweep `'requires'` prerequisite overrides

Two-part edit to `transformer/vfe/vfe_ablation_suite.py`.

**1. SWEEPS registry entries (~lines 211-230).** Added a `'requires'`
dict to the orphaned entries:
```python
'e_sigma_lr': {
    ...
    'requires': {'n_e_steps': 2},
},
'e_phi_lr': {
    ...
    'requires': {'n_e_steps': 2},
},
```
`n_e_steps=2` is the smallest config change that lifts both: iteration
#2's `nat_grad_mu` reads the retracted sigma, and iteration #2's
attention/transport reads the retracted phi. Other lift conditions
(`n_layers>=2`, `use_prior_bank=True`, `norm_type='mahalnorm'`) would
be larger architectural deviations from baseline.

**2. `make_run_configs` (~line 400-430).** Read the optional `requires`
key and merge it into each generated config BEFORE the swept param is
set, so the swept param always takes precedence over the prerequisite
override:
```python
requires = sweep.get('requires', {})
...
for val in _sweep_values(sweep):
    cfg = copy.deepcopy(base_config)
    cfg.update(requires)   # prerequisites first
    cfg[param] = val       # then the swept param
    ...
```
Sweeps without `'requires'` keep the BASELINE values unchanged (no
extra compute cost for sweeps that already work).

### Cost trade

`n_e_steps=2` doubles the E-step work for the two affected sweeps
(e_sigma_lr: 8 runs, e_phi_lr: 6 runs = 14 runs total). All other
sweeps remain at `n_e_steps=1`.

### Verification

Inline smoke test (no actual training run; covers the wiring only):

- `make_run_configs('e_sigma_lr', base)` -> every run cfg has
  `n_e_steps=2` and the swept `e_sigma_lr` value preserved. PASS.
- `make_run_configs('e_phi_lr', base)` -> every run cfg has
  `n_e_steps=2` and the swept `e_phi_lr` value preserved. PASS.
- `make_run_configs('learning_rate', base)` -> `n_e_steps=1` unchanged.
  PASS.
- `VFEConfig(**lifted_cfg)` -> orphan warning silenced under

## Per-group M-step learning rates (replace `learning_rate`)

User asked for separate M-step learning rates, named in the same style
as the E-step (`e_mu_lr`, `e_sigma_lr`, `e_phi_lr`). The previous
`VFEConfig.learning_rate` was a single master scale that
`trainer.py::_build_optimizer` spread across five param groups with
hard-coded multipliers; the multipliers are now removed and each group
has its own field.

### Decisions

1. `learning_rate` removed entirely from `VFEConfig`. Five independent
   fields take its place — no master scale, no back-compat shim.
2. Naming for the catch-all bucket that holds the E-step's *learnable
   hyperparameters* (`raw_c0`, `raw_b0`, `log_kappa`,
   `_phi_preconditioner`): `m_hyper_lr`. These are M-step-trained, but
   they belong to the E-step modules — `m_hyper` is more accurate
   than `m_estep` (which would conflict with the inner-loop `e_*_lr`
   names) and broader than `m_alpha` (which doesn't cover `log_kappa`).
3. Defaults are clean values, not arithmetic preservation of the old
   `3e-4 × multiplier` products:

   | Field        | Default | Targets                                                                    |
   |--------------|---------|----------------------------------------------------------------------------|
   | `m_mu_lr`    | 1e-3    | `PriorBank.base_mu`                                                        |
   | `m_sigma_lr` | 5e-5    | `PriorBank.base_log_sigma`, `PriorBank.decode_log_scale`                   |
   | `m_phi_lr`   | 1e-4    | `PriorBank.phi_embed`, `Positional.pos_phi`                                |
   | `m_hyper_lr` | 1e-5    | `raw_c0`, `raw_b0`, `log_kappa`, `_phi_preconditioner` (when registered)   |
   | `m_other_lr` | 3e-4    | catch-all: `head_mixer.mixer_delta`, `non_flat.W_raw`, output projection   |

   The previous effective defaults (3e-4 × {3.0, 0.15, 0.3, 0.03, 1.0}
   = {9e-4, 4.5e-5, 9e-5, 9e-6, 3e-4}) are abandoned. Existing user
   configs that set `learning_rate=0.02` or `=0.05` are migrated to
   the new clean defaults — their old tuned scales are gone.

### Edits

**`transformer/vfe/config.py:189-204`.** Replaced the single field
`learning_rate: float = 3e-4` with the five fields above, each with a
trailing comment naming the parameters it touches. No `__post_init__`

## VFE theory-auditor agents — new project agents and knowledge base

User asked for a custom agent that is a resident domain expert in VFE,
active inference, the gauge VFE, information geometry, and gauge theory,
to be used both for theoretical-purity audits of the codebase and for
peer-reviewing the manuscripts in `Attention/`. Designed and built as
**two** sibling agents sharing a knowledge directory (Option B from the
brainstorming pass), rather than one agent with two modes, so that each
agent's `description` triggers cleanly under the Agent tool's routing
and the two output contracts (severity-tagged code audit vs structured
peer review with `MR-N` math-reviewer items) stay distinct.

```
.claude/agents/vfe-manuscript-reviewer.md     # the peer-review agent

  README.md                                   # index + hard constraints from CLAUDE.md
  canonical_math.md                           # F, β, Ω, sandwich, τ=κ√K, KL forms
  em_modes.md                                 # 5-mode gradient table + vfe/ profile
  e_step_constraints.md                       # E-step hygiene, LRs, SPD retraction, IFT vs amortized
  codebase_map.md                             # where each construct lives, core/ vs vfe/ differences
  manuscript_index.md                         # per-.tex synopsis + load-bearing claims
  audit_methodology.md                        # ordered audit checklist + report format
  review_methodology.md                       # ordered review checklist + review format
  style_constraints.md                        # banned phrases, banned LaTeX, pushback rule
docs/superpowers/specs/2026-05-18-vfe-theory-auditor-agents-design.md
```

### Design choices

- **Two agents not one.** Their deliverables differ (audit report vs
  peer review) and their tool surfaces differ (the manuscript reviewer
  needs `WebFetch` / `WebSearch` for citation verification; the
  auditor needs `Bash` for running targeted pytest invocations). One
  agent doing both forced an awkward branching prompt; splitting
  cleaned that up.
- **Sidecar knowledge files instead of inline.** Keeps each agent body
  small (persona + workflow + output contract), lets both agents share
  the same canonical math, and lets the user edit knowledge without
  redeploying the agent prompts.
- **Model `opus`** on both. Math verification and manuscript review
  benefit from the strongest model; 1M-context Opus lets the
  manuscript reviewer hold the canonical math + the relevant code +
  the manuscript section without summarization loss.
- **`Skill` listed in both agents' tools** so they can call `sympy`,
  `peer-review`, `literature-review`, `arxiv-database`,
  `math-skills/symbolic-computation-guide`, and
  `topology-geometry-guide` when warranted.
- **The CLAUDE.md pre-fix protocol is baked into the auditor workflow.**
  Step 2 of every audit is "open the user's actual entry point, trace
  every relevant config key through the loader and any override
  logic, state the resolved values in the report preamble" — failing
  to do this is what produces wrong findings.
- **Pushback rule encoded in both agents.** From CLAUDE.md:
  "Maintain position under pushback — ask 'What am I missing?'
  rather than capitulating." Both agent prompts explicitly tell the
  agent not to retract a finding backed by `canonical_math.md`
  evidence just because the user expresses doubt.
- **Banned Claude-isms apply to the agents' own output** too — not
  just to the manuscripts they review.

### What this does not do

- Does not auto-commit fixes. Both agents emit reports; the user
  decides what to do.
- Does not duplicate `code-reviewer`, `debugger`,
  `refactoring-specialist`, or `performance-engineer`. The auditor
  defers to those for non-theoretical concerns.
- Does not honor `em_mode` switching in the `transformer/vfe/`
  package — the agent knows about the hardwired profile and flags
  divergence rather than papering over it.

### Known limitations to monitor

- Knowledge files are a snapshot of the codebase as of 2026-05-18.
  When `vfe/e_step.py` or `core/block_config.py` change in ways that
  affect the canonical math, `canonical_math.md` and `em_modes.md`
  need updating. The agents are explicitly told to read the *current*
  code, not just the knowledge files, so they should detect drift —
  but the knowledge files themselves can become stale.
- The `Skill` tool may not be available in the agent's tool surface
  on every platform; the auditor falls back to in-prompt math
  derivation if `sympy` is unavailable.

## VFE theory-auditor agents — source-of-truth orientation flipped

User clarified the design after the initial build: the agents must not
treat the user's own CLAUDE.md, codebase, or manuscripts as sources of
truth. They must treat the **standard tools and math of the relevant
fields** as canon — Friston's free energy principle, Amari & Nagaoka's
information geometry, Nakahara / Kobayashi-Nomizu / Frankel for
differential geometry and gauge theory, Vaswani for transformer
attention, Bai-Kolter-Koltun for IFT gradients, Amari for natural
gradient, Absil-Mahony-Sepulchre for manifold optimization — and
evaluate the user's work against that canon. This is what makes them
peer reviewers rather than internal-consistency checkers.

### Why the flip matters

The first build had the agents check internal consistency between the
user's code and the user's manuscripts. That is useful but it is not
peer review — a manuscript can be internally consistent with its
codebase and still wrong against the standard literature. The flipped
design lets the agents produce findings like "the user's free energy
extends Friston's standard F with a multi-agent coupling structure
that is not in the standard FEP literature; the manuscript should
label this as a novel construction and justify it independently,
rather than presenting it as derived from FEP." That kind of finding
is the load-bearing content of peer review.

The flip does not change the project-policy constraints (sandwich
product, E-step blindness, no NN, etc.) — those remain. But the
agents now cite *Nakahara* for the sandwich, *Friston* for the E/M
separation, *Vaswani* for the √d_k scaling, etc., rather than citing
CLAUDE.md.

### `vfe-experiment-analyst`

Reads training output (CSV from `MetricsTracker`, sibling
system-info JSON with git commit, scaling-fit JSONs, `PowerLawFit`
outputs from `transformer/analysis/scaling_stats.py`). Identifies
anomalies and classifies them as **expected dynamics** (cite the
standard reference for the expectation), **implementation suspect**
(defer to `vfe-codebase-auditor`), **theoretical drift** (defer to
`vfe-manuscript-reviewer`), or **genuine result** (recommend next
experiment). Recommends experiments with explicit falsification
criteria and seed counts. Never declares significance from a single
seed.

The agent's knowledge base entry `diagnostics_glossary.md` was
written from the actual code in `transformer/analysis/` and
`transformer/training/metrics_tracking.py` — `HolonomySnapshot`
fields, `PowerLawFit` structure, `PublicationMetrics::record_*`
methods, etc. — not from invented metrics. The agent is told to
confirm each diagnostic still exists (via Grep) before citing it.

Standard interpretive priors are calibrated against the external
canon: β-entropy collapse cites Vaswani 2017's saturated-softmax
discussion, holonomy non-decrease cites Nakahara's flatness ⇔ zero
curvature, convergence-plateau cites Bai-Kolter-Koltun 2019 as
invalidating IFT-claimed gradients, geodesic deviation cites Amari
1998 natural gradient.

### `vfe-cross-manuscript-consistency`

Walks all `.tex` in `Attention/` simultaneously. Six-phase scan:
**notation drift** (`\newcommand` / `\providecommand` definitions
across manuscripts), **equation drift** (sandwich `Σ → Ω Σ Ωᵀ`,
transport factorization, attention form, F with entropy term, `τ =
κ√K`), **citation drift** (same paper with different bibkeys),
**`MR-N` status** (math-reviewer items), **style drift**
(per-manuscript counts of banned Claude-isms and forbidden LaTeX
macros), and **narrative consistency** (claim/scope/limitation
alignment across the set).

The `notation_dictionary.md` knowledge file already documents three
real drifts found via grep at build time:
- `\KL` is `\operatorname{KL}` in the GL(K) pair but `\mathrm{KL}`
  in `belief_inertia_unified.tex` and `Participatory_it_from_bit.tex`.
- `\softmax` has the same split.
- `\Tr` vs `\tr` (capitalization) differs between the GL(K) papers
  and `belief_inertia_unified.tex`.

These three drifts are listed in the dictionary as starting evidence
for the agent — when invoked, the agent will detect them again
(and any others that have appeared) and emit a structured drift
report with recommended unifications.
