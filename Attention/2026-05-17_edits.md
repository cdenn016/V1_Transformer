### Lesson

The audit's verification pipeline (`py_compile` + import smoke)
catches missing imports and syntax errors but does not exercise
function bodies. For deletion passes that remove local variables,
the appropriate check is either:

1. Search-then-trim — grep the variable name in the surviving file
   before deleting its definition; trim only if zero references
   remain.
2. Function-call smoke — actually invoke the affected forward
   path with a tiny config before declaring the audit verified.

Both regressions would have been caught by (1).

## Monotone-descent investigation (post-closure)

User asked for a follow-up on the remaining
``RuntimeWarning: E-step iter ...: F increased`` lines. The investigation
identified two layers of problem and fixed the routing layer.

**Layer 1 (FIXED) — envelope-routing of `lambda_softmax`.**

When ``include_attention_entropy=True`` (default) the monitor at
``transformer/vfe/e_step.py:377-399`` measures the manuscript
F = ``sum_j beta * KL + tau * sum(beta * log beta)``. By the envelope
theorem at the softmax stationary point of beta, the gradient
simplifies to ``dF/dtheta = sum_j beta * dKL/dtheta`` — the
softmax-coupling term ``sum_j KL * dbeta/dtheta`` cancels exactly
against the entropy-gradient term ``tau * sum_j log(beta) * dbeta/dtheta``.
Five call sites (`vfe/e_step.py:288, :344` and
`core/variational_ffn.py:1947, 1964, 2006`) were passing
``lambda_softmax=self.lambda_soft`` unconditionally, which means the
descent direction was the gradient of the entropy-suppressed surrogate
``F_surr = sum_j beta * KL`` rather than F itself. Three of those
sites are in the legacy path; two are in the production vfe path.

The fix is a one-liner per site: gate with
``lambda_softmax=0.0 if self.include_attention_entropy else self.lambda_soft``
(or the shared `_lambda_softmax_eff` variable on the legacy path).
The closed-form path (``vfe_closed_form.py``) already gates
``ffn.lambda_softmax > 0`` internally and is unaffected.

Regression coverage added at
``tests/transformer/test_vfe_package.py::TestFMonitorEnvelopeRouting``
spies on the gradient-kernel kwarg via monkeypatch and asserts both
the ``include_attention_entropy=True`` (lambda_softmax=0.0) and the
legacy ``=False`` (lambda_softmax=self.lambda_soft) pathways.

**Layer 2 (DOCUMENTED, not fixed) — query-side-only partial gradient.**

After the routing fix, F still climbs between E-step iterations on
some configurations (e.g. ``f_history = [10.48, 11.78, 13.15, 14.61]``
at conservative step sizes). The culprit is that
``compute_vfe_gradients_gpu`` returns the QUERY-side partial gradient
``partial F / partial mu_k | mu_{j != k} fixed``, per its own FD
oracle at
``tests/transformer/test_vfe_gradients.py::test_exact_diagonal_transport_fd_mu``.
The KEY-side contribution — how F changes when ``mu_k`` enters the
transported key term ``Omega_ik * mu_k`` for ``i != k`` — is not
summed in. The phi update at ``_update_phi`` (which goes through
autograd) DOES get both query and key contributions, so the phi
gradient is the total derivative. The asymmetry between the analytic
(query-side-only) (mu, sigma) gradient and the autograd (full) phi
gradient is what surfaces as monitor non-monotonicity.

This is a long-standing design choice (the BP-style synchronous-
update convention for self-attention) and is orthogonal to the
audit's lambda_softmax-routing scope. Closing it would require
either:

- extending ``compute_vfe_gradients_gpu`` with a key-side branch
  (needs a new FD oracle that perturbs ``mu_q[j]`` while ``mu_q[i]`` is
  fixed and verifies against the analytic);
- routing ``mu, sigma`` updates through autograd like
  ``_update_phi`` does (slower but matches the phi path); or
- softening the monitor tolerance / reframing it as informational.

Training still converges because the cross-entropy loss on logits is
the real M-step target, gradient accumulation averages query/key
role asymmetry, and AdamW provides momentum and adaptive step size.
The F monitor is informational.

**Final test gate (monotone-descent fix)**: ``pytest --no-header
-p no:randomly`` → **911 passed · 3 skipped · 0 failed** (+3 routing
regression tests). The `RuntimeWarning` lines persist because of
layer 2, with the routing-layer correctness now mathematically

## Manuscript fix — "message" terminology in `GL(K)_attention.tex`

The user flagged that `GL(K)_attention.tex` calls "the message" two
different things. Three parallel investigation agents
(general-purpose) plus an independent verifier confirmed the flag and
extended it to **four** distinct mathematical objects sharing the
words "message" or the symbol `$m_i$`:

1. Aggregation / posterior mean
   `$m_i = \sum_j \beta_{ij}\,\Omega_{ij}\mu_j$` —
   lines 999, 1223, 1243, 1253, 1579, 1790.
2. Euclidean gradient term
   `$\sum_j (\beta_{ij}/\sigma^2)(\Omega\Omega^\top)^{-1}(\mu_i-\Omega\mu_j)$` —
   lines 1353-1367, called "attention-weighted message aggregation"
   at line 1369.
3. Per-edge residual `$e_{ij} \equiv \mu_i - \Omega_{ij}\mu_j$` —
   line 1813, called "The message $e_{ij}$".
4. Model latent in `$\mathcal{E}_p$`,
   `$m_i \in \mathbb{R}^{K_p}$` — lines 583, 593, 604, 881.
   Unrelated to attention.

The verifier additionally found that the natural-gradient reduction
at line 1399 was overclaimed: Fisher preconditioning with
`$\Sigma_i = \sigma^2 I$` cancels only the scalar `$\sigma^{-2}$` —
the boxed update at 1379-1397 still contains
`$(\Omega\Omega^\top)^{-1}(\mu_i - \Omega\mu_j)$`. Full reduction to
the value aggregation `$\sum_j \alpha_{ij} V_j$` requires Limit~2
(absorption of `$(\Omega\Omega^\top)^{-1}$` into
`$W_Q W_K^\top$`) and Limit~3 (constant gauge `$\Omega$` absorbed
into `$W_V$`).

### Fixes applied

1. **Aggregation symbol rename** `$m_i \to \hat\mu_i$` at lines 999,
   1223, 1243, 1253, 1579, 1790. Mnemonic preserved: `$\hat\mu_i$`
   is the posterior-mean estimator of `$\mu_i$` under mixture
   responsibilities, matching the manuscript's own derivation at
   line 1220. The model-latent `$m_i$` (lines 583/593/604/881)
   is unchanged, preserving the q↔k, p↔m duality.

2. **Premature intro rewrite (line 996-1002)**. Replaced
   "The message (update) received by agent $i$ is then" with a
   forward-pointing summary that defers to §Value Aggregation:
   "The value-aggregation output received by agent $i$ is the
   posterior mean of the mixture variational model under
   responsibilities $\beta_{ij}$ (derived formally in
   Section~\ref{sec:value_aggregation})". The misleading "(update)"
   gloss is dropped — `$\hat\mu_i$` is an output, not a parameter
   update.

3. **Bridge sentence at line 1369**. Recast the Term-A clause to

## Layer 2 key-side-gradient closure (Option B autograd path)

Following the Layer 2 investigation (5 parallel investigators +
verifier, audit closure landed earlier today), the user authorized
Option B: route the `(mu, sigma)` E-step gradient through
`torch.autograd.grad` over the full manuscript F, gated behind a
new `VFEConfig.use_autograd_mu_sigma: bool = False` toggle.

### Motivation

The analytic kernel `compute_vfe_gradients_gpu` computes only the
query-side partial `dF/dmu_k |_{mu_j fixed for j != k}` (mean-field
convention, documented in `scripts/verify_vfe_gradients_fd.py:147`).
The key-side contribution — the term that arises when `mu_k`
appears as the transport key in `KL(q_i || Omega_ik q_k)` for
`i != k` — is not summed in. The `_update_phi` path uses
`torch.autograd.grad` and computes the total derivative, so the F
monitor (which evaluates the full F over all `(i, j)` pairs)
surfaces the asymmetry as monotone-descent warnings.

The analytic key-side term is
`dKL_ik/dmu_k = Sigma_k^{-1} (mu_k - Omega_ik^{-1} mu_i)`, of
comparable magnitude to the query-side. Under `Omega = I`, the two
sum to zero but neither vanishes individually.

### Changes

| File | Change |
|---|---|
| `transformer/vfe/config.py` | New field `use_autograd_mu_sigma: bool = False` with documentation of the query-side vs total-derivative tradeoff and the ~40-60% per-iteration runtime cost. |
| `transformer/vfe/e_step.py` | Threaded the flag into `VFEEStep.__init__`; gated the call site at `:338-362` to dispatch between the analytic kernel and the new autograd method; added `_compute_mu_sigma_grad_autograd` method mirroring `_update_phi`'s autograd block. |
| `tests/transformer/test_vfe_package.py` | New `TestKeysideTotalDerivative` class with four regression tests covering routing (autograd called iff flag on), default preservation (analytic called when flag off), gradient divergence (autograd `grad_mu` differs from analytic by the key-side term), and monotone descent (F descends overall when the autograd path is active). |
| `docs/audits/audit-2026-05-17.md` | Appended closure section documenting the fix. |

### Implementation summary

`_compute_mu_sigma_grad_autograd` builds the full F under
`torch.enable_grad()` with `mu_g = mu.detach().requires_grad_(True)`
and `sigma_g = sigma.detach().requires_grad_(True)`. `phi`, `mu_p`,
`sigma_p` are detached (E-step on beliefs only). The F functional
is `alpha * sum_k KL(q_k || p_k) + lambda_align * sum (beta.detach() * KL_attention)`.
At the softmax stationary point of beta, the envelope theorem gives
`dF/dtheta = sum beta . dKL/dtheta` directly — `beta.detach()`
realizes this in the autograd graph. The entropy term has no
`(mu, sigma)` dependence under detachment and is therefore omitted.

`E_learnable_alpha=True` is supported by reconstructing
`alpha_in_graph` from `mu_g, sigma_g` inside the autograd graph,
which captures the product-rule correction
`d(alpha_k)/d(mu, sigma) = -(alpha_k^2 / c0) . d(KL_k)/dtheta`
automatically (analogous to the analytic path's `alpha_c0`
correction in `compute_vfe_gradients_gpu`).

The autograd path is currently restricted to
`diagonal_covariance=True` (full-cov self-KL via autograd needs
logdet through eigendecomposition; out of scope). A clear
`NotImplementedError` is raised under full covariance.

## W_O construction RNG isolation

User reported different training results between
`use_output_projection=True` and `False` under `skip_attention=True`,
even though W_O is unreachable in that branch (the only call site,
`transformer/core/attention.py:2022-2028`, sits inside the
short-circuited `if not self.skip_attention:` block at
`blocks.py:794`).

Root cause: `nn.Linear(embed_dim, embed_dim, bias=False)` at
`attention.py:1629` consumes random draws from the shared torch
generator during Kaiming init. With `embed_dim=20` that is 400
draws per block. Every parameter initialized *after* W_O within the
same block reads from a shifted RNG offset, so toggling the flag
gives the FFN's `raw_lr`/`raw_sigma_lr` (and any later module like
the decoder linear) different random initializations from the same
seed. Same seed → different init → different trajectory.

Also confirmed: optimizer routing sends `attention.output_proj.weight`
to the `attention_params` group (`optimizer.py:750`, matched by the
`'attention' in name` check before the `'out_proj'` check), giving it
`M_attention_lr=0.013`. But W_O receives zero gradient under
`skip_attention=True`, so the LR only governs decoupled-weight-decay
drift on a parameter that is never read — does not affect training.

### Fix

`transformer/core/attention.py:1628-1640` — wrapped the W_O
construction in `torch.random.fork_rng(devices=_cuda_devices)`. The
forked RNG context captures both CPU and per-CUDA-device generator
state on entry and restores it on exit, so W_O's init still draws
fresh random values but the shared generator's downstream state is
unchanged. Effect: toggling `use_output_projection` under
`skip_attention=True` now produces bit-identical training to leaving
it off, because every other parameter initializes from the same RNG
position. The flag remains functional when `skip_attention=False`
(W_O is constructed and used normally; only the downstream RNG
side-effect is isolated).

### Verification

Not yet runtime-verified; the user should rerun their A/B (one seed,
two flag settings) and confirm the loss curves now overlap. If they
diverge after the fix, the residual cause is the
optimizer-routing/weight-decay path, not the init path, and we
chase that next.

### Not changed

- `IrrepMultiHeadAttention` is still instantiated unconditionally in
  `GaugeTransformerBlock.__init__` (i.e. the module + its unused
  parameters still live in `model.parameters()` and `state_dict`
  under `skip_attention=True`). Removing the instantiation entirely
  would clean up checkpoint pollution but is a larger architectural

- The same `nn.Linear`-shaped RNG side effect could exist for other

### Re-investigation and correct fix (same day, later session)

The prior fork_rng wrap targeted the wrong RNG draw. Four parallel
investigators plus a verifier traced the divergence to a SECOND init
pass, not the construction-time Kaiming init.

#### Mechanism

`GaugeTransformerLM.__init__` calls `self.apply(self._init_weights)` at
`transformer/core/model.py:367` AFTER all submodules are constructed.
`_init_weights` at `model.py:996-999` re-initializes every `nn.Linear`
via `torch.nn.init.normal_(mean=0.0, std=0.02)`. `apply()` is a
depth-first walk; under `use_output_projection=True` it visits
`transformer.blocks.0.attention.output_proj` (20×20 = 400 RNG draws)
BEFORE `out_proj` (50257×20 ≈ 1.0M RNG draws). Those 400 prior draws
shift the RNG cursor for `out_proj.weight` by exactly 400 samples,
producing a different vocab-head init from the same user seed.

The original fork_rng wrap at `attention.py:1628-1640` isolated only
the construction-time Kaiming init; the apply re-init pass at
`model.py:367` ran outside that fork and was the real perturbation
site. That explains why the previous fix appeared to fail empirically.

#### Verifier evidence

`scripts/_tmp_decisive_wo_test.py` builds two models under
`set_all_seeds(6)` with `EM_CONFIG` differing only in
`use_output_projection`:

| Check | Pre-fix | Post-fix |
|---|---|---|
| Shared tensors that differ at init | 1 (`out_proj.weight`, max-abs-diff = 0.14058) | 0 |
| `max\|logits_T − logits_F\|` after copying `out_proj.weight` | 0.0 | n/a (no copy needed) |
| `max\|logits_T − logits_F\|` without copy | n/a | 0.0 |
| `\|loss_T − loss_F\|` | 0.0 (post-copy) | 0.0 |
| Worst shared-param `max\|grad_T − grad_F\|` | 0.0 (post-copy) | 0.0 |
| Worst post-AdamW-step shared-tensor diff | 0.0 (post-copy) | 0.0 |
| `attention.output_proj.weight.grad` under `skip_attention=True` | None (dead weight) | None (unchanged) |

The pre-fix pair was bit-identical only after forcibly copying
`out_proj.weight` between models. Post-fix the two models are
bit-identical on shared parameters without any intervention.

#### Fix (two parts)

**Part A — `out_proj.weight` sub-seeded re-init.** After
`self.apply(self._init_weights)` at `transformer/core/model.py:367`,
re-initialize `out_proj.weight` from a sub-generator whose seed is
derived from `torch.initial_seed()` XOR with a fixed magic number.
This makes the vocab head's init a deterministic function of the
user seed alone, independent of which optional submodules
(`output_proj`, future flags) the apply traversal hit beforehand.

```python
self.apply(self._init_weights)

_out_proj_seed = (torch.initial_seed() ^ 0xC0FFEE5A11D0E4) & 0xFFFFFFFFFFFFFFFF
_out_proj_gen = torch.Generator(device=self.out_proj.weight.device)
_out_proj_gen.manual_seed(_out_proj_seed)
with torch.no_grad():
    self.out_proj.weight.normal_(mean=0.0, std=0.02, generator=_out_proj_gen)
```

Robust to `tie_embeddings=True` (tying overwrites `out_proj.weight`
immediately afterward at `model.py:377` — re-init is wasted but
harmless) and to `use_prior_bank=True` (`out_proj` is still allocated
but unused — re-init has no functional effect on decode).

**Part B — global RNG snapshot/restore around the full `__init__`.**
Part A alone was insufficient. After Part A, parameters inside the
model were bit-identical between configs, but a real training A/B
still diverged (Step 100 loss differed by 0.0007, Step 200 by 0.0039).
The cause: `__init__` still consumes ~400 extra global RNG draws when
`use_output_projection=True` (construction-time Kaiming of W_O at
`attention.py:1629`, plus the apply re-init pass at `model.py:367`),
leaving downstream consumers (DataLoader shuffling, batch sampling)
at a shifted RNG position. The two training runs therefore saw
different batches from step 0 onward.

Wrap the entire body of `GaugeTransformerLM.__init__` in a snapshot/
restore: capture `torch.random.get_rng_state()` (and per-device CUDA
states if available) immediately after `super().__init__()` and
restore at the end of `__init__`. Model construction is now
RNG-neutral for the surrounding stream — `__init__` consumes
internally for parameter init but does not advance the global cursor.

```python
super().__init__()
_rng_entry_cpu = torch.random.get_rng_state()
_rng_entry_cuda = (
    [torch.cuda.get_rng_state(i) for i in range(torch.cuda.device_count())]
    if torch.cuda.is_available() else []
)
# ... full existing __init__ body, including Part A above ...
torch.random.set_rng_state(_rng_entry_cpu)
if _rng_entry_cuda:
    for _i, _s in enumerate(_rng_entry_cuda):
        torch.cuda.set_rng_state(_s, _i)
```

Verified empirically: with Part A + Part B, the post-`__init__`
global RNG state byte-tensor is bit-identical between the two
configs (`(rng_after_T != rng_after_F).any() == False`). All
parameters, gradients, and post-step states match bit-for-bit on
shared keys as before.

#### Remaining loose ends

- Other optional `nn.Linear` submodules later added to the model could
  re-introduce the same pattern if they are inserted into the apply
  traversal between `output_proj` and `out_proj`. Under EM_CONFIG the
  attention sublayer's `output_proj` is the only optional `nn.Linear`,
  so this single fix is sufficient. If the user adds another optional
  `nn.Linear` (e.g., a learned bias projection, a feature mixer), the
  same sub-seeded re-init pattern should be applied to any downstream
  Linear whose init must be reproducibility-stable across config flips.
- `IrrepMultiHeadAttention` is still instantiated unconditionally under
  `skip_attention=True` and its `output_proj` is dead weight in the
  state dict (`grad is None`). Removing the instantiation entirely is
  a larger architectural change and remains deferred.
- `scripts/_tmp_decisive_wo_test.py` was left in place during this
  session for re-verification; it is marked `# TEMP - delete after

  share dataset, vocab_size, max_seq_len, batch_size — none of the

  sweep and every run. This avoids re-loading the multi-hundred-MB
  token cache and respawning DataLoader workers on every run,
  matching the per-sweep optimization in the core suite.
- `_set_all_seeds` is a local helper (random / numpy / torch /
  torch.cuda) so this script has no dependency on the legacy
  `transformer.training.utils` package — the `vfe/` package stays
  self-contained.
- Per-run `VFETrainer` writes its full output tree
  (`metrics.csv`, `checkpoints/`, `attention/`, `figures/`,
  `vfe_dynamics_figures/`) into `run_dir`; the suite then writes
  `ablation_result.json` and `run_result.csv` alongside.
- Per-sweep `sweep_results.csv` and `sweep_meta.json` aggregate
  across runs; cross-sweep `figures/sensitivity_summary.png` ranks
  sweeps by Val PPL range (worst − best).
- `resume=True` skips runs whose `ablation_result.json` already
  exists, identical to the core suite. Failures are caught,
  printed with traceback, and recorded with `final_ppl=inf` so
  downstream filters (`df[df['final_ppl'] < inf]`) exclude them.

### Caveats and design notes

- The suite does NOT touch `scripts/run_ablation_suite.py` — the
  user explicitly asked for a separate file under `/vfe`, which
  matches the broader convention that `transformer/vfe/` is a
  clean re-implementation independent of `transformer/core/`.
- `lambda_softmax` is exposed at the sweep-registry level (the
  user's spelling), but the per-run override writes into the
  `lambda_soft` field that VFEConfig actually owns. The sweep
  description text spells this out so future readers don't grep
  in vain.
- `phi_trace_clamp` is `Optional[float]` in `VFEConfig`. The
  default sweep list is purely numeric (no `None`); a user who
  wants to ablate the disable state can add `None` to the
  `'values'` list — it will produce a label `"phi_trace_clamp=None"`
  that lands on the bar-plot branch (non-numeric labels trip the
  `float()` cast and fall through to the categorical path).
- `rope_full_gauge` is intentionally NOT in the sweep set. With
  `diagonal_covariance=True` (the active baseline), VFEConfig
  rejects any non-`'off'` value at `__post_init__`. Adding it
  would require a coupled sweep that flips both fields together,
  which belongs in a follow-up suite if needed.
- The seed pass uses the suite-level `CONFIG['seed']` (default 6,
  matching `train_vfe.py::SEED`) and reseeds before every run, so
  cross-run drift in initialization is suppressed. To sweep seeds
  themselves, add a `'seed'` sweep with a `'values'` list and
  thread it through `_set_all_seeds` in `run_single_vfe_experiment`
  — currently the seed override is suite-global, not per-run.

## Deletion of deprecated `prior_handoff_phi` field

The `VFEConfig.prior_handoff_phi` field had carried a "Deprecated
no-op" comment for several commits: `VFEStack.forward` always wrote
`new_prior_phi = initial_priors.phi` regardless of the flag's value,
and `VFEEStep` never read `priors.phi` (phi enters the E-step from
the belief state, not the prior). The field's only behavioural
effect was a `DeprecationWarning` in `__post_init__` when set True.
The user asked for complete removal.

### Touched

- `transformer/vfe/config.py` — removed the field declaration and
  the `__post_init__` `DeprecationWarning` branch.
- `transformer/vfe/stack.py` — removed `self.prior_handoff_phi =
  cfg.prior_handoff_phi` from `__init__`; updated the module
  docstring so phi handoff is stated as "frozen at embedding"
  (matching what the code actually does) rather than gated on the
  no-op flag; tightened the inline comment at the phi assignment
  site.
- `transformer/vfe/train_vfe.py` — dropped `'prior_handoff_phi':
  False` from the live config dict.
- `transformer/vfe/vfe_ablation_suite.py` — dropped the same key
  from `BASELINE_CONFIG`.
- `tests/transformer/test_vfe_package.py` — deleted
  `test_phi_handoff`. That test passed `prior_handoff_phi=True`
  and only asserted `out.mu.shape == (B, N, K)`; it never checked
  whether phi actually propagated, so it was a vacuous test that
  would pass with the field removed entirely. Removing it is
  strictly clearer than refactoring it to assert the no-op nature.

### Verification

- `grep -r prior_handoff_phi` repo-wide: zero matches after the
  edits.
- `VFEConfig(..., prior_handoff_phi=True)` now raises
  `TypeError: VFEConfig.__init__() got an unexpected keyword
  argument 'prior_handoff_phi'`. Python's dataclass `__init__`
  helpfully suggests `'prior_handoff_rho'` as a near-miss.
- `python -m pytest tests/transformer/test_vfe_package.py` ran
  37/38 tests passing. The one failure
  (`TestExactDiagonalTransport::test_config_default_true`) is a
  pre-existing config-vs-test drift introduced by commit
  `47fb35d0 chore: training config tweaks` which flipped the
  `exact_diagonal_transport` default; the test still hardcodes the
  old value. Unrelated to this deletion.
- `transformer.vfe.vfe_ablation_suite` still loads, lists its
  registered sweeps, and constructs `VFEConfig` for every entry in
  `SWEEP_ORDER` without error.
