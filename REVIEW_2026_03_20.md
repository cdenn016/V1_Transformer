## Bugs Fixed in This Review

### 1. CRITICAL: `pyproject.toml` build-backend is invalid

**File**: `pyproject.toml:3`
**Was**: `build-backend = "setuptools.backends._legacy:_Backend"`
**Fix**: `build-backend = "setuptools.build_meta"`

`setuptools.backends._legacy:_Backend` does not exist. The package could not be installed via `pip install -e .`, causing `ModuleNotFoundError: No module named 'setuptools.backends'`. This blocked all development workflows that depend on editable installs.

### 2. HIGH: `embed_weight_decay` default contradicts docstring

**File**: `transformer/training/config.py:79-83`
**Was**: Docstring says "Default 0.0" but actual default was `0.01`
**Fix**: Changed default to `None` (inherit from `weight_decay`)

The mismatch caused **double regularization** on embeddings: both optimizer weight decay AND the VFE loss terms (`α·KL(q||p)` + `α_φ·||φ||²/2`). With `embed_weight_decay=0.01`, embeddings got an extra L2 penalty pulling them toward zero, potentially limiting expressiveness beyond what the principled VFE hierarchy intends.

### 3. MEDIUM: `BlockConfig.from_config()` key mismatch for `ffn_learnable_alpha`

**File**: `transformer/core/block_config.py:174`
**Was**: `config.get('learnable_alpha', False)` — only looked for `'learnable_alpha'`
**Fix**: `config.get('ffn_learnable_alpha', config.get('learnable_alpha', False))` — checks both keys

Configs built with the `ffn_learnable_alpha` key (matching the BlockConfig field name) would silently fail to activate Bayesian precision. Only configs using the shorter `learnable_alpha` key worked.

### 4. MEDIUM: Silent fallback to sample data in `datasets.py`

**File**: `transformer/data/datasets.py:296-303`
**Was**: `print(f"Using embedded sample data for {split}")` — easy to miss
**Fix**: Added `logging.warning()` and prominent `*** WARNING ***` messages

When all download sources fail, training silently proceeds on a ~5KB embedded text sample (duplicated 50x for train). Results are meaningless but no error is raised. Users could train for hours without realizing they're using toy data.

### 5. LOW: Deprecated BlockConfig fields lack clear markers

**File**: `transformer/core/block_config.py:38,44`
**Was**: Comments like "kept for config compat, not used by blocks"
**Fix**: Changed to explicit "Deprecated:" prefix for clarity

## Issues Documented (Not Fixed — Require Design Decisions)

### 6. MEDIUM: KL scaling by `√K` is undocumented

**File**: `transformer/train.py:474-475, 495-496, 536-537, 554-555`

All KL loss terms are divided by `math.sqrt(max(K, 1))`. This ad-hoc scaling makes loss weights dimension-dependent — the same `alpha=0.1` has different effective strength for K=64 vs K=256. No mathematical justification is documented.

**Recommendation**: Either (a) document the scaling rationale, or (b) absorb into the loss weights themselves so users don't need to account for it.

### 7. MEDIUM: Cholesky failure handling is inconsistent across modules

**Locations**:
- `train.py:302-318` — Progressive regularization loop (5 attempts, 10x increase)
- `variational_ffn.py:95-101` — One-shot pseudoinverse fallback
- `numerical_utils.py:147-154` — Raises exception

Three different strategies for the same failure mode. Should be centralized in `math_utils/`.

### 8. MEDIUM: RG metrics errors are silently swallowed

**File**: `transformer/train.py:125-126`

```python
except Exception as e:
    print(f"[WARNING] Enhanced RG metrics failed: {e}")
```

When RG computation fails, training continues without error. Users see incomplete metrics without realizing computation failed. Consider logging to file or using Python's `warnings` module.

### 9. MEDIUM: `MetricsTracker` CSV has irregular columns

**File**: `transformer/training/metrics.py:85-98`

When metrics are sparse (present in some steps, absent in others), the CSV has irregular columns. The `save()` method handles this by collecting all keys, but downstream analysis tools may trip on missing values (empty cells instead of NaN).

### 10. LOW: LR scheduler `min_ratio` uses global `learning_rate`

**File**: `transformer/training/optimizer.py:262`

```python
min_ratio = min(config.min_lr / max(config.learning_rate, 1e-12), 1.0)
```

This is actually correct (LambdaLR applies factors multiplicatively to each group's base LR), but it's confusing because `config.learning_rate` (default 3e-4) is the "simple mode" LR, not any group's actual LR. The decay ratio mu/phi stays constant throughout training. Consider adding a comment explaining this.

### 11. LOW: Cache path not cross-platform

**File**: `transformer/data/datasets.py:122-130`

Uses `Path.home() / '.cache'` which is non-standard on Windows (typically `~/AppData/Local/cache`). Could use `platformdirs.user_cache_path()` for portability.

### 12. LOW: `PureVFETransformer` is not an `nn.Module`

**File**: `transformer/pure_vfe/model.py`

Stores raw tensors instead of `nn.Parameter`. Must manually call `.to(device)` — won't auto-migrate with standard PyTorch patterns. This is by design (no autograd, no backprop) but should be documented more prominently.

### 13. LOW: Test generators aren't real SO(3)

**File**: `tests/transformer/test_attention.py:28-32`

Test fixtures create random skew-symmetric matrices, not actual SO(3) Lie algebra generators satisfying `[X_i, X_j] = ε_{ijk} X_k`. Tests validate shapes and gradients but not gauge-theoretic properties.

### 14. LOW: `numerical_monitor` events are scattered and undocumented

**File**: Multiple files via `from math_utils.numerical_monitor import record as _nr`

Event types like `_nr("chol_recover")`, `_nr("chol_fail")`, `_nr("nan_replace")`, `_nr("inv_pinv")` are scattered throughout but no central documentation lists them or explains how to access/analyze the logs.

### 15. LOW: `gauge_mode`/`use_obs_in_vfe` interaction is unclear

**File**: `transformer/training/config.py:105,147`

No documentation or validation for what happens when `gauge_mode='trivial'` AND `use_obs_in_vfe=True`. Should observations affect gauge transport when there's no transport?

### 16. LOW: Data tests marked `@pytest.mark.slow` — skipped in CI

**File**: `tests/transformer/test_data.py`

All data pipeline tests are marked slow and likely skipped in automated testing. Data loading bugs only surface in production training runs.

### 17. LOW: Training tests don't verify per-group LR values

**File**: `tests/transformer/test_training.py:84-150`

`test_param_groups_creation` checks that param groups exist but never verifies that each group has the correct LR value (mu ≠ sigma ≠ phi).

- **Robust device handling** — consistent `device = tensor.device` extraction, proper buffer registration
- **Good optional dependency fallbacks** — Numba, Triton, CuPy all degrade gracefully
- **Comprehensive gauge group support** — SO(3), SO(N), GL(K) via unified generator framework
- **18 test modules** covering core components with proper fixtures
