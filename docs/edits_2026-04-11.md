### 6. generators.py -- BCH Composition (FINDING: WRONG COEFFICIENT)

**Lines 1748-1806**: `soN_compose_bch_torch` (and `glK_compose_bch_torch` at lines 2097-2144, `lie_compose_bch_general_torch` at lines 1966-2011)

**BUG at line 1791**: The third-order BCH coefficient is -(1/12) but should be -(1/24).

The code computes:
```
result = result - (1.0/12.0) * bracket_2_1_12
```

The correct BCH formula through degree 4 is:
```
Z = X + Y + 1/2[X,Y] + 1/12[X,[X,Y]] - 1/12[Y,[X,Y]] - 1/24[Y,[X,[X,Y]]] + ...
```

The standard BCH degree-4 terms (from Rossmann, Hall, or Wikipedia) are:
- `(1/24)[Y,[X,[Y,X]]]` which, substituting [Y,X] = -[X,Y], gives `(-1/24)[Y,[X,[X,Y]]]`
- By the Jacobi identity, [X,[Y,[X,Y]]] = [Y,[X,[X,Y]]] (since [[X,Y],[X,Y]] = 0), so the full third-order correction combines to **-(1/24)[Y,[X,[X,Y]]]**, not -(1/12).

Numerical verification confirms:
- With -(1/24): convergence rate is 2^5 per doubling of scale (correct O(eps^5) residual)
- With -(1/12): convergence rate stays at 2^4 (same as order 2, meaning the coefficient fails to cancel the degree-4 error)

This same error appears in three functions:
1. `soN_compose_bch_torch` line 1791
2. `glK_compose_bch_torch` line 2136
3. `lie_compose_bch_general_torch` line 2003

**Severity**: Low in practice. The BCH is used only in retraction (phi updates), and the trust region typically keeps updates small enough that order-1 BCH suffices. The incorrect coefficient makes order >= 3 slightly worse than order 2 at moderate angles but does not cause NaN or divergence.

**BUG at line 1804 (and 2142, 2009)**: The fourth-order coefficient is also wrong. The code uses -(1/24) for [X,[Y,[Y,[X,Y]]]], but numerical fitting shows the optimal coefficient is approximately 0 or -(1/120). The degree-5 BCH has multiple terms and this single bracket does not appear alone with coefficient -1/24. The fourth-order term in the code makes accuracy worse, not better, at all tested scales.

### 7. gauge_preconditioner.py -- Cartan Projector (PASS)

**Lines 59-119**: `build_cartan_projector`

- Gram matrix G_ab = tr(T_a^T T_b): Correctly uses `einsum('aij,bij->ab', G, G)` (Frobenius inner product without transpose). Verified = I for E_ij basis.
- Trace product P_ab = tr(T_a T_b): Uses `einsum('aij,bji->ab', G, G)`. This computes sum_{i,j} G_a[i,j]*G_b[j,i] = tr(G_a @ G_b). Correct.
- Symmetric projection P_sym = G^{-1} (G + P) / 2: Verified idempotent (P^2 = P) and rank K(K+1)/2 for GL(K).
- Preconditioner C = I - (1-c)*P_sym: Eigenvalues verified as {c for sym, 1.0 for so(K)}. For K=3 with c=0.1: 6 eigenvalues at 0.1 (sym) and 3 at 1.0 (so). Correct.
- For pure SO(N) generators (all skew): P_sym = 0 (since trace_prod = -gram for skew generators), so C = I (no dampening). Correct behavior.

### 13. lie_bracket_general_torch Coordinate Extraction (MINOR ISSUE)

**Lines 1925-1963**: `lie_bracket_general_torch`

Uses diagonal Gram projection: phi_a = <G_a, bracket>_F / ||G_a||^2_F instead of the full Gram inverse phi_a = sum_b G^{ab} <G_b, bracket>_F.

This is exact ONLY for orthogonal generators (Gram is diagonal). All standard bases in the codebase (E_ij, L_{ij}) are orthogonal, so this is not a bug in practice. But the docstring at line 1938 says "which is exact for orthogonal generators (all standard bases)" -- this is correct but should be strengthened to a warning that non-orthogonal generators will give wrong results.

Numerically tested with a randomly perturbed basis: 77% relative error in extracted coordinates.
