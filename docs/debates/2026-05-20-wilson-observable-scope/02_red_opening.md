# Red Opening — wilson-observable-scope

## Steelman (opposing position)

The Wilson observable, the `holonomy_penalty` regularizer, and the `cocycle_relaxation` parameter belong to the manuscript's explicitly-labeled Regime II extension (PIFB:824, PIFB:826, PIFB:880), which sits inside the framework's gravitational/signature-related extrapolations rather than its Regime I language-modeling implementation, so their absence from `transformer/vfe/` is not a defect of the language-modeling pure path but the correct enforcement of scope; PIFB:880 itself states that in the Regime I implementation the holonomy diagnostics are degenerate, and `non_flat.py` at initialization is bitwise-identical to the flat path (`non_flat.py:73-79`, `raw_strength = 0` ⇒ `s = 0` ⇒ `δ ≡ 0`), so the language-modeling default is Regime I in fact even when the toggle is on.

## Position

The claim holds: the Wilson observable `W_ijk = Re Tr[exp(δ_ij·G) exp(δ_jk·G) exp(δ_ki·G)]`, the Wilson holonomy-penalty regularizer `S_Wilson[δ]`, and the `cocycle_relaxation` parameter α are *active language-modeling constructions in the manuscript's own description of the implementation*, are required because `transformer/vfe/non_flat.py` is a reachable runtime branch under `use_non_flat_transport=True` (`config.py:677-705`) that demonstrably leaves the Regime I subset whenever `raw_strength` and `W_raw` drift from zero under gradient descent, and are *absent* from `transformer/vfe/` — making the /vfe pure path either incomplete relative to its own manuscript or its manuscript false relative to /vfe.

## Falsification conditions

This position fails if any of the following holds:

1. Blue exhibits an existing /vfe code path — at any toggle, at any default — that materializes `W_{ijk} = Re Tr[H_{ijk}]` (or the squared-Frobenius variant `Σ ‖H_{ijk} − I‖_F²`) as a contribution to the optimized free-energy loss, not as a free-floating diagnostic.
2. Blue produces a configuration field in `transformer/vfe/config.py` that maps to PIFB:874's α homotopy parameter — i.e., a knob that scales `δ_ij` continuously, with α = 0 the bitwise-flat default. The closest analogue, `non_flat_max_strength`, is the *upper bound* on a learnable `tanh(ρ)` gate, not the α homotopy parameter; the relaxation defaults to learned, not fixed at zero.
3. Blue establishes that PIFB's "actually implemented" language at PIFB:868 and PIFB:874 refers to the wider `transformer/` tree rather than `transformer/vfe/`, *and* that the wider tree is the canonical pure path the user has accepted in the prior verdict — which would mean the original verdict already permitted `transformer/core/connection.py` and `transformer/analysis/holonomy_metrics.py` to count as canonical, contradicting the claim file's restriction.
4. Blue produces a manuscript line that withdraws or qualifies PIFB:868 ("actually implemented as the optional `holonomy_penalty` regularizer") and PIFB:874 ("The implementation exposes α as the `cocycle_relaxation` configuration parameter") sufficiently that those statements no longer constitute direct claims about the /vfe code.
5. Blue shows that `non_flat.py` is in practice unreachable in /vfe under the user's active training config — i.e., that `use_non_flat_transport=False` is binding for the runtime entry point. This would weaken the urgency, though not the manuscript-vs-code consistency gap.
