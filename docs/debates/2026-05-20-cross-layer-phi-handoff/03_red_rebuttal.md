# Red Rebuttal — cross-layer-phi-handoff

## Concession

I grant blue's load-bearing empirical fact: `priors.phi` has zero downstream readers inside `transformer/vfe/`. Independent verification:

- `transformer/vfe/e_step.py:779` reads `phi = beliefs.phi`, not `priors.phi`. The phi-mode E-step never extracts `priors.phi`.
- `transformer/vfe/e_step.py:2025` (`_forward_omega_direct`) likewise reads `phi = beliefs.phi`; `mu_p = priors.mu` and `sigma_p = priors.sigma.detach()` at lines 2027 and 2034, with no `priors.phi` extraction.
- Grep over `transformer/vfe/` for `priors.phi` returns four hits, all in `stack.py`: line 14 (docstring), line 142 (comment), line 145 (write), line 150 (BeliefState constructor). No reads.
- `transformer/vfe/e_step.py:835-838` builds `block_exp_pairs = compute_gauge_transport(phi, ...)` from the live `phi = beliefs.phi`, so Ω is constructed from the cascaded belief-side gauge, never from the prior-side slot.

The posterior gauge frame *does* cascade across layers — through the `beliefs.phi` channel that `VFEBlock.forward` returns from `e_step` and that `VFEStack.forward` (stack.py:95) passes into the next block. My opening's strict reading of stack.py:145 as "no φ-cascade" was wrong; the cascade exists, it just routes through the belief slot, not the prior slot. Within the L-block forward pass, the operative gauge state at layer ℓ+1 entry IS the converged posterior φ from layer ℓ. That part of blue's defense holds.

## Core attack

Blue's symmetry argument — "the σ slot and the φ slot are both stationary under default `(ρ_μ=1, ρ_σ=0)` and therefore equivalent" — collapses under inspection of the σ slot's actual operational role. `priors.sigma` is NOT a dead slot. `transformer/vfe/e_step.py:785` extracts `sigma_p = priors.sigma.detach()`; that `sigma_p` then enters the KL functional at line 538 (`mahal_k = delta_mu ** 2 / sigma_p_safe`) and the trace term at line 537, and feeds `get_bayesian_alpha` at line 850. The same path holds in `_forward_omega_direct` (line 2034). So `priors.sigma` is read every E-step iteration, every block, by the operational KL machinery; only its *gradient* is detached (per CLAUDE.md's "E-step reads sigma_p but must not write gradients" rule). The σ slot has a downstream consumer AND a `prior_handoff_sigma` toggle (`transformer/vfe/config.py:120`) that, when set to `1.0`, actually changes the KL ground truth at every layer ℓ+1 by blending the posterior σ into the operationally-read prior σ (stack.py:120-140). That toggle does real work.

The φ slot has neither. `priors.phi` has no downstream consumer, no `prior_handoff_phi` field in `VFEConfig`, and no equivalent mechanism. The design's own internal logic — keeping a separately-cascaded prior σ slot with a damping toggle for the case where the user *wants* canonical hierarchical-VI semantics — establishes that the framework treats "prior at level ℓ+1 = posterior at level ℓ" as a meaningful, configurable mode for distribution parameters. The φ slot has been excluded from that pattern. This is the gap. The module docstring itself acknowledges the gap at `transformer/vfe/stack.py:33-34`: "Set `prior_handoff_sigma=1.0` (and a matching mechanism for phi if desired) to recover the canonical scheme." The phrase "and a matching mechanism for phi if desired" admits the mechanism does not exist; the docstring frames its absence as future work, not as architectural inertness.

Under the canonical hierarchical-VI rule [Friston 2017; Parr-Pezzulo-Friston 2022 Ch. 4; Bishop 2006 §10.1.3] that the level-ℓ posterior becomes the level-ℓ+1 prior for *every* distribution parameter, a system that exposes a per-parameter damping toggle (ρ_μ for the mean, ρ_σ for the covariance) but omits the toggle for φ has structurally privileged μ and σ over φ. That asymmetry is the falsifying deviation: not "φ is frozen" (blue's adiabatic defense permits that), but "φ has no toggle at all to recover the canonical handoff if the user wants it" — which violates CLAUDE.md's explicit requirement (`## Hard Constraints`) that a theoretically pure path must exist under appropriate toggles. The σ slot satisfies this requirement; the φ slot does not.

Blue's PIFB:1549 adiabatic-approximation defense argues φ should be "effectively constant" across L blocks. That argument is internally consistent for the *default* configuration, but it does not address the structural asymmetry: the σ toggle exists precisely because the user might want to deviate from the adiabatic approximation and run canonical hierarchical VI. The same option is not available for φ, even though the same canon mandates the same handoff structure for both.

## Defense

My opening's strict version — "stack.py:145 alone falsifies cross-layer handoff" — was too narrow, and I withdraw the "no φ cascade" framing. The defensible weaker version, which my evidence and the user's strategic note both support, is:

(i) PIFB §1502-1551 read as a whole prescribes a hierarchical structure in which all three distribution parameters (μ, σ, φ) of the slow-variable cascade have a defined evolution channel; PIFB:1545 places φ on the slowest of the three timescales but does not exempt it from the canonical handoff. The transformer limit (PIFB:1571-1582) is the zero-dimensional point-token reduction in which τ-evolution and L-block depth collapse together; under the user's mapping of "depth = slow-variable evolution channel" (the same mapping that yields the μ cascade via `prior_handoff_rho` and the σ cascade via `prior_handoff_sigma`), the canonical hierarchical-VI prescription applies to φ as well.

(ii) `transformer/vfe/config.py:119-120` exposes `prior_handoff_rho` (for μ) and `prior_handoff_sigma` (for σ) but no `prior_handoff_phi`. `transformer/vfe/stack.py:145` hard-wires the prior slot to the embedding without a damping branch. The "matching mechanism for phi" the stack.py:33-34 docstring describes as desirable does not exist. There is no opt-in toggle by which a user can replace the embedding-frozen `priors.phi` with the posterior φ (even cosmetically, even if no downstream consumer reads it yet), nor — more importantly — extend any future downstream consumer that *does* read `priors.phi` (e.g., a future entropy regularizer on prior gauge dispersion) onto the canonical handoff branch.

The falsification is not "the cascade fails today" — blue is right that the cascade succeeds operationally via `beliefs.phi`. The falsification is "the canonical-handoff design pattern (separately-cascaded prior slot + ρ toggle + opt-in to full canonical scheme) is implemented for μ and σ and structurally absent for φ, in violation of the canonical hierarchical-VI requirement that all three parameters be handled symmetrically." This is a weaker but defensible deviation from the manuscript's canonical structure. I do not have evidence that this asymmetry causes a measurable model defect today, and I grant that blue's "operationally equivalent" defense is sound for the default configuration. The deviation is structural, not numerical.

Honest uncertainty: under blue's adiabatic argument (PIFB:1549, Born-Oppenheimer treatment of φ across the L-block fast pass), the asymmetry has zero operational effect at default settings, and the canonical-VI critique applies equally to the existing σ approximation (which the framework already admits as a known deviation in `stack.py:18-34`). If the judge weights operational equivalence and adiabatic-approximation consistency above canonical-handoff symmetry, blue's position is the stronger one. If the judge weights CLAUDE.md's "theoretically pure path must exist under appropriate toggles" requirement and the canonical hierarchical-VI symmetry across all distribution parameters, the missing `prior_handoff_phi` toggle is the falsifying gap.
