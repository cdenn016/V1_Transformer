# pure_vfe critical fixes тАФ 2026-05-18

Implemented the 5 CRITICAL findings from
`docs/audits/audit-2026-05-17-pure_vfe.md` per user directive. All
critical-tier perf and security gaps in `transformer/pure_vfe/` closed.

**Tests:** 117 / 117 pure_vfe pass; 973 / 974 full transformer suite pass
(the single failure is the pre-existing `test_config_default_true` the
user said to skip).

## #1 тАФ `torch.load` RCE default

**Was:** `model.py:240` defaulted `trusted=True` тЖТ `weights_only=False`,
enabling pickle-based RCE from any malicious checkpoint.

**Now:** `model.py:271-303` defaults `trusted=False`. The `weights_only=True`
path needs to unpickle our own `PureVFEConfig` dataclass, so the loader
calls `torch.serialization.add_safe_globals([PureVFEConfig])` before
`torch.load`. Hostile globals stay blocked; our own dataclass is
allowlisted. Falls back to `trusted=True` with a `RuntimeWarning` on
PyTorch < 2.6 (no `add_safe_globals`).

Also fixed the wrong return-type annotation `-> "PureFEPModel"` to
`-> "PureVFETransformer"` (the actual class name).

## #2 тАФ `.item()` sync storms in E-step

**Was:** `inference.py` fired 4 GPUтЖТCPU syncs per E-step iteration
(`compute_vfe` `.item()` at `:155`, three `grad_*.norm().item()` calls at
`:358, :396, :427`). With `n_esteps=12` that was 48 sync stalls per
forward pass.

**Now:**
- `compute_vfe` (`gaussians.py:140`) returns a 0-dim GPU tensor; the
  inline `.item()` is removed.
- VFE history accumulates as GPU tensors during the loop; converted to
  Python floats via a single `torch.stack(vfe_history).tolist()` after
  the loop exits.
- The divergence-check `if` block uses tensor comparison
  (`((vfe - prev) > 0.1 * (prev.abs() + 1e-10)).item()`) тАФ one unavoidable
  sync to drive the Python branch, down from four per iter.
- `diagnostics['grad_norm_*']` lists store GPU tensors; one
  `torch.stack(...).tolist()` per list at the end of the function.

Net: from 4├Ч n_esteps syncs per forward to ~1├Ч n_esteps + 4 (a 75%
reduction at the default config; more at larger n_esteps).

## #3 тАФ Cache prior-bank decode quantities

**Was:** `gaussians.py:683-684` recomputed `safe_inverse(Sig_v)` and
`safe_logdet(Sig_v)` over the **entire** prior bank
(V ├Ч K ├Ч K = ~50k inversions, V log-dets) on every forward pass тАФ
including from the M-step which calls `kl_decode_logits` to get its
CE gradient. The bank only changes during M-step writes.

**Now:**
- `kl_decode_logits` (`gaussians.py:660`) accepts optional
  `prior_Sigma_inv_bank` and `prior_logdet_bank` kwargs; computes them
  internally only when not provided (backward compatible).
- `PureVFETransformer` gained two cache attributes
  (`_prior_decode_inv`, `_prior_decode_logdet`) plus two methods:
  - `prior_decode_cache()` тАФ lazily populates and returns
    `(prior_Sigma_inv, prior_logdet)`.
  - `invalidate_prior_decode_cache()` тАФ drops the cache; called from
    the M-step after `model.prior_Sigma[update_tokens] = Sigma_new`
    writes at `learning.py:461` and `:873`.
- All three `kl_decode_logits` call sites in production code
  (`inference.py:475-481`, `learning.py:197-204`, `learning.py:646-653`)
  pull from the cache via `model.prior_decode_cache()`.

After warm-up, decode-bank inversions drop from V ├Ч forwards to V ├Ч M-steps.

## #4 тАФ Deduplicate prior-KL per E-step iter

**Was:** `state_dependent_alpha` (`gaussians.py:633`) and `compute_vfe`
(`gaussians.py:140`) both called `kl_divergence(mu, Sigma, prior_mu,
prior_Sigma)` per E-step iteration, with identical arguments. Each
call ran one full `safe_inverse(prior_Sigma)` plus two `safe_logdet`
passes.

**Now:** Both accept an optional `precomputed_kl` kwarg. `inference.py`
computes `kl_divergence(mu, Sigma, prior_mu, prior_Sigma)` **once** per
E-step iteration (line ~309) as `_kl_prior_shared`, then passes it to
both consumers. Halves the per-iter `safe_inverse(prior_Sigma)` cost.

## #5 тАФ `vfe_grad_Omega` H-loop vectorization + reuse precomp

**Was:** `gauge.py:161-243` looped over heads in Python, materializing
five `[B, N, N, K_h, K_h]` tensors per head and running two batched
inversions per head:
- `Om_inv = torch.linalg.inv(Om_h_s)` тАФ already in `precomp['Omega_inv']`
  (phantom param, never read).
- `Om_ij_inv = torch.linalg.inv(Om_ij)` тАФ explicit inversion of the full
  `[B, N, N, K_h, K_h]` tensor.
- `Sig_j_inv = safe_inverse(Sig_h_s)` тАФ already in
  `precomp['Sigma_inv']` (phantom param).

**Now:** Fully vectorized over H. Both inversions eliminated by reusing
`precomp` and an algebraic identity:
- `Om_ij = Om_i @ Om_jтБ╗┬╣` and `Om_ijтБ╗┬╣ = Om_j @ Om_iтБ╗┬╣` (no inversion
  needed тАФ the per-token inverses are already computed in
  `precompute_tokens`).
- Single matmul over `[B, H, N, N, K_h, K_h]` instead of H separate
  `[B, N, N, K_h, K_h]` matmuls.

Per-head Python loop eliminated; two redundant batched inversions per
head removed; `precomp`'s phantom parameters now actually consumed (one
of the audit's medium findings closed as a side effect).

The `kl_ij` parameter remains in the signature with an explicit
`_ = kl_ij` line for API stability; it was never read in the function
body and the call-site at `inference.py:422` passes it without
consequence (CONFIRMED-MOOT in the audit verifier).

# m_hyper_lr no-op fix тАФ 2026-05-18 (afternoon)

User reported `m_hyper_lr` had no observable effect when swept. Root cause
diagnosed and fixed across `transformer/vfe/e_step.py` + `transformer/vfe/model.py`.

## Root cause

The `m_hyper` optimizer group at `transformer/vfe/trainer.py:141` collects
three parameters when enabled: `raw_c0`, `raw_b0` (`E_learnable_alpha=True`)
and `log_kappa` (`learnable_kappa=True`). Every consumption site in the
E-step is inside an inner `torch.autograd.grad(loss, [target])` call whose
`grad_target` is restricted to `(mu_g, sigma_g)` (analytic + autograd
kernels) or `(phi_for_grad)` (`_update_phi`). The outer CE
`loss.backward()` never finds these parameters in its subgraph.

Empirical evidence (pre-fix, user's active config):

| param        | grad_norm |
|--------------|-----------|
| raw_c0       | 0.0       |
| raw_b0       | 0.0       |
| log_kappa    | 6.6e-6    |
| base_mu      | 8.3e-2    |
| phi_embed    | 2.2e-1    |

`raw_c0`/`raw_b0` were exactly zero. `log_kappa` had a tiny grad through
some path but 4 orders of magnitude smaller than M-step params тАФ m_hyper_lr
sweep at typical scales was imperceptible. Pre-existing test
`test_gradient_flows_to_log_kappa` passed because it only asserts
`grad is not None`, never magnitude.

## Fix

Added `VFEEStep._auxiliary_hyperparam_loss(mu, sigma, phi, mu_p, sigma_p, mask)`
that evaluates a per-token-normalized scalar F with all five tensors
detached and only `effective_kappa` / `get_bayesian_alpha(...)` (i.e.,
`log_kappa`, `raw_c0`, `raw_b0`) attached. Form:

```
F_aux = [alpha(c0,b0) * KL(q || p)].sum() / (B * N_q)
      + lambda_align * [
            sum beta * KL(q || Omega q)
          + tau * sum beta * log beta
          + tau * log(N_valid) * sum beta
        ].sum() / (B * N_q)
```

The `+ tau * log(N_valid) * sum beta` constant is the entropy term with
uniform prior `pi = 1/N`. Without it, the ╬║-equilibrium is shifted by
`sqrt(K) * B * N_q * log N` (manuscript `eq:free_energy_functional_final`).

`EStep.forward()` caches the result on `self._aux_hyperparam_loss` at end-
of-forward; `VFEModel.forward()` walks `self.stack.blocks` and sums the
cached values into `loss` before returning.

Gated: returns `None` when neither `E_learnable_alpha` nor `learnable_kappa`
is enabled, so configs that opt out pay zero overhead.

## Detachment discipline

The aux loss must not introduce a second gradient path into `base_mu`,
`base_log_sigma`, `phi_embed`, or the previous layer's `phi_for_grad`.
Verified by comparing M-step gradient norms before/after the fix on the
user's active config тАФ they are byte-identical:
`base_mu.grad_norm = 8.271346e-02`, `base_log_sigma.grad_norm = 1.595303e-05`,
`phi_embed.weight.grad_norm = 2.220475e-01` both pre- and post-fix.
