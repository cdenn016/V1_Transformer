# /vfe additions — 2026-05-17

Four new features added to `transformer/vfe/` per user request. Each is opt-in,
defaults preserve existing behavior. All gates that activate non-flat or
non-default behavior start at zero so a freshly-instantiated model with any of
the flags on is mathematically equivalent to the flag-off path at step 0
(modulo small floating-point drift from alternate compute orderings — see
"Numerical caveats" below).

Default-flip on 2026-05-17 (separate user request): `VFEConfig.mask_self_attention`
default changed from `True` to `False`. Rationale: under causal masking, row-0
has only the diagonal as a valid target; the old `True` default required an
"escape hatch" to avoid a fully-masked row. `False` matches the active
`train_vfe.py` configs and removes the edge case from the hot path.

## Q3 — `use_prior_bank` toggle

- `VFEConfig.use_prior_bank: bool = True` (default).
- When `False`, the decoder is `nn.Linear(K, V)` on `mu_final` (sigma is
  discarded). Encode still uses `VFEPriorBank` — only the decode boundary
  changes. This is the one documented "neural" exception in CLAUDE.md.
- Implementation: `transformer/vfe/config.py` (new field); `transformer/vfe/model.py`
  (constructs `self.output_proj = nn.Linear(K, V, bias=False)` when off, branches
  in `forward`). Cache-invalidation call is skipped when off.

## Q4 — Equivariant head mixer

- `VFEConfig.use_equivariant_head_mixer: bool = False` (default off).
- New module: `transformer/vfe/head_mixer.py` defining `VFEHeadMixer`.
- Per-irrep-type `A_t = I + Δ_t ∈ R^{n_t × n_t}` Schur-commutant mixer applied
  as `μ ↦ kron(A_t, I_{d_t}) · μ` and `Σ ↦ M Σ M^T` (sandwich product). The
  `Δ_t` parameter is stored separately from `I` so init is bitwise identity.
- Applied inside `VFEBlock.forward` after the E-step, before normalization.
- For `diagonal_covariance=True` (the default), uses the closed-form
  `σ'[m, c] = sum_n A[m, n]² · σ[n, c]` — diagonal-preserving because
  `kron(A, I_d)` only mixes the head index. Full-cov uses the exact
  per-type sandwich.
- Gauge equivariance holds under tied per-token gauges, which `/vfe` always
  satisfies (one `φ_i` per token shared across heads via block-diagonal
  generators). `maybe_warn_independent_gauges` is a single grep-point that
  would fire if a future change ever introduces per-head gauges.

## Q7 — Non-flat parallel transport (built from scratch)

Per user instruction: NOT a port of `transformer/core/connection.py` — designed
from gauge-equivariance + antisymmetry first principles.

- `VFEConfig.use_non_flat_transport: bool = False` (default off).
- Two robustness knobs: `non_flat_max_strength` (global gate amplitude),
  `non_flat_per_edge_delta_max` (per-edge Frobenius clamp on `‖δ_ij · G‖_F`).
- New module: `transformer/vfe/non_flat.py`.
- Connection: `δ_ij^a = s · (μ_i^T B^a μ_j) / d_h` where
  - `B^a = (W^a - W^{a,T}) / 2` — antisymmetric part of a per-generator
    bilinear; antisymmetry in `(i,j)` gives `δ_ji = -δ_ij`.
  - `W^a` is block-masked at construction so the connection respects
    block-diagonal gauge structure (each generator only touches its own
    irrep block).
  - `s = s_max · tanh(ρ)`, `ρ` initialized at zero ⇒ `s = 0` at init.
- Transport: `Ω_ij = exp(φ_i · G) · exp(δ_ij · G) · exp(-φ_j · G)`. Pairwise
  built per block in `compute_pairwise_omega_with_delta`, returning
  `(Ω_ij, Ω_ij^{-1})` pairs via three matrix exponentials (no explicit
  `torch.linalg.inv` in the hot path).
- KL attention: `compute_kl_attention_pairwise` consumes the pairwise Ω
  tensor; mirrors the flat kernel's masking, RoPE, and self-attention
  edge-case handling.
- Aggregation: `aggregate_beliefs_pairwise` with optional `tile_size` for
  chunked j-axis aggregation on long sequences.
- Diagnostic: `triangle_holonomy_norm` samples random `(i, j, k)` triples
  and returns the mean Frobenius norm of `Ω_ij Ω_jk Ω_ki - I`. Zero at
  init (flat), grows when the connection learns.
- Constraints (enforced in `__post_init__`): `diagonal_covariance=True`,
  `rope_full_gauge='off'`, and `use_autograd_mu_sigma=True` (the analytic
  gradient kernel is φ-only; the autograd path is required).
- Integration: new `_iter_nonflat` and `_update_phi_nonflat` methods on
  `VFEEStep` that replace the existing flat KL kernel calls with the
  pairwise kernel. The standard flat path is unchanged.

## Q8 — Pure-Ω gauge parameterization

- `VFEConfig.gauge_parameterization: str = 'phi'` (default), or `'omega_direct'`.
- Per-block `(Ω_h, Ω_h^{-1})` pair stored on `BeliefState.omega: Optional[List[Tuple[Tensor, Tensor]]] = None`
  (new optional field; default `None` preserves all existing call sites).
- New module: `transformer/vfe/omega_direct.py`.
- Update rule: right-invariant Riemannian step on `GL_+(d_h)` per block,
  `Ω_new = Ω · exp(-η · X_proj)` and `Ω_new^{-1} = exp(η · X_proj) · Ω^{-1}`
  via two batched `matrix_exp` calls. `X` is the projection of the unconstrained
  `Ω^{-1} · ∂F/∂Ω` onto `span(G^a)` via the per-block Frobenius Gram-inverse
  (built once at construction in `_build_killing_matrix_per_block`).
- Drift control: `omega_project_slk: bool` plus `omega_normalize_every: int`
  periodically rescale each block to `det = 1` (mirrors the `phi_project_slk`
  control on the algebra side). Necessary because the Killing form on
  `gl(K) = sl(K) ⊕ R·I` is degenerate on the trace direction.
- Initialization: `PriorBank.encode` runs as usual, producing `(μ, Σ, φ)`;
  `VFEModel.forward` then calls `init_omega_from_phi` once per forward pass
  to populate `beliefs.omega` from the post-`pos_enc` `φ`.
- Iteration: `VFEEStep.forward` dispatches to `_forward_omega_direct` when
  the flag is on. The φ-mode loop is untouched.
- Constraints (enforced in `__post_init__`): `diagonal_covariance=True`,
  `rope_full_gauge='off'`, `use_autograd_mu_sigma=True`.

### Known limitation for Q8

In `omega_direct` mode, `pos_enc.pos_phi_free` does not receive a gradient
from `loss.backward()`. Reason: in `omega_direct` the gauge state lives on
the group, and the E-step's iteration boundary is detached at the algebra
side. The φ-mode autograd chain that carries `pos_phi_free` through the
`_retract_phi` step has no direct analog when the state is `Ω` rather than
`φ`. This is documented and tested. To recover positional-encoding learning
under `omega_direct`, route an auxiliary `‖log Ω‖`-style loss to the M-step
or accept positional encoding as a frozen feature in this mode.

## Numerical caveats

The non-flat and omega-direct paths use the per-pair Ω kernel
(`compute_kl_attention_pairwise`) instead of the flat kernel
(`compute_kl_attention`). Both kernels compute the same mathematical
quantity (sum-of-block diagonal-of-sandwich KL between gauge-transported
Gaussians); on the same inputs they agree to ~1e-6 at fp32.

When wrapped through full-model forward passes, the cumulative drift can
reach ~0.2 in logit space because:

- The pairwise kernel materializes `Ω_ij` and squares it element-wise to
  obtain `σ_target = Σ_l Ω_ij[k, l]² σ_j[l]`.
- The flat kernel uses the factored form
  `σ_target = einsum(exp_h, exp_h, einsum(exp_neg_h, exp_neg_h, σ))`
  which has a different summation order.

The two are bitwise different but mathematically equivalent; no path is
"more correct" than the other. The drift does not affect training stability.

## Legacy mixer defect fixes (2026-05-17 follow-up)

User asked for an ultrathink audit of the legacy `transformer/core/`
`use_equivariant_head_mixer` flag (the analog of the new `/vfe` head mixer).
Three governance defects were found via parallel investigators + verifier,
and the fixes below convert each silent failure mode into a loud, traceable
signal. No mathematical changes; no behavior changes for any currently
correct config.

### Defect 1 — dead-code equivariance warning

`attention.py:1672-1690` carries a `RuntimeWarning` meant to fire when
`use_equivariant_head_mixer=True` collapses to a generic W_O under GL(K)
multi-head with independent per-head gauges. The pre-fix predicate
included `irrep_dims_override is None`, but `model.py:338` auto-populates
`ffn_irrep_dims = self._get_effective_irrep_dims(irrep_spec)` whenever
`use_block_diagonal_kl=True` (the default), so the override is never
`None` in production. The warning was dead in every real config — verified
empirically by trapping `warnings.warn`.

**Fix**: in `attention.py` replace the dead clause with a label-based check
that fires for both `irrep_dims_override=None` AND the auto-populated
`[d_head]*n_heads` case (both produce `glk_head_*` labels at lines
1466 / 1482), while staying silent for true cross-coupled super-blocks
(`glk_superblock_*` at line 1473).

```python
_independent_glk = (
    gauge_group == 'GLK'
    and getattr(self, 'glk_multihead', False)
    and any(lbl.startswith('glk_head_') for lbl in self.irrep_labels)
    and any(len(h) > 1 for _, _, h in self._mixer_groups)
)
```

Kept as `RuntimeWarning`, matching the codebase's convention for
"mathematically suspect but allowed to run" (e.g. `embeddings.py:999`,
`kl_computation.py:348`, `block_config.py:298`).

### Defect 2 — silent mutual exclusion of `use_output_projection` and `use_equivariant_head_mixer`

Both flags True allocated `mixer_params` and `output_proj`; the forward at
`attention.py:2016-2022` dispatches W_O when both exist (mixer
silently dead). Verifier grep confirmed no test or production config
currently hits this combination — zero blast radius for a hard error.

**Fix**: `block_config.py::BlockConfig.__post_init__` now raises
`ValueError` when both are True, modeled on the existing
`n_picard_steps`/`closed_form_e_step` precedent (the codebase's "allocated
but silently ignored" fail-fast pattern).

### Defect 3 — `skip_attention=True` allocates dead mixer params

`blocks.py:483` constructs `IrrepMultiHeadAttention` (with mixer)
unconditionally; `blocks.py:774` then short-circuits the attention forward
when `skip_attention=True`. `mixer_params` end up in `state_dict`, count
toward trainable param totals, attract weight decay (if any), and never
participate in math or grad.

Option (a) — set `self.attention = None` — is ruled out: `blocks.py:558`,
`587`, `629`, `669`, `672` read `self.attention.<attr>` unconditionally
in `__init__` (`constant_omega`, `rope_full_gauge_mode`,
`log_kappa_per_head`, `irrep_dims`). Option (e) — relocate mixer into the
FFN — is also ruled out: the mixer's math is post-head-concatenation;
the VFE FFN operates on `mu_q ∈ (B, N, K)` without a head split.

**Fix** (option b, the surgical one): `blocks.py:501` now passes
`use_equivariant_head_mixer=(cfg_flag and not self.skip_attention)`, so
`mixer_params` is never allocated when the attention forward will not
run. A `UserWarning` is emitted at `BlockConfig.__post_init__` so the
user's config-level intent is loud.

### Files modified

- `transformer/core/attention.py` — 1 predicate replacement at L1672-1690.
- `transformer/core/block_config.py` — 2 new `__post_init__` blocks

## Block-level equivariant mixer (`use_block_equivariant_mixer`) — 2026-05-17 follow-up

Adds a second insertion point for the Schur-commutant head mixer. The
existing `use_equivariant_head_mixer` flag wires the mixer inside
`IrrepMultiHeadAttention.forward` (post-multi-head-concat); under
`skip_attention=True` that forward is bypassed and the just-merged
fix (576e4e80) force-disables the flag. The new flag wires the same
math at a different location: `GaugeTransformerBlock.forward` after the
FFN's E-step output, before the residual + norm. The mixer is now
available in the pure-VFE (`skip_attention=True`) regime.

### Math

Same as the in-attention path. For `V = ⊕_t (V_t)^{⊕ n_t}` with irrep
type `t` of dim `d_t` and multiplicity `n_t`, the gauge-equivariant
linear maps form the Schur commutant `⊕_t M_{n_t}(R)`. We parameterize
one `A_t = I + Δ_t ∈ R^{n_t × n_t}` per type (storing `Δ_t` separately
so init is bitwise identity), assemble `M = ⊕_t kron(A_t, I_{d_t})`,
and apply

```
μ' = M μ        Σ' = M Σ Mᵀ        (full-cov: explicit sandwich;
                                    diag-cov: closed form
                                    σ'[m·d+k] = Σ_n A[m,n]² σ[n·d+k])
```

The diagonal closed form is mathematically equal to `diag(MΣMᵀ)` but
avoids the O(K²) intermediate.

### Mutual exclusions

`BlockConfig.__post_init__` raises `ValueError` for:

1. `use_block_equivariant_mixer + use_equivariant_head_mixer` — same
   operation, two locations. Stacking is almost certainly user error.
2. `use_block_equivariant_mixer + use_output_projection` — W_O is the
   gauge-breaking K→K linear that the block mixer is the principled
   replacement for; reproduces the silent-dead-weight defect already
   rejected for the attention-side pair.

### Independent-gauge warning

`BlockEquivariantMixer.__init__` emits a `RuntimeWarning` when
constructed on GLK multi-head with `irrep_labels='glk_head_*'` (the
default config without `cross_couplings`). Under independent per-head
GL(d) gauges the commutant collapses to per-head scalar diagonals and
off-diagonal `A_t` entries break equivariance the same way
`use_output_projection` does. To recover the full Schur commutant,
the user must tie gauges via `cross_couplings`.

### Files

**New**:
- `transformer/core/block_equivariant_mixer.py` — `BlockEquivariantMixer` module.
- `tests/transformer/test_block_equivariant_mixer.py` — 12 tests (identity init for both cov modes, Schur-Weyl param count parametrized, equivariance under tied gauge, diagonal closed-form matches full sandwich, block integration under both `skip_attention=True/False`, two mutual-exclusion errors, independent-gauge warning).

**Modified**:
- `transformer/core/block_config.py` — new `use_block_equivariant_mixer` flag (default False), two `__post_init__` ValueErrors, `from_config` dict-getter entry.
- `transformer/core/blocks.py` — `self.block_mixer = BlockEquivariantMixer(...)` allocation in `__init__`; post-FFN/pre-residual invocation in `forward`.

### Test results

12/12 new block-mixer tests pass. 11/11 existing `test_equivariant_head_mixer.py` tests still pass. 53/53 `/vfe` package tests still pass. 790/790 remaining legacy tests pass (3 pre-existing skips). No regressions.

### What this is NOT

- Not a remedy for the structural restriction of the commutant under
  independent per-head gauges. Under that gauge layout, the mixer is
  still restricted to per-head scalars — by Schur's lemma, no operator
  in the linear-map universe escapes this. The right unlock is
  `cross_couplings` (tied gauges within super-blocks), not a new
  operator.
- Not a gauge-equivariant `W_O`. `use_output_projection` remains the
  gauge-breaking option; this flag is the *replacement*, not a
  compatible variant.
- Not yet ablated empirically. Whether adding the block mixer improves
  perplexity is an open question (mechanism argument suggests yes
  under tied gauges; magnitude unknown).

## Attention β diagonal collapse — `mask_self_attention=True` fix

### Symptom

Run `transformer/vfe/vfe_runs/wikitext-103_20260517_190915/attention/attention_step_*.png`
shows a Kronecker-delta β: a bright yellow diagonal with dark off-diagonals at every
logged step (1000, 2000, 3000, 4000, 5000). β is therefore the identity attention map,
which carries no information and makes the message-aggregation residual a no-op.

### Root cause (two-part)

**A. Structural (sufficient on its own).** With `mask_self_attention=False` in the
active config at `transformer/vfe/train_vfe.py:79`, the diagonal logit is exactly
zero by construction:

- `Ω_ii = exp(φ_i) · exp(−φ_i) = I`, so `KL(q_i ‖ Ω_ii · q_i) = 0`.
- `logits = -kl_matrix / (kappa · √K)` at `transformer/core/attention.py:374-377`
  therefore gives `logits[i, i] = 0`.
- Every off-diagonal `KL(q_i ‖ Ω_ij · q_j) ≥ 0`, so every off-diagonal logit is
  `≤ 0`. After `softmax`, the diagonal is the largest entry of every row from
  initialization onward.
- The escape hatch exists at `transformer/core/attention.py:411-426` (gated on
  `mask_self_attention`) — it replaces the diagonal with `-inf` and re-normalizes
  — but the active config has it disabled.

**B. Dynamic — NOT confirmed by the logged metrics.** The `metrics.csv` for this
run shows `beta_mean`, `kl_mean`, `attention_entropy`, `train_loss_belief_align`
all reading `0` at steps 1, 100, 1000, 3000, 5000, and `sigma_q_*`,
`vfe_kl_pairwise_*`, `kappa_*` columns are unwritten (empty). This is because the
diagnostics dict at `transformer/vfe/e_step.py:534` is gated on
`track_layer_diagnostics`, which is off in this run. There is therefore no data
to support the claim that λ_align=10 drove off-diagonal KL up and made the
softmax sharper over time — the structural attractor (A) alone explains the
plot from step 0.

### Reference points

- β captured for the plot at `transformer/vfe/e_step.py:532`:
  `self._last_attention = beta.detach()` (raw runtime β, not rescaled).
- Plot rendered by `_plot_attention_patterns` at
  `transformer/vfe/trainer.py:279-363`; `imshow` at `:331` uses
  `vmin=0, vmax=max(b)` with no per-row normalization and no `eye()` injection.
  Upper triangle is the causal mask (light grey).
- Diagonal-masking gated path: `transformer/core/attention.py:411-426`.

### Fix (one line)

`transformer/vfe/train_vfe.py:79`:

```
- 'mask_self_attention':      False,      # Mask diagonal (prevents KL=0 self-attention)
+ 'mask_self_attention':      True,       # Mask diagonal (prevents KL=0 self-attention)
```

This eliminates the only true degeneracy. It is the documented escape hatch
already wired into `transformer/core/attention.py:411-426`. It does not modify
the gauge math.

### What was deliberately NOT changed

- `lambda_align = 10.0` — user research choice (10× the VFEConfig default of 1.0).
- `sigma_init = 0.4`, `e_sigma_q_trust` defaults — user research choices; no
  explicit σ floor is being added.
- `kappa = 1.0`, `learnable_kappa = False` — user research choices.
- `alpha = 1.0`, `alpha_divergence = 0.3`, `norm_type = 'layernorm'`,
  `n_layers = 1`, `skip_attention` (off — the attention sublayer is active in
  this config). All untouched.

### Expected outcome after the fix

- Off-diagonal β mass becomes visible in the next set of attention pngs
  (no more yellow diagonal).
- Per-row β entropy increases from ~0 toward `log(N_valid)` early in training,
  then drops as the model learns meaningful key selection.
- `train_loss_belief_align` may rise early because identity-attention was a
  free pass (β collapsed onto the diagonal where `KL(q_i ‖ Ω_ii · q_i) = 0`,
  giving a zero contribution to the alignment loss). With the diagonal masked
  out, the loss has to be paid against real off-diagonal KLs.
- CE / val loss trajectory: open question — the engineering heuristic of
  β-weighted message aggregation now has real signal to aggregate.
- Recommendation for next run: also flip `track_layer_diagnostics` on so the
  β / KL / σ columns in `metrics.csv` are actually populated; otherwise
  claim (B) above remains unverifiable.

## Train CE / PPL / BPC mis-reported under `normalize_ce_by_dim=True`

### Symptom

`figures/training_curves.png` for the 19:09:15 run shows train CE ≈ 1.2 vs.
val CE ≈ 5.8 — a 4.5× gap that's nearly exactly `√K = √20 ≈ 4.47`. End-of-run
log line `Final train PPL: 3.X` (= exp(1.21)) was being misread as a test PPL.

### Root cause

`transformer/vfe/model.py:185-200` correctly produces two scalars on the
forward path: `ce_for_log` (raw, returned third) and `loss` (raw `ce_loss`
divided by `√K` when `normalize_ce_by_dim=True`, plus the optional `mass_phi`
penalty). The training step at `trainer.py:408-411` correctly stores both
as `metrics['loss']` (combined optimizer loss) and `metrics['ce']` (raw CE),
and derives `metrics['ppl']`, `metrics['bpc']` from the raw `ce`.

The bug was downstream — when wiring these into the CSV and the publication
tracker, `metrics['loss']` (the scaled value) was being passed under the
`ce_loss` / `ce_loss_raw` keys instead of `metrics['ce']`:

- `trainer.py:548-552` — `'train_loss_ce'` and `'train_loss_ce_raw'` both
  set to `metrics['loss']` instead of `metrics['ce']`.
- `trainer.py:591-593` — `'ce_loss'` and `'ce_loss_raw'` both set to
  `metrics['loss']` instead of `metrics['ce']`.

`publication_metrics.py::TrainingTracker.record` at lines 228-237 then takes
`train_metrics['ce_loss_raw']` (now the scaled value), labels it `train_ce`,
and computes `train_ppl = exp(train_ce)` and `train_bpc = train_ce / ln 2 ·
tokens_per_char` from it. All three panels of the training-curves figure
draw from these fields, hence the consistent 1/√K offset across CE, PPL,
and BPC for the train series. Validation goes through a separate path
(`trainer.py:813-819` passes `val_metrics['val_loss']` which is already raw
avg CE) and was correct.

`summary['final_train_ppl']` at `publication_metrics.py:328` reads
`history[-1].train_ppl` — same broken `train_ppl` — so the end-of-training
log line `Final train PPL: {...}` at `trainer.py:859` was printing
