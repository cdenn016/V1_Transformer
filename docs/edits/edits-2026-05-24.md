### Tier 1 — config-validation footguns (`config.py::__post_init__`)

Added guards rejecting silent misconfiguration, mirroring the existing `n_e_steps` guard style: `n_layers < 1` (was a silent empty stack), unrecognized `gauge_group` (the `Literal` is not runtime-enforced; a typo previously fell through the `n_gen` property's SON branch and used the wrong generator count), `epistemic_samples < 1` (division by zero on the EFE path), `kappa <= 0` (meaningless temperature; crashed `math.log(kappa)` under `learnable_kappa=True`), and `phi_trace_clamp <= 0` (silently disabled det-control instead of capping). New tests: `tests/transformer/test_vfe_package.py::TestVFEConfigValidation` (7 tests).

### Tier 2 — checkpoint-load hardening (`run_semantic_clustering.py:89`)

`torch.load(..., weights_only=False)` → `weights_only=True`. The trainer checkpoint (`trainer.py:1506`) is tensors + plain dicts/numbers, so the stricter loader round-trips. New test: `tests/transformer/vfe/test_load_model_checkpoint.py`.

### Tier 3 — omega_direct stale-φ aux loss (`e_step.py::_auxiliary_hyperparam_loss`, `_forward_omega_direct`)

Under `gauge_parameterization='omega_direct'`, `phi` is frozen at encode value and the converged transport lives in `omega`, but the hyperparameter aux loss rebuilt attention from `compute_kl_attention(..., phi, ...)` (i.e. `exp(phi)` transport) — so the `log_kappa`/`raw_c0`/`raw_b0` gradient was computed against a stale transport. Added an optional `omega` parameter; when supplied, the κ term builds attention from the converged transport via the pairwise-Ω kernel (`compute_pairwise_omega_from_endpoints` + `compute_kl_attention_pairwise`, with κ attached so β's κ-dependence and the explicit τ factor both deliver the gradient), mirroring `_forward_omega_direct`'s own iteration kernel exactly (global softmax, scalar `effective_kappa`, optional non-flat δ). `omega=None` preserves the phi-mode path bit-for-bit. New tests: `test_q8_aux_hyperparam_loss_uses_converged_omega` (two transports → different aux) and `test_q8_aux_hyperparam_loss_phi_path_unchanged`.

### Tier 4 — E-step hot-path performance (`e_step.py`)

- **Skew-symmetry cache (`attention.py::_skew_cache_key`).** The 4 `.item()` GPU→CPU syncs fired on every `compute_gauge_transport` call (per E-step iteration) to fingerprint a *static* generator buffer. Skew-symmetry is invariant under `.to()/.half()`, so it is now computed **once** in `VFEEStep.__init__` (`self._generators_skew`) and passed via the existing `skew_symmetric=` argument to both call sites (804, 1396), bypassing the content-fingerprint cache entirely. NOTE: the audit's literal suggestion (key on `data_ptr`) was **rejected** — the cache comment correctly documents that pointer keying is unsafe across model lifetimes (allocator reuse); computing once and passing the verdict is the correct fix.
- **`_iter_nonflat` double pass.** A `no_grad` pass computed `beta_det/kl_det/omega_pairs_det` and the `enable_grad` pass recomputed the same values (identical inputs: `mu_g`/`sigma_g` carry the same values as `mu.detach()`/`sigma.detach()`, `phi_d==phi.detach()`, `_kappa_d==kappa.detach()`, same `cached_block_exp_pairs`). Deleted the redundant `no_grad` pass; the returned detached β/KL/Ω now derive from the autograd pass (`beta_g.detach()` etc.) — numerically identical, ~halving non-flat per-iteration transport+KL compute. The gradient path is untouched, so this does not interact with the `use_autograd_mu_sigma` accuracy concern. NOTE: the audit's literal suggestion (gate the detached pass behind monitor flags) was **rejected** — that pass produces the iteration's actual `beta`/`kl_matrix` returns (consumed at the caller, line 838), not monitor-only output; gating it would have broken the non-flat path.
- **`beta_heads` single stack** (per-head softmax branch): `torch.stack(beta_heads, dim=1)` was called twice; now stacked once and reused for both `_last_attention_per_head` and the head-mean β.

Deliberately **not** changed (documented in the audit, judged not worth the risk): `_capture_attention_state` default stays `True` — the trainer already gates the rebuild explicitly (`trainer.py:1651` sets it `False` on every train step, `1700` `True` around eval/plots), so the training hot path is already optimal; flipping the default is zero training-path benefit and risks silently disabling diagnostic capture for ad-hoc `model.forward()` callers (the documented intent at `e_step.py:350-354`). The redundant `.contiguous()` calls (952-955) were left — they copy small per-head slices and the downstream fused kernel's contiguity contract was not worth disturbing for a sub-noise gain.

## 7. Run-directory gauge-group label fix (`trainer.py`)

The end-of-training run-directory rename (`VFETrainer`, finalize path) tagged the gauge group as `GL(embed_dim)` — e.g. a 2-head GL config of head-dim 10 (`embed_dim=20`) was labeled `K=20_GL(20)`. The reduced gauge group is the block-diagonal product `GL(d_1) ⊕ ... ⊕ GL(d_B)`, so the tag must be `GL(d_h)` — the per-head block dimension — i.e. `GL(10)`. The `K=20` prefix is correct (K *is* `embed_dim`); only the `GL(...)` label was wrong.

Extracted a pure, testable helper `_gauge_group_label(cfg)` (module-level in `trainer.py`) that derives the GLK label from `sorted(set(cfg.effective_block_dims))` — joining mixed dims as `GL(1,3)` and reporting the merged `GL(d_super)` for cross-coupled super-blocks. SON/SO3 labels are unchanged (`SO(embed_dim)` / `SO(3)`). The inline branching in the rename block now calls the helper. New tests: `tests/transformer/vfe/test_run_dir_label.py` (5 tests, incl. the GL(20)→GL(10) bug case). The ablation suite routes through `VFETrainer`, so it inherits the fix.
