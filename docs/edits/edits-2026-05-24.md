## 9. Full-covariance & exact-diagonal-transport audit fixes

Acts on the focused full-cov / exact-diagonal-transport audit (`docs/audits/audit-2026-05-24-vfe-fullcov.md`: four domain experts + verifier). The audit's headline was a negative result — the full-cov belief KL is the *true* full Gaussian KL (sandwich + Cholesky + triangular solve in `core/gauge_utils.py:660-705`), not a diagonal collapse — so the fixes are the real defects the experts surfaced around it.

### D1 — config guard for the autograd + full-cov cliff (`config.py`)

`use_autograd_mu_sigma=True` + `diagonal_covariance=False` constructed cleanly then raised `NotImplementedError` mid-forward (`e_step.py` has no full-cov autograd kernel). Added a `__post_init__` guard rejecting the pair at config time. Placed after the omega_direct/non_flat promotion blocks (both already require `diagonal_covariance=True`), so it only fires on a manual flag combination. Test: `TestVFEConfigValidation::test_autograd_with_full_cov_rejected`.

### C — gauge-breaking default SPD clamp (`config.py`)

`gauge_covariant_ridge` default flipped `False → True`. The full-cov prior SPD floor must be a congruence (`ε·A Aᵀ`, `prior_bank.py:354-364`); the alternative diagonal clamp (`prior_bank.py:366-367`) is gauge-breaking under non-orthogonal Ω. Both entry points already set this `True`; this aligns the dataclass default with that intent. Inert on the default diagonal path. The conditioning probe vindicated it — `ridge=True` doubled the safe φ-scale (decode survives to scale ~40 vs ~20 for `ridge=False`). Test: `TestVFEConfigValidation::test_gauge_covariant_ridge_default_true`.

### B — full-cov exact-decode SPD robustness (`prior_bank.py::_decode_exact_full_cov`)

At large ‖φ‖ the fp32 sandwich `A diag(s) Aᵀ` loses positive-definiteness to catastrophic cancellation (genuinely indefinite, min eigenvalue ≈ −0.1 at φ-scale 40 in an empirical probe). The old single `cholesky(Σ + 1e-6·I)` cannot lift a large negative eigenvalue and raised `_LinAlgError` mid-forward. Replaced with an SPD-cone projection before the Cholesky: symmetrize → `eigh` → clamp eigenvalues to `[1e-4, sigma_max]` → reconstruct (the same primitive `stack.py:149-156` uses). Clamping **both** ends is required: bounding λ_max keeps the fp32 reconstruction well-conditioned so the 1e-4 floor survives the `V diag(λ) Vᵀ` product (an unbounded λ_max swamps 1e-4 below fp32 epsilon and leaves the matrix singular again). Test: `tests/transformer/vfe/test_full_cov_decode_conditioning.py` (RED: scale-40 decode raised `_LinAlgError`; GREEN: finite logits).

Diverged from the audit's literal B sub-recommendations, documented so they are not re-applied:
- **B(a) symmetrize the sandwich** — *not applied*. An empirical probe showed the fp32 sandwich is already bit-symmetric on this path (`torch.equal(Σ, Σᵀ)` holds at every φ-scale tested); the recommended symmetrize would be a no-op. The eigen-clamp symmetrizes anyway as its first step.
- **B(b) relative Cholesky ridge** — *superseded*. A relative ridge (`eps_small·tr/d ≈ 1e-6`) still cannot lift a −0.1 eigenvalue; only the eigen-clamp survives genuine indefiniteness.
- **B(c) `cholesky_solve` instead of `cholesky_inverse`** — *not applicable*. The explicit `Σ_p⁻¹` is flattened into a feature vector (`vec(Σ_p⁻¹)`) and consumed in the vectorized trace matmul over all vocab (`lhs @ rhs.T`); replacing it with per-vocab solves would defeat the path's single-matmul design. Forming the inverse is structurally required here.

NOTE: at φ-scale ≥ 20 the prior covariance is numerically corrupt; this fix hardens decode against runaway φ / operator misuse, it is **not** a substitute for `phi_trace_clamp` in a real training run (the logits are robust-but-not-useful at that scale).

## Investigation — full-covariance NaNs (no code changed yet)

Triggered by two training reports under `diagonal_covariance=False` (full cov), the
`transformer/vfe/train_vfe.py` config (`embed_dim=20`, `irrep_spec=[('fund',2,10)]`,
`rope_full_gauge='both'`, `gauge_covariant_ridge=True`, `sigma_init=1`, `kappa=0.5`):

1. `use_prior_bank=False`: `β: nan` while `Loss/CE/PPL` stay finite.
2. `use_prior_bank=True`: `Loss/CE/β/PPL` ALL nan from ~step 200.

Empirical repro: `scratch_nan_repro.py` (disposable) + a direct `_safe_eigh` backward test.

### Finding 1 — `β: nan` (use_prior_bank=False) is a LOGGING ARTIFACT, not a numerical NaN
Under `rope_full_gauge != 'off'` + `use_rope` + grad enabled, the E-step takes the
per-head rope-full-gauge branch at `vfe/e_step.py:898-966`, which sets
`kl_matrix = None` (that path returns only β, no per-pair KL). At `e_step.py:1207-1209`
`_last_kl_matrix` becomes `None`; `VFETrainer._attention_summary` (`trainer.py:388-402`)
hits `if beta is None or kl is None: continue` for the only block and returns `{}`; the
logger at `trainer.py:1705` then prints `attn.get('beta_kl', float('nan'))` = `nan`.
Repro Tier 1/2: beliefs finite, logits finite, `_last_kl_matrix is None`,
`_attention_summary()=={}`, and **400 synthetic train steps with zero non-finite values**.
The trigger is `rope_full_gauge != 'off'`, not full covariance. Training is healthy.

### Finding 2 — `use_prior_bank=True` everything-NaN is a real eigh-backward NaN
`torch.autograd.set_detect_anomaly` flags **`LinalgEighBackward0`**: forward loss is
finite (1.79) but `loss.backward()` produces a NaN σ-gradient (`grad_norm_sigma=nan`).
Source: the full-cov σ-retraction `retract_spd_torch` (`vfe_utils.py:534`, also 506/525)
calls `_safe_eigh(Σ, jitter=eps)`. `_safe_eigh` (`vfe_utils.py:328-339`) adds a
degeneracy breaker `linspace(0, jitter, K).diag()` to avoid the eigh-backward
`1/(λ_i−λ_j)` blow-up — but with `jitter=eps≈1e-6` and `K=20` the per-eigenvalue spacing
is `≈5.26e-8`, **below the float32 ULP at eigenvalue magnitude ≈ σ_init²=1 (1.19e-7)**.
At/near init Σ≈I (exactly degenerate), so the breaker is lost to round-off, eigenvalues
stay numerically equal, and the eigh backward divides by 0.

Direct test on exact Σ=I, K=20:
- `jitter=1e-6` → `gap_min=0`, grad has **400 NaNs**.
- `jitter≥1e-4` → `gap_min≥5.25e-6 > ULP`, grad finite, 0 NaNs.

Asymmetry explained: the linear head decodes from μ only (`∂loss/∂Σ_q=0`), so the eigh
backward is never exercised → stable. The prior-bank decode KL uses `diag(Σ_q)`
(`prior_bank.py:633,646,653`), so `∂loss/∂Σ_q≠0` flows back through the σ-retraction's
eigh and detonates the latent degeneracy NaN. (Note: `prior_bank.py:646`
`sigma_q_d.clamp(min=eps)` does NOT clean NaN — `torch.clamp(nan)=nan` — so once σ is
non-finite the whole logits row → CE nan.) The synthetic repro triggers at step 1 vs
the real run's ~step 200; the eigh-backward mechanism is confirmed identical, but the
timing difference is not pinned down (plausibly the `warmup_steps=100` LR ramp on the
real run and/or a larger initial σ-gradient from random targets — not verified).

### Finding 3 — `exact_full_cov_decode` + `gauge_fixed_priors=False` (NOT changed; see below)
The `config.py` guard is mathematically correct and was left in place. With
`gauge_fixed_priors=False` the per-token vocab prior `Σ_v` is diagonal
(`_per_token_priors`), and KL against a diagonal prior depends only on
`diag(Σ_q)` (the trace term `tr(Σ_v⁻¹Σ_q)=Σ_k Σ_q[k,k]/s_v[k]`; off-diagonal of
Σ_q contracts with the zero off-diagonal of Σ_v⁻¹). So `exact_full_cov_decode`
there would compute logits **identical** to the diagonal decode, just 10–20×
slower — a strict no-op. Removing the guard would ship a misleading no-op.
Giving it real off-diagonal mass requires building `Σ_v = exp(φ_v)·diag(s_v)·
exp(φ_v)ᵀ` from the per-token frame, which then differs from the ENCODE prior
(diagonal in direct mode) and violates Law 3 unless encode is changed too — a
new per-token full-covariance-prior parameterization, not a guard removal. Left
for separate scoping.

## Non-flat transport speedup — drop the dead `Omega_inv` (perf)

## 13. Exact full-cov decode crash on large vocab — cuSOLVER batched-eigh cap (`core/vfe_utils.py`)

Symptom (user, RTX 5090 / CUDA 12.8): with `exact_full_cov_decode=True`,
`diagonal_covariance=False`, `gauge_fixed_priors=True`, training died at step 0
in `_decode_exact_full_cov` (`prior_bank.py:521`) with
`_LinAlgError: cusolver error: CUSOLVER_STATUS_INVALID_VALUE ...
cusolverDnXsyevBatched_bufferSize(...)`. The error message's "may appear if the
input matrix contains NaN" hint is a red herring here.

Root cause (reproduced, not inferred). `_decode_exact_full_cov` projects each
per-token prior block onto the SPD cone with a batched `eigh` over the whole
vocabulary: `sigma_p_h` is `(V, d_h, d_h)` with `V ≈ 50257`. PyTorch routes
batched `eigh` on CUDA to `cusolverDnXsyevBatched`, which rejects
`batchSize ≥ 32767` (signed-int16 batch limit). A standalone repro on the user's
GPU confirmed: clean NaN-free SPD input succeeds at `V=16384`/`28000` and fails
at `V=32767`/`50000` with this exact error, whereas an *actually* NaN input
raises a *different* error ("algorithm failed to converge ... Batch element k").
`torch.linalg.cholesky`/`cholesky_inverse` on the same `(V, d_h, d_h)` do **not**
share the cap (different cusolver routine), so only the `eigh` needed fixing.
This is a pre-existing latent bug in the decode path (the bare
`torch.linalg.eigh` would have crashed identically); it surfaced now because this
was the first real run of the exact-full-cov-decode branch at GPT-2 vocab size.
It is independent of the earlier gap-regularized-backward fix (#12 / Fix A),
which only touches the backward.

Fix. `safe_eigh_backward` now chunks the batched `eigh` along the flattened
leading batch dimension when `M.is_cuda and prod(M.shape[:-2]) > 16384`
(`_EIGH_BATCH_CHUNK`, half the empirical 32767 ceiling for margin). Each chunk is
a separate `_GapRegularizedEigh.apply` call; `torch.cat` over the outputs and the
`reshape` back to the original batch shape keep the result mathematically
identical to one call and gradient-correct (each chunk is an independent autograd
node). CPU paths and small-batch CUDA callers (attention, small retractions) take
the unchanged single-call fast path, so no existing caller's behavior changes.

Verification (RTX 5090). (1) `safe_eigh_backward` forward at `V=50257` and
`V=100000` returns finite eigenpairs (was a hard crash). (2) Custom vs native
eigh gradient on a small separated-spectrum batch: `max|Δ| = 1.8e-7` (float32).
(3) Chunked (`V=32773`, ≥3 chunks) eigenvalues match a CPU `eigvalsh` reference
to `4.2e-5`; chunked backward grad finite and nonzero. (4) End-to-end: building
the user's crashing config (`exact_full_cov_decode=True`,

- Everything else: 2685 ms (67.4%)
- Ratio = **1.00 : 1.13 : 4.40**

The BWD/FWD ratio of ~1.13 means backward is slightly more expensive than forward (expected:
matrix_exp backward requires a matrix-vector product over the Frechet derivative).
Together, matrix_exp accounts for ~32.6% of total CUDA time; bmm/matmul dominates at 54.7%.

**Interpretation:** The primary bottleneck is NOT matrix_exp itself — it is the downstream
`aten::bmm` operations from the pairwise Omega construction
(`exp_phi_i @ exp_delta @ exp_neg_phi_j`, shape (B,N,N,d,d)) and the KL attention
bmm chain. Eliminating matrix_exp entirely would only recover ~32% of non-flat overhead;
the remaining ~68% is bmm over O(N^2) matrices.

**Task 4 — flat vs non-flat wall-time (B=64, N=64, 10 CUDA-synced iters after 5 warmup):**

| Path | Mean (ms) | Std (ms) |
|---|---|---|
| Flat | 78.2 | 4.3 |
| Non-flat | 235.9 | 2.2 |
| **Slowdown** | **3.02x** | — |

Absolute overhead: ~158 ms/iter at N=64. At N=128 the N^2 scaling means the overhead
grows ~4x (delta and algebra tensors scale as N^2), projecting to ~6-7x slowdown at N=128.

**Task 5 — `return_inverse=False` is already the default.**
`compute_pairwise_omega_with_delta` has `return_inverse: bool = False`. Both call sites
in `e_step.py` (lines 1886 and 1948) omit the argument, so `exp_neg_delta` (the second
`matrix_exp` per block) is never computed on the live path. The "remove Omega_inv" optimization
described in the task is already implemented. There is no second matrix_exp to remove.

### Optimization priority ranking (from measurements)

1. **bmm over (B,N,N,d,d) tensors (54.7% of CUDA time)** — the dominant cost. The only way
   to reduce this is to avoid materializing the full pairwise Omega tensor (e.g., chunked
   N-dimension processing, or reverting to the flat path which builds only (B,N,d,d) transport).
2. **matrix_exp backward (17.3%)** — detaching delta from mu in `_iter_nonflat` would eliminate
   this (the backward through `LinalgMatrixExpBackward0` comes from the autograd path through
   `non_flat_connection(mu_g)` inside `_iter_nonflat`). This is a real lever: ~17% CUDA savings.
3. **matrix_exp forward (15.3%)** — unavoidable for non-flat. The second matrix_exp
   (`exp_neg_delta`) is already skipped; no further savings possible here.
4. **Memory ops (copy_, mul, add_, fill_) combined ~18%** — secondary; reduction follows from
   reducing tensor counts, not a primary target.

## Non-flat speedup lever #2 — opt-in `nonflat_delta_through_mu` (drop the δ-through-μ E-step grad)

Motivation: non-flat measured ~7× slower than flat at the user's config (1.4 vs
10 it/s; profiler: matrix_exp bwd 17.3% of non-flat CUDA time). That backward
comes entirely from autograd tracing through `non_flat_connection(mu_g)` inside
`_iter_nonflat` (`e_step.py:1885`): since `δ = connection(μ)`, the E-step
μ-gradient flows back through `exp(δ·G)` (→ matrix_exp backward) and the
`d(Omega)/dμ` portion of the bmm chain. That is the second-order "δ moves with μ"
term in `∂F/∂μ`, ~0 early in training (`s = s_max·tanh(ρ)` starts at 0).

Change: added `VFEConfig.nonflat_delta_through_mu: bool = True` (config.py, beside
the `non_flat*` fields). When `False`, `_iter_nonflat` builds δ from `mu_g.detach()`
(`mu_for_delta = mu_g if self.nonflat_delta_through_mu else mu_g.detach()`), so the
μ-gradient no longer flows through `exp(δ·G)`. `e_step.py:__init__` reads the flag;
both `train_vfe.py` and `vfe_ablation_suite.py` CONFIG dicts expose it (the
click-to-run speed toggle). `VFEConfig(**config)` splats the dict, so dict↔dataclass
stay in sync. Default `True` = exact (canonical) path.

Why it is safe: `_iter_nonflat` is an isolated local graph — `mu_g`/`sigma_g` are
fresh `detach().requires_grad_(True)` leaves, `phi_d`/`mu_p_d`/`sigma_p_d`/`κ_d`
are detached, and `autograd.grad(F, [mu_g, sigma_g], create_graph=False)` returns
plain grads for the retraction. The connection's `W_raw`/`raw_strength` do **not**
learn here (autograd.grad requests only `[mu_g, sigma_g]`); they learn via the
M-step `m_other_lr` group (`config.py:400`). So the detach changes only `grad_mu`
(drops one term); it cannot affect parameter learning or the forward.

Verification (CPU, throwaway): mirrored `_iter_nonflat`'s pattern with the flag
ON vs OFF — (a) `beta`/`kl` **bit-identical** (detach changes the graph, not the
value); (b) `grad_sigma` identical (δ has no σ-dependence); (c) `grad_mu` differs
by max 2.45 (the dropped term is real); (d) with μ detached from δ, a
`loss.backward()` still populates `W_raw.grad` and `raw_strength.grad` (M-step
learning survives). `pytest tests/transformer/test_vfe_new_features.py` → 15
passed with the flag at its `True` default (unchanged); the lone failure
(`test_q8_omega_direct_model_forward_and_backward`) is the pre-existing
omega_direct issue documented above, unrelated. `VFEConfig(..., use_non_flat_transport=True,
nonflat_delta_through_mu=False)` constructs and exposes the field.

Real-GPU speed confirmation pending a user run with `nonflat_delta_through_mu=False`
(expect ≥17% it/s improvement; the forward and the dominant 54.7% bmm are
unchanged, so this lever's ceiling is the matrix_exp-backward + d(Omega)/dμ-bmm
share). The dominant bmm reduction remains a separate, designed task.

Note: enabling `use_non_flat_transport=True` forces `use_autograd_mu_sigma=True`
(config promotion; the analytic gradient kernel has no non-flat path) — a gradient
path previously found worse (σ-grad 20–145× too large under analytic-tuned LRs).
Flagged so the speed work is weighed against whether non-flat earns its cost.

## fix(vfe): rope_full_gauge no longer freezes phi_embed / pos_phi (M-step)

**Symptom.** A full-cov GL(10) run (`rope_full_gauge='both'`) produced an M-step
gradient figure with `μ embed` and `Σ embed` curves but no `φ embed` curve. The
plotter (`analysis/publication_metrics.py:535`, `if any(v>0)`) drops an all-zero
series, so `grad_norm_phi == 0` at every step means `phi_embed` received no
gradient and stayed frozen at its `N(0, phi_scale)` init for the entire run
(Ω = exp(φ·G) ≈ I throughout — the gauge transport was inert).

**Root cause.** `phi_embed` learns by backprop-through-the-E-step: the analytic
VFE-gradient kernels (`compute_vfe_gradients_gpu`,
`_fused_attention_and_vfe_gradients_block_diag`) write ∂F/∂μ, ∂F/∂σ as
differentiable functions of φ (through Ω), so the retracted beliefs carry φ and
the outer CE loss backprops to `phi_embed`. The `rope_full_gauge != 'off'` branch
(`vfe/e_step.py:~901`) instead routes through
`core/vfe_gradients.py::_compute_rope_full_gauge_gradient_per_head`, which (a)
detached φ (`phi_d = phi.detach()`), (b) detached the cached `exp_phi` pairs, (c)
called `torch.autograd.grad(..., create_graph=False)`, and (d) returned
`grad_mu_h.detach(), grad_sigma_h.detach()`. Any one of these severs the φ path;
all four were present. Because `rope_full_gauge != 'off'` is only reachable with
`diagonal_covariance=False` (or exact-diag), the symptom *appeared* covariance-
specific — but it is purely the RoPE-gauge knob. Confirmed empirically: with the
run config `phi_embed.grad is None`; with `use_rope=False` it is nonzero;
`gauge_fixed_priors=True` and `mass_phi>0` are independent reconnection routes.

**Fix** (`core/vfe_gradients.py::_compute_rope_full_gauge_gradient_per_head`,
local to the function). Gate on `_need_phi_graph = bool(phi.requires_grad)`:
- keep φ attached (`phi_var = phi.to(_f32)`, no detach);
- when `_need_phi_graph`, rebuild the per-head `exp_phi`/`exp_neg_phi` pair from
  the attached `phi_var` via `fused_block_matrix_exp_pairs` — a FRESH,
  function-local subgraph (NOT the shared cached pair). This is what makes the
  next item safe: the historical detach existed to stop a `retain_graph=False`
  `autograd.grad` from freeing a *shared* φ→exp_phi→Ω subgraph (double-backward /
  freed-saved-tensor errors). A fresh local subgraph cannot collide;
- `torch.autograd.grad(..., create_graph=_need_phi_graph, retain_graph=_need_phi_graph)`
  so ∂F/∂μ, ∂F/∂σ stay differentiable w.r.t. φ (second-order — unavoidable for an
  `autograd.grad`-based inner gradient; the analytic path gets first-order φ for
  free because it writes closed-form grads);
- return `grad_mu_h, grad_sigma_h` without `.detach()` (β stays detached — it is
  attention weights, not a gradient route, matching the analytic `beta.detach()`
  alignment weighting). When φ is frozen (`requires_grad=False`, e.g. eval) the
  function keeps the historical detached fast path + cache.

No downstream change needed: the nat-grad → retraction → decode chain already
carries φ for the analytic path (proven by `phi_embed.grad ≠ 0` there), so making
the rope grads φ-attached is sufficient.

**Tradeoff.** `create_graph=True` builds a second-order graph over the per-pair
full-cov KL (slogdet + Cholesky-inverse + einsums) for this opt-in path only;
activation memory roughly doubles for this kernel on large-N full-cov configs.
Acceptable for an experimental opt-in path; flagged.
