## Monotone-Descent Investigation (post-closure)

After the closure pass the `RuntimeWarning: E-step iter ...: F
increased ... Monotone descent violated` line still surfaced from
`transformer/vfe/e_step.py:407-413`. User asked for an investigation.

**Findings**:

1. **Confirmed mismatch (FIXED)**: the (mu, sigma) gradient passed to
   `compute_vfe_gradients_gpu` and
   `_compute_rope_full_gauge_gradient_per_head` used
   `lambda_softmax=self.lambda_soft` (typically 1.0) regardless of
   `include_attention_entropy`. With the entropy term in F, the
   envelope theorem at the softmax stationary point of beta gives
   `dF/dtheta = sum_j beta * dKL/dtheta` — the softmax-coupling
   term `sum_j KL * dbeta/dtheta` cancels exactly against the
   entropy-gradient term `tau * sum_j log(beta) * dbeta/dtheta`.
   The original code descended the entropy-suppressed surrogate
   `F_surr = sum_j beta * KL` while the monitor measured the
   manuscript F.

   **Fix**: when `include_attention_entropy=True`, pass
   `lambda_softmax=0.0`. Applied to:
   - `transformer/vfe/e_step.py:288` (rope_full_gauge per-head loop)
   - `transformer/vfe/e_step.py:344` (standard E-step gradient)
   - `transformer/core/variational_ffn.py:1947` (legacy rope path)
   - `transformer/core/variational_ffn.py:1964` (legacy fused path)
   - `transformer/core/variational_ffn.py:2006` (legacy standard path)

   The closed-form path (`vfe_closed_form.py`) already gates
   `lambda_softmax > 0` internally and is unaffected.

   Regression tests added under
   `tests/transformer/test_vfe_package.py::TestFMonitorEnvelopeRouting`
   verify the routing by spying on the gradient kernel kwarg.

2. **Residual non-monotonicity (DOCUMENTED, not fixed)**: even
   with envelope-correct gradient routing the F monitor continues
   to flag iteration-over-iteration increases. Root cause traced
   to a design-level partiality:
   `compute_vfe_gradients_gpu` returns the QUERY-side partial
   gradient only (`partial F / partial mu_k | mu_{j!=k} fixed`),
   per its FD oracle at
   `tests/transformer/test_vfe_gradients.py::test_exact_diagonal_transport_fd_mu`.
   The KEY-side contribution — how F changes when `mu_k` enters
   `KL_ik` for `i != k` via the transport `Omega_ik * mu_k` — is
   not summed in.

   For descent of the manuscript F (which is symmetric in
   i/j summation), the total gradient at `mu_k` should include
   both roles. The query-side-only step therefore does not
   strictly descend F; the monitor surfaces this.

   This is a long-standing design choice that pre-dates the audit
   (standard self-attention BP-style update). The transformer
   nonetheless trains successfully because (a) gradient
   accumulation over batches averages the query/key role
   asymmetry, (b) AdamW momentum + adaptive step compensates,
   and (c) the actual training target is the cross-entropy loss
   on logits, not the manuscript F. The F monitor is informational
   — its violation reflects "the natural-gradient step direction
   is the query-side partial, not the total" rather than
   "training is broken".

   **Status**: not fixed in this session. Closing it would require
   either (a) extending `compute_vfe_gradients_gpu` to compute the
   key-side contribution (substantial refactor; new FD oracle
   needed), (b) routing (mu, sigma) updates through autograd like
   `_update_phi` does (slower; loses the analytic fast-path), or
   (c) softening the monitor's tolerance / reframing its semantics.

**Test gate (after monotone-descent fix)**: `pytest --no-header
-p no:randomly` → **911 passed · 3 skipped · 0 failed** (+3 routing
regression tests). The lambda_softmax-routing fix did not eliminate
the warnings (deeper issue above), but it ensures the descent
direction matches the manuscript F at the softmax stationary point
of beta — a correctness improvement even if monotonicity remains
imperfect.

### Analytic key-side derivation (verified)

For `KL_ik = KL(N(mu_i, Sigma_i) || N(Omega_ik mu_k, Omega_ik Sigma_k Omega_ik^T))`:

```
dKL_ik / dmu_k = Sigma_k^{-1} (mu_k - Omega_ik^{-1} mu_i)
```

The cancellation `(Omega Sigma_k Omega^T)^{-1} = Omega^{-T} Sigma_k^{-1} Omega^{-1}`
collapses `Omega^T Omega^{-T} = I`, leaving an inverse-pulled-back
residual whitened by `Sigma_k^{-1}`. The query-side derivative is
related by `dKL_ik/dmu_k = -Omega^T dKL_ik/dmu_i`, so **the two
contributions are of comparable magnitude**.

**Special case Omega = I**: query and key terms sum to zero (SymPy
verified) but neither vanishes individually. The bug is therefore
testable even in trivial-gauge: two tokens with `mu_1 != mu_2`,
`Omega = I`, `beta = 1` would expose the discrepancy via end-to-end
F evaluation. The +1/iter F ascent observed in production
(`[10.48, 11.78, 13.15, 14.61]`) is consistent with the magnitude
of the missing term.

### Caveat

the kernel's query-side-only structure as **"the mean-field

not an unrecognized bug. Whether to promote it to a full
correctness fix or retain it as a mean-field approximation is a
scientific judgment, not a code-correctness call. The recommended
implementation path (Option B, autograd gated behind a toggle)

### Implementation details

`_compute_mu_sigma_grad_autograd` mirrors the existing
`_update_phi` pattern at `vfe/e_step.py:548-601`:

1. `with torch.enable_grad():` — defensive against `torch.no_grad()`
   contexts.
2. `mu_g = mu.detach().requires_grad_(True)`,
   `sigma_g = sigma.detach().requires_grad_(True)`.
3. Detach `phi`, `mu_p`, `sigma_p` (E-step on beliefs only).
4. Build full F: `self_kl_term + alignment_term` where
   - `self_kl_term = alpha * sum_k KL(q_k || p_k)` (diagonal Gaussians
     via closed-form formula).
   - `alignment_term = lambda_align * sum (beta.detach() * KL_attention).sum()`
     — `beta.detach()` invokes the envelope theorem so the entropy
     term has no `(mu, sigma)` dependence under detachment and is
     omitted.
5. `E_learnable_alpha=True`: reconstruct `alpha_in_graph` from
   `mu_g, sigma_g` inside the autograd graph; the product-rule
   correction `d(alpha_k)/d(mu, sigma) = -(alpha_k^2/c0) . d(KL_k)/dtheta`
   is captured automatically (matches the analytic path's
   `alpha_c0` correction).
6. `torch.autograd.grad(F_total, [mu_g, sigma_g])` returns the
   total derivative including BOTH query-side and key-side
   contributions.

Constraint: currently restricted to `diagonal_covariance=True`.
Full-covariance self-KL requires `logdet` through eigendecomposition,
which is out of scope for this fix. A clear `NotImplementedError`
is raised when the toggle is enabled under full covariance.

The closed-form path `vfe_closed_form.py` retains the same
query-side-only stationary condition; fixing that would require
a different (Newton-step) design and is deferred.

### Adoption guidance

The toggle defaults to `False` to preserve bit-identical behavior
for existing checkpoints and the production fast path. Users who
want strict monotone descent of the manuscript F (e.g., for
debugging, theory verification, or research where convergence
quality matters more than per-step speed) should set
`use_autograd_mu_sigma=True` in their `VFEConfig`. The runtime
penalty is approximately +40-60% per E-step iteration; verify
empirically on the target hardware before promoting it to a
default.

## SUPERSEDED — 2026-05-19

The Adoption Guidance above ("users who want strict monotone descent
should set `use_autograd_mu_sigma=True`") is wrong and has been
superseded by the Wave 6 diagnostic in `post_edit_2026-05-19.md`. The
Layer 2 framing that called the analytic kernel's query-side-only
gradient a "key-side bug" was also a mis-framing. Concretely:

1. **The analytic kernel is canonical, not buggy.** The query-side
   partial `∂F/∂(μ_i, σ_i)` holding all other `q_j` fixed is the
   mean-field variational message-passing gradient — exactly what
   Beal 2003 Ch. 2, Winn & Bishop 2005, and Friston 2010 prescribe
   for a mean-field q with coupling KLs. The earlier audit treated
   the missing key-side term as a correctness defect; it is in fact
   a structural feature of mean-field VMP. The `verify_vfe_gradients_fd.py`
   self-description "the mean-field convention of the analytic
   gradient" (cited in the Caveat above) was the correct reading.

2. **The autograd path is non-canonical, not "fully principled."**
   `torch.autograd.grad(F_total, [μ, σ])` computes the *total*
   derivative of F. Under parallel q-updates (which is what the
   E-step loop does) this double-counts asymmetric couplings: each
   q_i receives both its incoming-message gradient and the gradient
   it should be sending to q_j in the same step.

3. **The F-monotone monitor warning is expected, not a bug.**
   Parallel mean-field VMP is *not* guaranteed to produce monotone
   F decrease per outer iteration; only fully sequential coordinate
   descent has that property (Jordan et al. 1999 §6). The
   `[10.48, 11.78, 13.15, 14.61]` per-iter ascent observed above
   is consistent with parallel-update behaviour at finite LR, not
   with a "missing gradient term." Silencing the warning by routing
   to the total-derivative gradient (Option B) cured a symptom that
   was not a disease.

4. **Empirical magnitude check (2026-05-19 probe at
   `scripts/diagnostic_autograd_gradient_scale_probe.py`).** At the
   production config (K=20, GL(20), λ_align=4, σ_init=0.4), the
   autograd-vs-analytic gradient RMS ratio is 1.2–1.8× on μ but
   19.6 → 144.6× on σ across 6 iterations, with the σ key-side
   contribution one-signed (mean −5.65 → −15.58). The Adoption
   Guidance implicitly assumed the two gradients agreed up to a
   small correction; they do not. The σ direction is partly
   orthogonal (per-token cosine reaches −0.3 in the tail), so
   re-tuning `e_sigma_lr` cannot recover analytic-path behaviour
   from the autograd path.

### Replacement guidance

- **Default `use_autograd_mu_sigma=False` is correct.** Keep it.
- The flag remains opt-in for research into non-canonical gradients
  but should not be promoted to a default and should not be presented
  as "the principled path."
- The docstring on `_compute_mu_sigma_grad_autograd` was rewritten
  2026-05-19 to reflect this; the F-monotone monitor docstring and
  warning message were updated to note that per-iteration violations
  are expected under parallel mean-field VMP and only large or
  persistent violations indicate a real LR / conditioning issue.
- The "Option B" recommendation in the cost-analysis table above
  remains a *valid* engineering option for researchers who want the
  total-derivative gradient; it is not the appropriate default.
- The closed-form path `vfe_closed_form.py` keeps the canonical
  query-side stationary condition; no fix needed there.
