# Post-Edit Log — 2026-05-18

## Issue investigated

User reported that sweeping `e_sigma_lr` and `e_phi_lr` in
`transformer/vfe/vfe_ablation_suite.py` produced no measurable effect on the
final loss. Three parallel investigation agents plus a verifier traced this
to **orphaned E-step retractions** under the suite's BASELINE_CONFIG:

- `n_e_steps = 1`
- `n_layers = 1`
- `use_prior_bank = False`
- `norm_type = 'layernorm'`  (σ-blind via `_LayerNormSigmaAdapter.forward`)

Under that combination the E-step inner loop in `transformer/vfe/e_step.py`
runs once with order `μ-update → σ-retract → φ-retract` (lines 665–696)
and then exits. The returned `sigma_new` and `phi_new` are autograd leaves
that no downstream operation reads: `model.py:183` decodes
`logits = output_proj(mu_final)` (σ, φ discarded), `block.py:49-50`
LayerNorm-adapter drops σ, and there is no next iteration or next block to
consume them.

Smoking-gun evidence:
`transformer/vfe/vfe_ablation_results/e_sigma_lr/sweep_results.csv` and
`.../e_phi_lr/sweep_results.csv` show `final_ppl = 325.1898297631453`
bitwise-identical across all 8 (σ) and 6 (φ) sweep rows.
The companion `e_mu_lr/sweep_results.csv` varies smoothly
(326.020 → 325.313) — μ-LR is on the autograd path; σ/φ-LRs are not.

## Lift conditions

Different for σ and φ:

| What reads it downstream                          | Lifts σ-orphan | Lifts φ-orphan |
|---------------------------------------------------|----------------|----------------|
| `n_e_steps >= 2` (next iter's `nat_grad_mu(μ,σ)`) | yes            | yes            |
| `n_layers >= 2` (next block's `compute_kl_attention(μ,σ,φ)`) | yes  | yes            |
| `use_prior_bank = True` (`decode(μ_q, σ_q, τ)`)   | yes            | no             |
| `norm_type in {'mahalnorm','centered_mahalnorm'}` | yes            | no             |

`prior_bank.decode` at `prior_bank.py:292-297` takes `(mu_q, sigma_q, tau)`
only — φ is not in the signature. No norm reads φ.

## Change

`transformer/vfe/config.py` — appended an orphan-detection warning at the
end of `VFEConfig.__post_init__` (after the `alpha_divergence` warning, before
the `irrep_dims` property). The warning:

- Fires when either σ- or φ-orphan condition is detected.
- Lists exactly which LRs are dead (`e_sigma_lr`, `e_phi_lr`, or both).
- States the full lift conditions for each so the user can pick a switch.
- Uses `UserWarning` with `stacklevel=2`, matching the existing
  `alpha_divergence` warning style; Python's default warning filter
  dedupes per source location so 60+ sweep rows fire it once.

Verified on three combinations:
1. baseline orphan → warns about `e_sigma_lr, e_phi_lr`
2. `use_prior_bank=True` → warns about `e_phi_lr` only (σ lifted)
3. `n_e_steps=2` → no warning (both lifted)

## What this change does NOT do

The warning makes the trap visible at config construction time. It does
**not** silently change the user's BASELINE_CONFIG (per CLAUDE.md
"LOCALLY DEFINED CONFIGS" and the `feedback_no_lenient_config_aliases`
memory). To actually make `e_sigma_lr` and `e_phi_lr` move the loss, the
user must change one of the four switches in their config — the warning
text enumerates the options.

Also untouched:
- The E-step inner loop order (`μ → σ → φ`) — reordering would change
  algorithmic semantics; out of scope for a "fix issues" pass.
- The whitelist filter at `vfe_ablation_suite.py:340-342`
  (`_strip_to_vfe_kwargs`) that silently drops unknown keys
  (e.g., `ffn_n_iterations`, `use_residual`, `evolve_sigma`,
  `evolve_phi_e_step`, `gauge_dim`, `em_mode`, ...). These legacy/driver
  keys in BASELINE_CONFIG never reach VFEConfig; whether to warn or
  hard-error on them is a separate question.

### Issue 2 — Rogue `transformer/vfe/outputs/figures/` from `train_vfe.py`

User reported a "dead no-op" `/output` folder appearing on every
`train_vfe.py` run. Traced to a default-argument bug in
`transformer/analysis/publication_metrics.py:198-199`:

```python
self.save_dir = Path(save_dir) if save_dir else Path("./outputs/figures")
self.save_dir.mkdir(parents=True, exist_ok=True)
```

`VFETrainer._init_metrics` at `transformer/vfe/trainer.py:644-654` was
constructing `TrainingTracker` **without** a `save_dir` argument, so the
constructor fell back to `./outputs/figures` relative to CWD. Running
`train_vfe.py` from `transformer/vfe/` therefore created
`transformer/vfe/outputs/figures/`. The directory was never written to:
the VFE trainer's actual figures land in `vfe_runs/{DATASET}_{timestamp}/figures`
via a separate `PublicationFigures(fig_dir)` instance (trainer.py:768-770),
and `VFETrainer` never calls `_pub_tracker.save_json`/`save_csv`.

Confirmed by directory listing: `transformer/vfe/outputs/figures/` existed
and was empty.

### Fix

Surgical patch at `transformer/vfe/trainer.py:647-650` — pass an explicit
`save_dir` rooted at the trainer's actual `output_dir`:

```python
self._pub_tracker = TrainingTracker(
    save_dir=(self.output_dir / 'figures') if self.output_dir else None,
    tokens_per_char=tokens_per_char_from_dataset(...),
)
```

When `output_dir` is `None` (no-disk smoke tests), `save_dir=None` still
hits the fallback and creates `./outputs/figures` — but those code paths
are explicitly opting out of disk persistence and the mkdir is the
existing pre-patch behavior, so this is no regression. The user-facing
bug (CWD pollution under normal `train_vfe.py` runs) is gone because
`self.output_dir` is always set in the `train_vfe.py` flow
(`output_dir = f'vfe_runs/{DATASET}_{timestamp}'` at line 184).

Removed the leftover empty `transformer/vfe/outputs/` to verify the new
run won't recreate it under the same cwd.

### Root-cause fix — lazy `mkdir` in `publication_metrics.py`

After the surgical patch above, user asked to "fix all". `PublicationFigures.__init__`
at `publication_metrics.py:370-372` had the identical default-arg bug —
unconditional `mkdir(./outputs/figures)` on construction — that doesn't
manifest in `VFETrainer` (always called with explicit `fig_dir`) but
could bite any other caller that constructs without an arg and never
plots.

Replaced the eager `mkdir` in both `TrainingTracker.__init__` and
`PublicationFigures.__init__` with a lazy property:

```python
def __init__(self, save_dir: Optional[Path] = None, ...):
    self._save_dir = Path(save_dir) if save_dir else Path("./outputs/figures")
    # ... no mkdir here

@property
def save_dir(self) -> Path:
    self._save_dir.mkdir(parents=True, exist_ok=True)
    return self._save_dir
```

The directory is now created on first read of `self.save_dir`. Read sites
are exclusively write contexts (`save_json`, `save_csv`, every `savefig`
in `PublicationFigures`), so the directory only materializes when
something actually writes. Constructed-but-unused trackers/figures
leave no trace.

Audit: only `__init__` sites assigned to `self.save_dir` — no external
mutation — so the property-only access pattern is safe. `tests/transformer/
test_publication_metrics.py` (8 tests) all pass post-patch.

Sanity-check confirming lazy behaviour:
- `TrainingTracker()` construction → no `./outputs/figures` created
- `PublicationFigures()` construction → no `./outputs/figures` created
- First `tracker.save_dir` read → directory materializes

### Batch E — VFE fused-path wiring (the architectural fix)

The single largest performance win. `_fused_attention_and_vfe_gradients_block_diag` (declared at `transformer/core/vfe_gradients.py:962`) was already used by the legacy `transformer/core/variational_ffn.py:1969` path but **had no caller in `transformer/vfe/`**. `VFEEStep.forward` was instead calling `compute_kl_attention` followed by `compute_vfe_gradients_gpu`, each rebuilding `Omega` from `block_exp_pairs` — a duplicate `(B, N, N, d, d)` einsum per block per iteration.

- `transformer/vfe/e_step.py:30, 656-748` — added `_can_use_fused` gate (`is_diagonal AND irrep_dims is not None AND not exact_diagonal_transport AND not use_autograd_mu_sigma`) mirroring the rope-branch's existing gating. When all conditions hold, route through the fused kernel; otherwise fall back to the unfused `compute_kl_attention` + `compute_vfe_gradients_gpu` pair (autograd μ/σ, exact-diagonal transport, full-cov, or unspecified `irrep_dims`).
- `_update_phi`'s third Omega build (`vfe/e_step.py:896-905`) genuinely cannot share the cache — it needs a fresh `phi_for_grad` autograd leaf — so the realised win is 3× → 2× Omega per E-step iteration, not 3× → 1×.
- `tests/transformer/test_vfe_package.py:597-674, 758-810, 1008-1082` — five existing tests monkeypatched `compute_vfe_gradients_gpu` to capture `lambda_softmax` / `alpha_c0` / call counts. Updated all five to also patch `_fused_attention_and_vfe_gradients_block_diag`, then assert against the union of captures. Test semantics unchanged; implementation-level coupling broadened.

### Batch F — documentation: implementation notes for F1/F2/F3 + decode + argparse

Three claims from the vfe-codebase-auditor were structural facts about the package, not bugs. Per the audit framing ("either implement or document"), and to avoid behaviour changes the user hadn't opted into, all three were documented as module-level implementation notes:

- `transformer/vfe/e_step.py` — module docstring expanded with a five-term-vs-three-term breakdown: only `α·KL(q‖p)`, `Σβ·KL`, and the optional `τ·β·log(β/π)` entropy term are realised. The γ-coupled model-coupling and `λ_h·KL(s‖h)` terms from the manuscript canonical F are not implemented in this package; "model coupling" is realised instead via the cross-layer cascade.
- `transformer/vfe/model.py` — module docstring documents that the outer M-step minimises `ce_loss + 0.5·mass_phi·‖φ‖² + Σ aux_hyperparam_loss`, not the converged-q value of F. The α·KL and β·KL terms enter only as gradients through unrolled E-step iterations — this is amortised inference, not classical alternating E/M on the same functional.
- `transformer/vfe/stack.py` — module docstring documents that the default cross-layer cascade is **mean-only**: posterior μ flows, posterior σ and φ are discarded. This is point-estimate handoff, not the full distributional handoff that canonical hierarchical VI prescribes. The codepath for full handoff exists (set `prior_handoff_sigma=1.0`); default is point-passing.
- `transformer/vfe/prior_bank.py:120-128` — `decode_log_scale` now has a clear comment: the canonical decode is `−KL/τ`; this module computes `−c·KL/τ` with a learnable `c = exp(decode_log_scale)`, clamped to `[0.05, 20]`, initialised at 0 so c=1 at start.
- `transformer/train_publication.py:814-822` — added the explicit `DOCUMENTED EXCEPTION` block above the `argparse.ArgumentParser(...)`. The CLI is grandfathered for publication-mode-switch convenience; new entry points must still follow the click-to-run pattern.

(`transformer/checkpoints_publication/.../metrics.csv`) being row-gated to
`log_interval=200` with rich column population. A console warning surfaced
the actual proximate cause:

```
VFE dynamics columns could not be written to CSV (log_step rejected extra
kwargs: PublicationMetricsTracker.log_step() got an unexpected keyword
argument 'sigma_q_mean'). Subsequent warnings suppressed.
```

Two parallel investigation agents plus one verifier mapped four overlapping
defects in the vfe training path:

1. **Cadence bug** — `transformer/vfe/trainer.py:923-930` called
   `_log_to_csv`, `_log_vfe_dynamics_to_csv`, and `_log_to_pub_tracker`
   inside the per-step loop with no `log_interval` gate. `log_interval=200`
   from `train_vfe.py:126` was consulted only at `trainer.py:961-962` for
   console output.
2. **Silent crash in the extra-columns path** — `_log_vfe_dynamics_to_csv`
   (`trainer.py:795-844`) tried to push `sigma_q_mean`/`alpha_*`/`kappa_*`
   keys to the tracker via `log_step(step, **vfe_row)`, but
   `PublicationMetricsTracker.log_step` (`metrics_tracking.py:113`) has a
   fixed positional signature with no `**kwargs`. Every call raised
   `TypeError`, caught + once-only warned, then suppressed for the rest of
   training.
3. **Key-namespace mismatch** — `PublicationMetricsTracker.log_step` reads
   most diagnostic columns via prefixed lookups (`bayesian/alpha_mean`,
   `cov/sigma_q_mean`, `transport/phi_norm_mean`, `kappa/per_head_mean`),
   but the vfe E-step writes bare names (`alpha_mean`, `sigma_q_mean`, ...)
   into `_last_diagnostics` (`e_step.py:829-862`). Even with `log_step`
   accepting the keys, the bare-name lookups silently returned `None`.
4. **Diagnostics turned off in the active config** —
   `train_vfe.py:130` had `track_layer_diagnostics: False`, suppressing
   population of `_last_diagnostics` upstream.

### Changes

**`transformer/training/metrics_tracking.py`**

- Added a keyword-only `disabled_columns: Optional[Iterable[str]] = None`
  argument to `PublicationMetricsTracker.__init__`. When provided, those
  names are removed from `self.headers` at construction time and stored in
  `self._disabled_columns`.
- `save()` now excludes disabled columns from the dynamic extra-keys
  discovery loop so they cannot re-enter via the rescue path used by
  gauge_* / fiber_* / holonomy_*.
- `csv.DictWriter` opened with `extrasaction='ignore'` so any leftover keys
  in history entries (which `log_step` always populates with `None`) are
  silently dropped rather than raising.

All three additions are no-op for existing callers (`experiment_runner.py`
at lines 317 and 2705, the three test instantiations in
`tests/transformer/test_experiment_runner.py`) — they don't pass
`disabled_columns` so `self._disabled_columns` stays empty and the save
path behaves identically.

**`transformer/vfe/trainer.py`**

- New class attribute `VFETrainer._CSV_DISABLED_COLUMNS` — the 40 columns
  that depend on PublicationTrainer-only instrumentation
  (`_compute_phi_diagnostics`, `_VFE_GRAD_DEBUG`,
  `numerical_monitor._flush_numerical_events`) which `transformer/vfe/`
  does not import. Comprises phi-spectral diagnostics (8),
  vfe_grad_*/vfe_kl_* decomposition (13), numerical fallback counters (4),
  sigma_q_cond / sigma_p_{min,max} (4), phi_pairwise_dist (2),
  attn_entropy_per_head + head_correlation (5), alpha_mahal_sq + kl_std +
  kappa_std (4).
- `_init_metrics` now passes `disabled_columns=self._CSV_DISABLED_COLUMNS`
  to the tracker.
- `_log_to_csv` row dict now carries **both** bare and namespaced keys.
  Bare names continue to feed the tracker's bare-name lookups
  (`train_loss_total`, `beta_mean`, `kl_mean`, `attention_entropy`,
  `attention_concentration`); newly added namespaced keys feed the
  prefixed lookups (`bayesian/alpha_mean`, `bayesian/c0`, `bayesian/b0`,
  `kappa/per_head_{mean,min,max}`, `cov/sigma_q_{mean,min,max,std}`,
  `cov/sigma_p_mean`, `cov/prior_belief_kl_{mean,max,std}`,
  `transport/phi_norm_{mean,std,max}`). Missing entries are emitted as
  `None` rather than `0` so empty cells render as blanks (matching
  publication-side semantics) instead of misleading zeros.
- `_log_vfe_dynamics_to_csv` and its call site at the top of the train
  loop removed entirely — the namespaced remap in `_log_to_csv` now
  populates every column it was trying to populate, through the standard
  `log_step` path. The two unique columns that method tracked
  (`e_step_nat_grad_mu`, `e_step_nat_grad_sigma`) were not in any
  publication CSV either and were dropped.
- Train loop at `trainer.py:923-930` now gates both `_log_to_csv` and
  `_log_to_pub_tracker` behind `is_log_step = (step + 1) % log_interval ==
  0`, matching the publication-side cadence at
  `experiment_runner.py:1321,1327`. Val rows continue to attach correctly
  because the active config has `eval_interval=2000` which is a multiple
  of `log_interval=200`.

**`transformer/vfe/train_vfe.py`**

- `track_layer_diagnostics: False` → `True` (line 130). This is the
  upstream toggle that populates `e_step._last_diagnostics` with the
  bare-name keys the namespaced remap consumes. Cost is ~30 `.item()`
  CUDA syncs on the final E-step iteration of each forward; on `n_e_steps=1`
  this is < 1% of step time at the user's measured ~150 ms/step.

### Verification

End-to-end smoke test (in-process tracker construction + log_step + log_val
+ save, with the production `_CSV_DISABLED_COLUMNS` set):

- 50 headers retained after pruning (down from 90+); pruned set includes

## Session 4 — final test-set evaluation in `transformer/vfe/`

User flagged that the end of a `train_vfe.py` run reports no test PPL,
unlike the publication path which calls `run_test_evaluation` at
`experiment_runner.py:2234-2241`. Two gaps:

1. **`create_dataloaders` was not asked for the test split.** The call at
   `train_vfe.py:165-170` unpacked a 3-tuple `(train, val, vocab)`;
   `include_test` (defaulting to `False` in `datasets.py:2198`) was never
   set, so the test loader was never constructed.
2. **`VFETrainer` had no final-evaluation method.** `train()` ended at
   final-checkpoint save + pub-tracker summary; no parallel to
   `run_test_evaluation` in `experiment_runner.py:119-239`.

### Changes

**`transformer/vfe/train_vfe.py`**

- `create_dataloaders(...)` call now passes `include_test=True` and
  unpacks a 4-tuple `(train_loader, val_loader, test_loader,
  vocab_size)`. Dataset log line extended with `test_batches=`.
- `VFETrainer(...)` invocation now passes `test_loader=test_loader`.

**`transformer/vfe/trainer.py`**

- `VFETrainer.__init__` accepts `test_loader: Optional[DataLoader] =
  None` (positioned after `val_loader`, before `device`); stored as
  `self.test_loader`.
- New `run_test_evaluation(max_samples=128000)` method (decorated
  `@torch.no_grad()`). Mirrors `experiment_runner.py:119-239`:
  token-weighted CE accumulation over up to `max_samples` test samples,
  `test_ppl = exp(min(ce, 20))`, BPC via `bpc_from_dataset` for the
  `tokens_per_char` correction, random baseline `= cfg.vocab_size`,
  improvement factor `random_ppl / test_ppl`. Uses
  `VFEModel.forward(input_ids, targets=target_ids)` directly (the same
  path as `evaluate()`), so reported test loss is directly comparable to
  the `val_loss` from periodic eval. Returns an empty dict when
  `self.test_loader is None`. Persists results to
  `output_dir/test_results.json` and emits a structured console block:

  ```
  ======================================================================
  FINAL TEST SET EVALUATION
  ======================================================================
    Cross-entropy loss: ...
    Perplexity:         ...
    Bits per character: ...
    Random baseline:    ...
    Improvement:        ...x better than random
  ```

- `train()` epilogue calls `self.run_test_evaluation()` once after the
  final checkpoint save, gated by `if self.test_loader is not None`.
  Placement is intentional — *after* final checkpoint (so test eval uses
  the saved model state) and *before* pub-tracker summary (so the
  console emit order matches the publication path).

### Verification

`python -c "...parse + signature..."` confirms both files parse, the
`test_loader` kwarg is in `VFETrainer.__init__`, and the
`run_test_evaluation` method is bound. Full test suite:

- `tests/transformer/test_experiment_runner.py`: 4 passed
- `tests/transformer/test_vfe_package.py`: 42 passed

Behavior is opt-in via `include_test=True` at the click-to-run site —
callers that previously instantiated `VFETrainer` without
`test_loader` still get `train()` finishing identically (the
`run_test_evaluation` step short-circuits on `None`).
