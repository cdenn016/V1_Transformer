# Post-Edit Log — 2026-05-18

## Issue investigated

User reported that sweeping `e_sigma_lr` and `e_phi_lr` in
`transformer/vfe/vfe_ablation_suite.py` produced no measurable effect on the
final loss. Three parallel investigation agents plus a verifier traced this
to **orphaned E-step retractions** under the suite's BASELINE_CONFIG:

- `n_e_steps = 1`
- `n_layers = 1`
- `use_prior_bank = False`
- `norm_type = 'layernorm'`  (σ-blind via `_LayerNormSigmaAdapter.forward`)

Under that combination the E-step inner loop in `transformer/vfe/e_step.py`
runs once with order `μ-update → σ-retract → φ-retract` (lines 665–696)
and then exits. The returned `sigma_new` and `phi_new` are autograd leaves
that no downstream operation reads: `model.py:183` decodes
`logits = output_proj(mu_final)` (σ, φ discarded), `block.py:49-50`
LayerNorm-adapter drops σ, and there is no next iteration or next block to
consume them.

Smoking-gun evidence:
`transformer/vfe/vfe_ablation_results/e_sigma_lr/sweep_results.csv` and
`.../e_phi_lr/sweep_results.csv` show `final_ppl = 325.1898297631453`
bitwise-identical across all 8 (σ) and 6 (φ) sweep rows.
The companion `e_mu_lr/sweep_results.csv` varies smoothly
(326.020 → 325.313) — μ-LR is on the autograd path; σ/φ-LRs are not.

## Lift conditions

Different for σ and φ:

| What reads it downstream                          | Lifts σ-orphan | Lifts φ-orphan |
|---------------------------------------------------|----------------|----------------|
| `n_e_steps >= 2` (next iter's `nat_grad_mu(μ,σ)`) | yes            | yes            |
| `n_layers >= 2` (next block's `compute_kl_attention(μ,σ,φ)`) | yes  | yes            |
| `use_prior_bank = True` (`decode(μ_q, σ_q, τ)`)   | yes            | no             |
| `norm_type in {'mahalnorm','centered_mahalnorm'}` | yes            | no             |

`prior_bank.decode` at `prior_bank.py:292-297` takes `(mu_q, sigma_q, tau)`
only — φ is not in the signature. No norm reads φ.

## Change

`transformer/vfe/config.py` — appended an orphan-detection warning at the
end of `VFEConfig.__post_init__` (after the `alpha_divergence` warning, before
the `irrep_dims` property). The warning:

- Fires when either σ- or φ-orphan condition is detected.
- Lists exactly which LRs are dead (`e_sigma_lr`, `e_phi_lr`, or both).
- States the full lift conditions for each so the user can pick a switch.
- Uses `UserWarning` with `stacklevel=2`, matching the existing
  `alpha_divergence` warning style; Python's default warning filter
  dedupes per source location so 60+ sweep rows fire it once.

Verified on three combinations:
1. baseline orphan → warns about `e_sigma_lr, e_phi_lr`
2. `use_prior_bank=True` → warns about `e_phi_lr` only (σ lifted)
3. `n_e_steps=2` → no warning (both lifted)

## What this change does NOT do

The warning makes the trap visible at config construction time. It does
**not** silently change the user's BASELINE_CONFIG (per CLAUDE.md
"LOCALLY DEFINED CONFIGS" and the `feedback_no_lenient_config_aliases`
memory). To actually make `e_sigma_lr` and `e_phi_lr` move the loss, the
user must change one of the four switches in their config — the warning
text enumerates the options.

Also untouched:
- The E-step inner loop order (`μ → σ → φ`) — reordering would change
  algorithmic semantics; out of scope for a "fix issues" pass.
- The whitelist filter at `vfe_ablation_suite.py:340-342`
  (`_strip_to_vfe_kwargs`) that silently drops unknown keys
  (e.g., `ffn_n_iterations`, `use_residual`, `evolve_sigma`,
  `evolve_phi_e_step`, `gauge_dim`, `em_mode`, ...). These legacy/driver
  keys in BASELINE_CONFIG never reach VFEConfig; whether to warn or
  hard-error on them is a separate question.

### Issue 2 — Rogue `transformer/vfe/outputs/figures/` from `train_vfe.py`

User reported a "dead no-op" `/output` folder appearing on every
`train_vfe.py` run. Traced to a default-argument bug in
`transformer/analysis/publication_metrics.py:198-199`:

```python
self.save_dir = Path(save_dir) if save_dir else Path("./outputs/figures")
self.save_dir.mkdir(parents=True, exist_ok=True)
```

`VFETrainer._init_metrics` at `transformer/vfe/trainer.py:644-654` was
constructing `TrainingTracker` **without** a `save_dir` argument, so the
constructor fell back to `./outputs/figures` relative to CWD. Running
`train_vfe.py` from `transformer/vfe/` therefore created
`transformer/vfe/outputs/figures/`. The directory was never written to:
the VFE trainer's actual figures land in `vfe_runs/{DATASET}_{timestamp}/figures`
via a separate `PublicationFigures(fig_dir)` instance (trainer.py:768-770),
and `VFETrainer` never calls `_pub_tracker.save_json`/`save_csv`.

Confirmed by directory listing: `transformer/vfe/outputs/figures/` existed
and was empty.

### Fix

Surgical patch at `transformer/vfe/trainer.py:647-650` — pass an explicit
`save_dir` rooted at the trainer's actual `output_dir`:

```python
self._pub_tracker = TrainingTracker(
    save_dir=(self.output_dir / 'figures') if self.output_dir else None,
    tokens_per_char=tokens_per_char_from_dataset(...),
)
```

When `output_dir` is `None` (no-disk smoke tests), `save_dir=None` still
hits the fallback and creates `./outputs/figures` — but those code paths
are explicitly opting out of disk persistence and the mkdir is the
existing pre-patch behavior, so this is no regression. The user-facing
bug (CWD pollution under normal `train_vfe.py` runs) is gone because
`self.output_dir` is always set in the `train_vfe.py` flow
(`output_dir = f'vfe_runs/{DATASET}_{timestamp}'` at line 184).

Removed the leftover empty `transformer/vfe/outputs/` to verify the new
run won't recreate it under the same cwd.

### Root-cause fix — lazy `mkdir` in `publication_metrics.py`

After the surgical patch above, user asked to "fix all". `PublicationFigures.__init__`
at `publication_metrics.py:370-372` had the identical default-arg bug —
unconditional `mkdir(./outputs/figures)` on construction — that doesn't
manifest in `VFETrainer` (always called with explicit `fig_dir`) but
could bite any other caller that constructs without an arg and never
plots.

Replaced the eager `mkdir` in both `TrainingTracker.__init__` and
`PublicationFigures.__init__` with a lazy property:

```python
def __init__(self, save_dir: Optional[Path] = None, ...):
    self._save_dir = Path(save_dir) if save_dir else Path("./outputs/figures")
    # ... no mkdir here

@property
def save_dir(self) -> Path:
    self._save_dir.mkdir(parents=True, exist_ok=True)
    return self._save_dir
```

The directory is now created on first read of `self.save_dir`. Read sites
are exclusively write contexts (`save_json`, `save_csv`, every `savefig`
in `PublicationFigures`), so the directory only materializes when
something actually writes. Constructed-but-unused trackers/figures
leave no trace.

Audit: only `__init__` sites assigned to `self.save_dir` — no external
mutation — so the property-only access pattern is safe. `tests/transformer/
test_publication_metrics.py` (8 tests) all pass post-patch.

Sanity-check confirming lazy behaviour:
- `TrainingTracker()` construction → no `./outputs/figures` created
- `PublicationFigures()` construction → no `./outputs/figures` created
- First `tracker.save_dir` read → directory materializes
