# 2026-05-01 Edits

## wiki-ja val/test PPL gap: representative eval coverage and optional source shuffle

### Problem

Run `29_K=40_GL(10)_N=128_wiki-ja_seed23` (240k steps, VFE_dynamic) reported
`val_ppl = 110.03` and `test_ppl = 29.68` on disjoint slices of the same
wiki-ja source. Train CE matched test CE (~3.39 nats) while val CE was stuck
at ~4.70 from step 200k onward — a flat 1.31-nat additive offset, not a
slowly-opening generalization gap. wikitext-103 runs do not show this
behavior.

### Root cause

Two compounding factors specific to wiki-ja:

1. **Tail-sliced splits.** `_load_wiki_ja_split`
   (`transformer/data/datasets.py:466-542`) takes train from indices
   `[0, max_articles=100K)` and val/test from `[n_train, n_train+n_val)` and
   `[n_train+n_val, total)` of the raw HuggingFace `wikimedia/wikipedia`
   `20231101.ja` dump. The HF dump is not randomized, so train sees the front
   of the corpus and val/test see the tail — a ~1.39M-article gap. wikitext-103
   ships with HuggingFace's curated, representative train/val/test splits, so
   it does not have this asymmetry.
2. **Stride-1 eval truncation.** With `eval_stride=None` (the default in
   `TrainingConfig.eval_stride`, `transformer/training/config.py:168`),
   `WikiText2TiktokenDataset.__getitem__` falls through to the legacy path
   `start_idx = idx` — i.e., a stride-1 sliding window. Combined with
   `validate(max_samples=12800)` in `transformer/training/train_fast.py:322`,
   each validation pass scores only the first ~12,927 unique tokens of the
   val concatenation. For wiki-ja's tail-sliced val split this lands on the
   first few articles of the tail (likely stub/list/redirect articles in
   Japanese Wikipedia's tail), pinning val_ce to those articles' difficulty
   for the entire training run. Test, with `max_samples=128000`, walks ~128k
   tokens of the test slice — enough to revert to mean.

The exp(1.31) ≈ 3.7 gap matches the observed val_ppl/test_ppl = 110/29.68.

### Fixes

**Change A — `transformer/train_publication.py` (~line 926).** When
`args.dataset == 'wiki-ja'` and the user has not explicitly set
`eval_stride`, auto-set it to match `stride` (default 128). With
`eval_stride=128`, validate's existing 12800 samples cover ~1.6M unique
tokens of the val split (vs ~13k previously) — enough to average over many
articles and remove prefix bias.

**Change B — `transformer/training/train_fast.py:322` and
`transformer/training/experiment_runner.py:2286`.** Bump the
`validate(max_samples=...)` and `_validate_pure_vfe(max_samples=...)`
defaults from 12800 to 128000 so val and test eval geometry are
byte-identical. Validation cost scales 10× per call (~1 s -> ~10 s on
typical hardware); negligible vs total training time. All four call sites
use the no-argument form so they pick up the new default automatically.

**Change C — `transformer/data/datasets.py` (`_load_wiki_ja_split`,
~line 466).** Add an opt-in deterministic shuffle of the source article
order, gated by env var `WIKI_JA_SHUFFLE_ARTICLES=1` (seed override via
`WIKI_JA_SHUFFLE_SEED`, default 42). Off by default so existing
checkpoints' val/test numbers stay byte-reproducible. New runs that opt
in get representative val/test draws from the full corpus distribution
rather than tail slices.

The token cache key for wiki-ja is also extended to include the shuffle
flag and seed (`tiktoken_cl100k_shuffled_s<seed>`) so toggling the env
var on a machine with a stale unshuffled cache forces re-tokenization
of the shuffled split rather than silently reusing old tokens.

### Verification

- `eval_stride` plumbing traced from `train_publication.py` config dict
  through `experiment_runner.py:2039` (TrainingConfig) and
  `experiment_runner.py:1719,1736` (`create_dataloaders`) into
  `WikiText2TiktokenDataset.stride` for val and test loaders. With the
  new auto-set, stride=128 reaches `__getitem__` at line ~1140 and the
  non-overlapping start_idx path is taken.
- All four `validate()` callsites use the no-argument form
  (`train_fast.py:481`, `experiment_runner.py:1454, 1666, 2159`) so the
  new default applies everywhere.
- The shuffle flag is read once per `_load_wiki_ja_split` call (no
  per-step overhead). When off, behavior is identical to pre-edit.

### Refuted candidates (left in place after agent investigation)

- `normalize_ce_by_dim` asymmetry: both `validate()` and
  `run_test_evaluation()` correctly retrieve `metrics['loss/ce_raw']` so
  the divide-by-sqrt(K) in `train.py:385-387` does not affect the
  reported number on either path.
- `loss/ce` vs `loss/ce_raw` fallback divergence: both keys are always
  written by `train.py:688-689`; the `.get(...)` fallback is defensive
  and never triggers in practice.
- Document-level data leakage: train, val, and test indices are disjoint
  by construction in `_load_wiki_ja_split`.

### Constraints respected

- No CLI flags added (CLAUDE.md hard constraint). Change A reads from
  the existing config dict; Change C reads from env vars.
- Default behavior preserved for existing runs (Change C defaults off;
  Change A only sets `eval_stride` when not already specified).
- No mathematical paths altered; eval geometry is now standard
  non-overlapping windowing matching training stride.

## Scaling analysis pipeline (WikiText-103 K-sweep)

### Problem

The user finished a multi-seed embedding-dimension scaling sweep at
`transformer/checkpoints_publication/0_scaling_data/` (3 seeds × 11 K
values, 32 of 33 runs present). They need publication-quality figures
plus accompanying numerical analysis, with the pipeline parameterized so
the upcoming K=48 / variable-irrep-dim sweep reuses the same modules.

### Approach

Four cleanly-separated analysis modules under `transformer/analysis/`
plus one click-to-run entry script under `scripts/`. The sweep axis
(name, regex, optional config-callback) flows through every module, so
swapping K for `irrep_dim` is a one-script-copy change with zero module
edits.

- **`transformer/analysis/scaling_ingest.py`** — `discover_runs` walks
  the sweep root and parses axis value via two-stage lookup (regex on
  dir name, then optional config callback). `aggregate_sweep` returns
  `(terminal_df, trajectory_df)` with `tokens_seen` computed per run
  from each run's own batch_size and seq_len (necessary because the K
  sweep uses bs=32 for K≤80 and bs=16 for K≥90).
- **`transformer/analysis/scaling_stats.py`** — `fit_power_law`
  fits PPL = a·K^b + c via `scipy.optimize.curve_fit`, with seed-
  resampled bootstrap CIs (resampling preserves the per-axis design,
  doesn't conflate seed variability with axis effects). Falls back to
  log–log linear when the nonlinear fit fails or seed counts are too
  low for bootstrap. Returns a `PowerLawFit` dataclass with point
  estimates, 95% CIs, R², and dense ribbon arrays for plotting.
- **`transformer/analysis/scaling_plots.py`** — four publication figures
  via `set_pub_style` + `PUB_CYCLE`: `fig_scaling_main` (1×2 log–log /
  linear-y with bootstrap ribbon), `fig_compute_frontier` (PPL vs FLOPs
  and PPL vs params, K-annotated), `fig_trajectory_by_K` (training and
  val PPL vs `tokens_seen` with smoothing window), and `fig_gauge_vs_K`
  (2×2 terminal-step gauge / attention diagnostics). Each saves as
  matched PNG + PDF at 300 DPI; missing-data panels render a muted
  placeholder rather than crashing.
- **`transformer/analysis/scaling_report.py`** — `write_aggregated_csv`
  produces the per-axis summary CSV (mean / std / sem / n_seeds /
  params / FLOPs); `write_methods_paragraph` renders a ≤250-word prose
  paragraph with rendered point estimates and CIs, then runs the
  banned-word linter (rejects the CLAUDE.md Claude-isms plus horizontal
  rules) before writing — fail-fast so manuscript text can't pick up
  Claude-isms silently.
- **`scripts/scaling_analysis_K.py`** — click-to-run, no CLI args. Top-
  of-file `CONFIG` dict pins sweep root, axis name / regex, output dir,
  bootstrap iterations, fit method. For the K=48 follow-up, copy the
  script and edit only the dict (sweep_root, sweep_axis_name,
  sweep_axis_from_config callback for the `irrep_spec`-based axis,
  axis_label).

### Reused without modification

- `transformer/visualization/pub_style.py::set_pub_style`, `PUB_CYCLE`,
  `PUB_COLORS` for matplotlib defaults and the Okabe-Ito palette.
- `transformer/visualization/training_plots.py::load_metrics_csv`
  (lines 36-69) for the `metrics.csv` loader.
- Error-bar style mirrors `holonomy_plots.py:271`
  (`fmt='o', capsize=3`).
- `transformer/analysis/publication_metrics.py::plot_scaling_study`
  left untouched — its skeleton is too narrow (no seeds / fits / FLOPs)
  to retrofit, and leaving it intact avoids regression risk to any
  caller of its current signature.

## BPC computation: tokens-per-character correction

### Bug

Every BPC ("bits per character") computation in the codebase ran
`bits_per_token = CE_nats / ln 2` and reported the result as BPC
without applying the BPE compression ratio. Empirical
tokens-per-character measurements on the actual tokenizers:

- **GPT-2 BPE on English**: ~0.23 tokens per character (4.4 chars/tok).
  Reported BPC was off by ~4.3x — what the code called BPC was
  actually bits-per-token.
- **cl100k_base on Japanese**: ~1.1 tokens per character (varies
  0.92–1.45 across hiragana / kanji / katakana / mixed). Reported BPC
  was off by ~10% — wiki-ja's BPC=5.32 should be ~5.85.

The right formula is
``BPC = (CE_nats / ln 2) * (n_tokens / n_chars_source)``, where
``n_chars_source`` is the character count of the raw text *before*
tokenization.

### Fixes

**Source of truth — `transformer/data/datasets.py`
(`WikiText2TiktokenDataset`)**: capture
``self._n_chars_source = len(full_text)`` during fresh tokenization;
persist it to a sidecar JSON
(``<token_cache>.meta.json``) alongside the token cache so cache
hits also have the value. Cache hits without sidecar fall back to
None and log a one-time message instructing the user to delete the
cache to rebuild with sidecar. Added a public ``tokens_per_char``
property that returns ``len(tokens) / n_chars_source`` or None.

**Helper — `transformer/training/bpc.py` (NEW)**: single source of
the conversion. ``compute_bpc(ce_nats, tokens_per_char,
fallback_key=...)`` applies the correction; falls back to bits-per-
token when ``tokens_per_char`` is None or non-positive, with a
one-time warning per fallback_key (so a single training run logs at
most one warning per call site, not per step). ``bpc_from_dataset``
wraps this for the common case of reading the ratio off a loader or
dataset.

**Wired through 9 BPC computation sites**:
- ``transformer/training/experiment_runner.py`` —
  ``run_test_evaluation`` (test BPC), validation log line, three pure-
  VFE training/val/test BPC sites.
- ``transformer/training/metrics_tracking.py`` —
  ``PublicationMetricsTracker`` now accepts ``tokens_per_char`` at
  construction, used in ``log_step`` and ``log_val``.
- ``transformer/analysis/publication_metrics.py`` —
  ``TrainingTracker`` and ``PublicationMetrics`` accept
  ``tokens_per_char`` at construction, used in ``record`` and
  ``record_validation``.

**Construction-site plumbing**: every ``PublicationMetricsTracker``,
``TrainingTracker``, and ``PublicationMetrics`` callsite now reads
``tokens_per_char`` off ``train_loader.dataset.tokens_per_char`` via
the helper ``tokens_per_char_from_dataset``. Sites updated:
``experiment_runner.py:317``, ``experiment_runner.py:2079``,
``experiment_runner.py:2673``, ``vfe/trainer.py:362``,
``vfe/trainer.py:375``, ``resume_training.py:544``.

### Tests

13 new tests in ``tests/transformer/test_bpc.py``: correction math,
None / zero / negative ``tokens_per_char`` fallbacks, dataset and
loader unwrapping, per-key warning suppression. All 53 new tests
across this session (40 scaling-analysis + 13 BPC) pass in 6.14 s.

### Verification

Smoke test confirms:
- ``compute_bpc(5.32 * ln 2, 1.1) = 5.852`` (wiki-ja correction)
- ``compute_bpc(5.32 * ln 2, 0.23) = 1.224`` (wikitext correction)

### Migration note for cached runs

Token caches built before this change have no sidecar. The dataset
will load them, set ``tokens_per_char = None``, and BPC will be
reported as bits-per-token with a one-time warning. To get correct
BPC, delete the affected token cache files (``<dataset>_<split>_*.pt``
under the HuggingFace cache dir) and re-run — the next load will
re-tokenize and write the sidecar.

### Constraints respected

- No mathematical paths altered. CE / PPL computations unchanged.
- No defaults flipped to silently change reported numbers; default
  behavior on legacy caches is "report bits-per-token + log warning"
  rather than "silently apply wrong correction".
- One-time warning per call site keeps logs quiet during training.
- All edits are surgical: every changed line traces to either the
  bug or the plumbing needed to deliver the fix.
