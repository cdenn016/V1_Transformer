# Edits — 2026-06-02

Deep multi-agent audit (workflow `wf_477d247d-238`) to deprecate/remove unnecessary
config code and add a compute-saving short-circuit. Investigators (config dead-code
sweep, phi-gradient routing, AMP/compile blast radius, runtime gradient-equivalence
test) plus an independent verifier; every change below is backed by static + runtime
evidence.

## 1. E-step φ short-circuit when `E_phi_lr == 0` (compute saving)

**File:** `transformer/core/variational_ffn.py`

Both φ-evolution blocks computed `grad_phi` every call even when the φ step size was
zero. With the production config `E_phi_lr = 0` (→ `self.phi_lr = 0`,
`evolve_phi_e_step = True`), the per-iteration block at the top of STEP 4b ran
`_compute_phi_grad` (a full `compute_attention_weights` pass + a localized
`torch.autograd.grad`) on *every* E-step iteration, then retracted with `step_size = 0`
— a pure value no-op. Added `and self.phi_lr != 0` to the guard of:

- the per-iteration block (STEP 4b), and
- the post-loop block (STEP 5, only reachable when `evolve_phi_e_step = False`).

**Why it is gradient-safe (not just a value no-op):** `_compute_phi_grad` builds
`grad_phi` on a *fresh detached leaf* (`phi_current.clone().requires_grad_(True)`) via
`torch.autograd.grad(..., create_graph=False)`, so `grad_phi` has no `grad_fn` and
carries no gradient to the learnable `phi_embed`. The M-step gradient that trains
`phi_embed` flows through a *separate* amortized straight-through chain
(`vfe_gradients.py` `create_graph=True` path, `em_phi_mode='amortized'` keeps the EM
boundary attached), which is independent of this inference block.

**Verification (`_verify_phi_gradient_equiv.py`, CPU):** production-faithful flags
(`em_mode='ift_phi'`, `E_phi_lr=0`, `evolve_phi_e_step=True`, `M_phi_lr=0.0025≠0`,
`gauge_param='phi'`, `diagonal_covariance=True`). Across `n_iter ∈ {1,3}` and
clamp-inactive/active stress paths: `phi_embed` grad nonzero at baseline (φ *is*
trained) and grad diff vs. the skip is `0.00e+00` after the edit (was `≤3.7e-9` before,
from the step-0 retraction's extra graph nodes — now removed). Verdict: SAFE_TO_SKIP.

The `_e_step_grad_norms['grad_phi']` diagnostic key is pre-initialized to `0.0` in
`__init__` and never deleted, so the bare-indexed readers in `experiment_runner.py`
(verbose diagnostics) cannot `KeyError` after the skip.

## 2. Remove AMP machinery (`use_amp`, `amp_dtype`) — unusable with Σ float32

AMP is inert on the production path: `PublicationTrainer` overrides both `train_step`
and `_run_forward_and_backward` and never enters an `autocast` context or touches the
`GradScaler` (the autocast/scaler lived only in the base `FastTrainer.train_step`,
which `PublicationTrainer` overrides away). It is additionally `device.type=='cuda'`-
gated, so the True path is unreachable on this CPU box, and all covariance ops require
float32 anyway.

Removed:
- `transformer/training/config.py` — `use_amp`, `amp_dtype` `TrainingConfig` fields.
- `transformer/training/experiment_runner.py` — `use_amp`/`amp_dtype` construction kwargs.
- `transformer/resume_training.py` — `use_amp`/`amp_dtype` construction kwargs.
- `transformer/train_publication.py`, `scripts/run_optuna_hpo.py`,
  `scripts/run_ablation_suite.py` — `use_amp` flat-CONFIG keys.
- `transformer/training/train_fast.py` — AMP setup (`self.use_amp`/`self.amp_dtype`/
  `self.scaler`), AMP init banner, the `autocast` wrapper in `train_step` (unwrapped),
  `scaler.scale(loss).backward()` → bare `loss.backward()`, `scaler.unscale_` removed,
  `scaler.step/update` → `optimizer.step()`, the `autocast` in `validate()`, and the
  `GradScaler` checkpoint save/load blocks.

**Kept (deliberately):** the defensive `torch.amp.autocast(enabled=False)` /
float32-upcast guard in `variational_ffn._forward_vfe` (and the ~40 sibling guards in
`attention.py`/`prior_bank.py`/`gauge_utils.py`/`embeddings.py`/...). These enforce the
"Σ must be float32" invariant against any *external* autocast and are protective, not
config plumbing. They are no-ops when no outer autocast exists.

## 3. Remove whole-model `torch.compile` (`use_compile`, `compile_mode`)

Never enabled; `device.type=='cuda'`-gated; the VFE E-step's data-dependent control
flow is hostile to graph capture. Removed the `TrainingConfig` fields, the construction
kwargs (`experiment_runner.py`, `resume_training.py`), the flat-CONFIG keys
(`train_publication.py`, `run_optuna_hpo.py`, `run_ablation_suite.py`), and the
`torch.compile` block in `FastTrainer.__init__`.

(The separate per-submodule `compile_vfe` toggle is also removed — see §6.)

## 4. Hardcode the always-constant `skip_attention` consumer

**File:** `transformer/core/blocks.py`. `self.skip_attention` is pinned `True`
(`blocks.py` `__init__`) and `BlockConfig.__post_init__` force-converts any `False`
request to `True` with a warning (a behavior an existing test depends on, so it stays).
Simplified the always-true diagnostic branch `if self.skip_attention and beta is None:`
→ `if beta is None:`. The field and the `__post_init__` forcing are unchanged.

## 5. Remove dead `use_layernorm` stored attribute

**File:** `transformer/core/blocks.py`. `self.use_layernorm = cfg.use_layernorm` was
never read again (the live backward-compat effect lives entirely in
`BlockConfig.__post_init__`, which is kept). Removed the dead assignment only; the
`BlockConfig.use_layernorm` field is untouched.

## 6. Remove `compile_vfe` (4th torch.compile toggle)

`compile_vfe` `torch.compile`d the `_vfe_iteration` inner loop. Never enabled; its own
comment states the E-step's data-dependent control flow (multihead vs single-β, fused
vs fallback, diagonal vs full cov) is incompatible with graph capture — the same reason
`use_compile` is unusable. Removed from `BlockConfig` (field + `from_config`),
`blocks.py` (FFN construction kwarg), and `VariationalFFNDynamic` (`__init__` parameter
+ the `self._compile_vfe`/`torch.compile` block). The sibling
`gradient_checkpoint_vfe` (activation checkpointing — a distinct memory feature) is
untouched.

## 7. Remove 8 mirror-only `TrainingConfig` fields

These fields had zero runtime readers — every live value comes from the flat CONFIG
dict via `BlockConfig.from_config` (the dataclass copies were read nowhere on the
training path). Their "preserved for display/logging" comment was stale (no such reader
exists). Removed the fields and their construction-site kwargs:

- `transformer/training/config.py` — `gauge_mode`, `gauge_param`, `omega_trust_region`,
  `isotropic_covariance`, `use_rope`, `rope_base`, `sigma_ce_scale`,
  `propagate_kl_nonfinite`.
- `transformer/resume_training.py` — `gauge_mode`/`gauge_param`/`use_rope`/`sigma_ce_scale` kwargs.
- `transformer/training/experiment_runner.py` — `sigma_ce_scale` kwarg.

Checkpoint-safe: configs are serialized as `vars(self.config)` dicts (`train_fast.py`),
so removing fields only drops the (unused) keys from new checkpoints; old checkpoints
still load (pickle restores extra attributes harmlessly).

**Kept (read off `self.config`, not mirrors):** `omega_lr`, `omega_det_penalty`
(`gauge_param='omega'` path — optimizer + `compute_free_energy_loss`).

**Test update:** removed `tests/transformer/test_training.py::TestTrainingConfig::
test_gauge_mode_default_learned`, which asserted the default of the removed
`TrainingConfig.gauge_mode` mirror. The `gauge_mode` default now lives on `BlockConfig`
(`= 'learned'`). The two override tests (`get_*_config(gauge_mode=...)`) still pass —
the builders apply overrides via generic `setattr`, so they no longer depend on the
field being declared (they now exercise the builder's pass-through, not gauge_mode
semantics).

## Kept on purpose (legitimate opt-in toggles, NOT dead code)

Per the "always a pure path under toggles" invariant, these were **not** removed even
though they are inert under the current production config — each gates a real opt-in
code path, and removing it would break that path:

- `fisher_ema_decay`, `fisher_damping` — the `optimizer_type='natural_gradient'` path.
- `pullback_series_order` — the `phi_optimizer_metric='pullback'` path.
- `omega_lr` — the `gauge_param='omega'` direct-Ω path.
- The defensive `torch.amp.autocast(enabled=False)` float32 guards in core modules —
  enforce the Σ-float32 invariant against any external autocast.
