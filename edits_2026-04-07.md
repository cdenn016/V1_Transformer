### Pipeline Walkthrough

**Stage 1: DataLoader → GPU**

`datasets.py:1485-1504`. The primary `create_dataloaders` path constructs a `WikiText2TiktokenDataset`. `__getitem__` returns `(input_ids, target_ids)` both of shape `(N,)` as `torch.long`. The DataLoader collates these into `(B, N)` and `(B, N)` batches. Padding uses `pad_token_id=0` for inputs and `-100` for targets (the `ignore_index` sentinel for cross_entropy). **DataLoader is configured with `pin_memory=True` (when CUDA available) and `persistent_workers=True` (when num_workers>0).** The `train_publication.py` EM_CONFIG sets `num_workers=10` in several configs — these are the configs with the full combination in scope. Prefetching: the DataLoader default prefetch factor (2) applies. The data transfer bottleneck is standard and reasonably well mitigated. No per-batch CPU↔GPU copies beyond the initial `.to(device)` call in `train_fast.py:205-206`.

Shape contract entering the model: `input_ids: (B, N), dtype=int64, on CPU → .to(device) → on GPU`.

**Stage 2: Embedding lookup** (`model.py:405-413`, `_embed_and_prepare`)

When `use_prior_bank=True`, the call is `self.prior_bank.encode(token_ids)` which delegates to `_get_prior_for_tokens(token_ids)` (`prior_bank.py:322`). This is fully vectorized: `self.phi_embed(token_ids)` is a standard `nn.Embedding` lookup returning `(B, N, phi_dim)`. When `gauge_fixed_priors=True`, the subsequent `fused_block_matrix_exp_pairs` call processes `(B, N, phi_dim)` in one fused batched operation. There is no per-token Python loop.

Output shapes at this stage:
- `mu_q: (B, N, K)` float32
- `sigma_q: (B, N, K)` float32 (diagonal mode) or `(B, N, K, K)` float32 (full)
- `phi: (B, N, phi_dim)` float32

**Stage 3: Cross-head permutation + prior clone + positional encoding** (`model.py:416-424`)

`mu_prior = mu_q.clone()` and `sigma_prior = sigma_q.clone()` are materialized here. These are separate tensors, not views. Shape unchanged: `(B, N, K)`. The positional encoding composes phi in the Lie algebra: `phi` becomes `(B, N, phi_dim)`, unchanged shape.

**Stage 4: Causal mask construction** (`model.py:427-429`)

`mask = self._causal_mask[:N, :N].unsqueeze(0).expand(B, -1, -1)` — shape `(B, N, N)`, float32 (0.0/1.0 values), on the model's buffer device. This is a view, not a copy.

**Stage 5: Transport operator precompute** (`model.py:432-455`)

When `evolve_phi=True` (as set in EM_CONFIG), `cached_head_transports = None`. Transport is computed fresh per block inside `blocks.py` using `fused_block_matrix_exp_pairs`.

**Stage 6: Per-block forward** (`model.py:985-999` or `1108-1119` in `forward_with_attention`)

Each block receives `(mu_q, sigma_q, phi)` and returns updated `(mu_q, sigma_q, phi)` with the same shapes. The KL attention matrix has shape `(B, n_heads, N, N)` internally, then softmax reduces to `beta: (B, n_heads, N, N)`. The E-step iterates mu in place over `ffn_n_iterations` steps.

**Stage 7: Output projection via PriorBank.decode** (`prior_bank.py:479-620`)

`decode(mu_q, sigma_q, tau)` receives `(B, N, K)` and `(B, N, K)` (or `(B, N, K, K)` if full-cov, which is immediately diagonalized at line 518). The fused matmul builds `lhs: (B, N, 2K)` and `rhs: (V, 2K)`, computes `torch.matmul(lhs, rhs.T) → (B, N, V)`. At B=16, N=128, V=50257, K=90: lhs is 16×128×180 = ~0.37M elements; the matmul output is 16×128×50257 = ~103M float32 elements = **~413 MB per call**. The user's config has B=16, so this is ~413 MB rather than the 1.65 GB figure from B=64 cited in the audit prompt.

Logits shape: `(B, N, V)` float32.

### Question-by-Question Findings

**Q1. Batch shape contract — clean.**

| Stage | Shape | dtype |
|---|---|---|
| input_ids | (B, N) | int64 |
| mu_q after embed | (B, N, K) | float32 |
| sigma_q after embed | (B, N, K) or (B, N, K, K) | float32 |
| phi after embed | (B, N, phi_dim) | float32 |
| mask | (B, N, N) | float32 |
| beta (per-head KL attn) | (B, n_heads, N, N) | float32 |
| logits | (B, N, V) | float32 |

No ambiguous shapes found. The sigma_q shape branches on `diagonal_covariance` and is consistent throughout.

**Q2. Masking and padding.**

The causal mask `(B, N, N)` propagates correctly from `_embed_and_prepare` through `block.forward → attention → compute_kl_attention` (`attention.py:383-386`). `masked_fill(mask == 0, float('-inf'))` is applied before softmax. With `mask_self_attention=False` (the EM_CONFIG default), no diagonal masking is applied. The consequence is that position 0 in any sequence attends only to itself (causal mask zeros out all future positions), and since `mask_self_attention=False`, its self-KL=0 is not suppressed. Softmax of `[-inf, -inf, ..., 0, -inf, ...]` equals 1.0 at the self-position, so `beta[b, h, 0, 0] = 1.0` for all heads. This is correct behavior with `mask_self_attention=False`; however the comment at `blocks.py:528` notes this means `mu_attn[0] ≈ mu[0]` (no new information from attention at position 0). Padding positions in input (token id=0, not -100) do participate in attention as if they were real tokens — the causal mask does not suppress them. This is a silent semantic issue: pad tokens at the end of a short sequence appear as valid context to all preceding positions.

**Q3. Token ID → prior lookup.**

`_get_prior_for_tokens(token_ids)` at `prior_bank.py:343-345`: `phi = self.phi_embed(token_ids)` — a single vectorized embedding lookup over `(B, N)` producing `(B, N, phi_dim)`. When `gauge_fixed_priors=True`, `fused_block_matrix_exp_pairs` is called once on the full `(B, N, phi_dim)` tensor. There is no per-token Python loop. De-duplication is not performed: if token id 287 appears 40 times in a batch, `matrix_exp` is computed 40 times for the same phi vector. At small K (block dim d_h=15 here), `matrix_exp` cost is O(d_h^3) = 3375 flops per token; for B=16, N=128: 2048 calls, many likely duplicate. De-duplicating by unique token id before the batched `matrix_exp` and then scatter-indexing back would save compute proportional to the vocabulary concentration of the batch. This is a missed optimization, not a correctness issue.

**Q4. Decode batching.**

`PriorBank.decode` is called once per forward in the standard path (via `_compute_logits`), but with `active_inference=True` it is called additionally from `_compute_active_inference_gradient` (pragmatic term: 1 call; epistemic term: 2S calls, S=2 by default → 4 more calls). With `aux_layer_loss=True` it is called once more per non-final layer. The fused matmul `(B, N, 2K) @ (2K, V)^T → (B, N, V)` is the dominant cost. At B=16, N=128, K=90, V=50257: each call allocates ~413 MB of output. With S=2, the active inference path alone generates 5 decode calls inside one E-step iteration: 1 pragmatic + 4 epistemic (pass 1: no_grad; pass 2: per-sample with_grad). Five calls × 413 MB = ~2 GB of peak transient allocation from decode outputs alone, though most are freed within the epistemic loop.

There is no batching optimization. Sequential decode calls are independent matmuls. A single fused call over the concatenated inputs would require materializing all (B×N×S, K) at once, which trades latency for memory. Given the memory budget concern the sequential approach is intentional and correct as documented.

**Q5. Active inference sample batching.**

`active_inference.py:228-271`: Welford two-pass design is verified correct. Pass 1 (`torch.no_grad`, lines 232-245) streams samples one at a time, accumulates a running mean of softmax probabilities, then discards each sample's logits tensor before the next sample. The `noise_list` stores `S` tensors each of shape `(B, N, K)` — at B=16, N=128, K=90, S=2: `2 × 16 × 128 × 90 × 4 bytes ≈ 1.5 MB`. This is negligible. Pass 2 (lines 251-271) builds and frees the autograd graph for one sample at a time. No `(S, B, N, V)` stack is ever materialized. There are no CPU↔GPU copies per sample: `torch.randn_like(template)` creates noise on the same device as `mu_f32`. The design is data-flow correct.

**Q6. Cache invalidation.**

`clear_decode_cache()` is called at the top of both `forward()` (line 921) and `forward_with_attention()` (line 1063) before any computation. The cache `_decode_cache` is a plain Python attribute on `PriorBank` (not a buffer), set to `None` by `clear_decode_cache()`.

One path where stale cache could theoretically arise: if `prior_bank.eval()` is called mid-forward (e.g., in a diagnostic callback), it switches `self.training` but does not clear `_decode_cache`. The eval mode path at `prior_bank.py:567-569` does populate the cache (`if self.cache_decode_priors and self.gauge_fixed_priors: self._decode_cache = (mu_p, sigma_p)`). The training-mode cache branch checks `self.cache_decode_priors and self._decode_cache is not None and self.gauge_fixed_priors` — it does NOT check `self.training`. So if `clear_decode_cache()` is not called between train and eval mode switches within a forward (which does not happen in normal operation — eval is only entered between `forward` calls in the validate loop), the cache is clean. The `validate()` loop in `train_fast.py:285` uses `torch.no_grad()` and calls the model's `forward()`, which fires `clear_decode_cache()` first. **No stale-cache path found in normal operation.**

Regarding reference counting: the cache stores `(mu_p, sigma_p)` as tensors. All consumers read from the same tensor object. None of the downstream consumers (`decode()`, active inference, aux layer loss) perform in-place operations on the cached tensors — they are only used in einsum/matmul inputs. The shared reference is safe.

**Q7. Data types.**

All belief state tensors (`mu_q`, `sigma_q`, `phi`) are float32 throughout. `stable_matrix_exp_pair` (`gauge_utils.py:77-84`) explicitly upcasts to float64 for block dimensions `d >= 8` (the default `dim_threshold`). With `irrep_dims = [15, 15, 15, 15, 15, 15]` (6 heads × d_h=15 for this config), every `matrix_exp` call promotes to float64 and casts back to float32 via `.to(orig_dtype)`. This is a **deliberate per-call double-precision promotion** — it is not a bug, but it does mean every transport operator computation in the hot path involves a float64 kernel. The `with torch.amp.autocast('cuda', enabled=False)` guard prevents AMP from overriding this.

The `decode()` method in `PriorBank` has an AMP guard at line 575 that casts to float32 if inputs are in a lower precision. Mixed-precision (fp16/bf16) is not enabled in the EM_CONFIG (`use_amp` defaults to False via the trainer). The system is uniformly float32 in the forward path, with targeted float64 promotion only inside `matrix_exp`.

**Q8. Device residency — clean.**

No `.cpu()` or `.to('cpu')` calls appear in the hot forward path. `_worker_init_fn` runs in DataLoader workers (CPU-side, correct). The `noise_list` in active inference is created with `torch.randn_like(template)` which inherits the device from `mu_f32` (which is `mu_current.detach().to(torch.float32)` — same device as the model). Generators are registered as buffers and move with the model. `_causal_mask` is a non-persistent buffer, also on the model's device.

**Q9. Dataset I/O — clean with one concern.**

`pin_memory=True` and `persistent_workers=True` (when `num_workers>0`) are set correctly. The EM_CONFIG sets `num_workers=10`, which is high for a Windows environment (processes, not threads) and may introduce overhead from process startup. The `WikiText2TiktokenDataset.__getitem__` is a pure tensor slice with no disk I/O per item (tokens are fully loaded into RAM at init). No I/O bottleneck during training; the concern is whether 10 Windows worker processes add more OS scheduling overhead than they save — this is environment-specific, not a correctness issue.

**Q10. Gradient accumulation and cache invalidation.**

EM_CONFIG has `'grad_accumulation_steps': 4` commented out (line 158), so `grad_accumulation_steps=1` is in effect (the `TrainingConfig` default). With accumulation disabled, `clear_decode_cache()` fires once per `forward()` call, which is once per micro-batch — correct.

If the user uncomments `grad_accumulation_steps: 4`: in `experiment_runner.py`, `forward_with_attention` is called once per micro-batch (line 984). Each call to `forward_with_attention` fires `clear_decode_cache()` at line 1063. So the cache is invalidated correctly at the start of each micro-batch's forward. There is no cross-micro-batch cache leakage.

**Q11. Active inference + decode cache interaction.**

When `cache_decode_priors=True` and `active_inference=True`, the first `decode()` call in `_compute_active_inference_gradient` (pragmatic term) computes `_get_prior_for_tokens(all_token_ids, only_forward=True)` and stores the result in `_decode_cache`. All subsequent `decode()` calls within the same forward pass (epistemic pass 1 no_grad calls, epistemic pass 2 grad calls, the final logit decode) hit the cache at `prior_bank.py:555`: `mu_p, sigma_p = self._decode_cache`. The cached tensors retain their autograd graph (they were computed with `only_forward=True` but not detached), so gradients from any consumer flow back through the same `matrix_exp` graph. No consumer modifies the cached tensors in-place. This is correct and is the intended behavior documented at `prior_bank.py:529-547`.

One subtle point: the pragmatic term in `_compute_active_inference_gradient` calls `prior_bank.decode(mu_var, sigma_f32)` where `mu_var` is a fresh leaf requiring grad. The `decode()` function accesses `mu_p` and `sigma_p` from the cache and computes `lhs @ rhs.T`. Because `mu_p` and `sigma_p` are shared references, if `autograd` decides to accumulate gradient through `mu_p` from multiple decode calls within `enable_grad()`, it could try to accumulate into the same tensor twice. However, `torch.autograd.grad(pragmatic_term, mu_var, ...)` only asks for gradient with respect to `mu_var`, not `mu_p` — so no gradient flows to the cache. The active inference gradients are detached before being applied (`grad_accum = grad_prag.detach()`). **No double-accumulation risk through the cache.**

### Three Biggest Data-Flow Concerns

**Concern 1: Pad-token positions are not masked in the KL attention matrix.**

Input padding (token id=0 for the tiktoken path) is inserted at sequence end when a sequence is shorter than `max_seq_len`. The causal mask suppresses these positions from being attended *to* by earlier positions (correct), but padding positions themselves form `mu_q` beliefs that participate in the E-step as if they were real tokens. Their `mu_q` is initialized from the prior of token id 0 (a real GPT-2 token, `!`), not from a special PAD prior. After the E-step they generate valid-looking logits and contribute to the loss only if the corresponding target is not -100 (the dataset correctly sets target padding to -100). The E-step VFE iterations for padding positions run wastefully and slightly corrupt the belief aggregation for the last few real tokens via the causal mask's last-real-position row, which does attend to padding-position beliefs during the E-step's internal KL computation. This is not catastrophic but is a subtle data-flow contamination: padding positions receive nonzero attention from each other (the causal mask permits `beta[pad_i, pad_j]` when `pad_i >= pad_j`), and the E-step updates them unnecessarily. The loss is correctly gated by the -100 ignore_index, but the belief evolution is not gated.

**Concern 2: Per-token `matrix_exp` for duplicate tokens — no de-duplication.**

In `_get_prior_for_tokens` (`prior_bank.py:343-428`) and the analogous path in `GaugeTokenEmbedding.forward` (`embeddings.py:427-505`), `phi_embed(token_ids)` returns one phi vector per position in `(B, N)`. When `gauge_fixed_priors=True`, `fused_block_matrix_exp_pairs` is called on the full `(B, N, phi_dim)` tensor, computing one `matrix_exp` per `(b, n)` position. Common tokens (e.g., "the", space, common punctuation) will appear many times per batch with identical `phi` vectors, yet `matrix_exp` is computed for each occurrence. At B=16, N=128: up to 2048 `matrix_exp` calls in the fused batched kernel. Since `phi_embed` maps token_id → phi deterministically, positions sharing the same token_id share the same phi and thus the same gauge transform. Gathering unique token ids, computing `matrix_exp` once per unique id, then indexing back would reduce compute in proportion to vocabulary concentration. For WikiText-103 with a Zipfian token distribution, the top 50 token ids account for roughly 40-50% of positions. This is a concrete optimization opportunity with no correctness impact.

**Concern 3: Grad accumulation + decode cache: the cache is NOT cleared between micro-batches in `train_fast.py`.**

`train_fast.py:228-252` implements gradient accumulation. The forward pass is `compute_free_energy_loss(self.model, input_ids, ...)` which calls `model.forward_with_attention()` — and `forward_with_attention()` does call `clear_decode_cache()` at line 1063. So the cache **is** cleared correctly per micro-batch in the `forward_with_attention` path. However, `train_fast.py` routes through `compute_free_energy_loss` (`train_fast.py:210`), which is a separate function. If that function calls `model.forward()` rather than `model.forward_with_attention()`, and `model.forward()` also calls `clear_decode_cache()` (line 921), then both paths are safe. Confirmed: both entry points clear the cache. **No actual bug here**, but the guarantee is implicit in two separate call sites rather than being enforced by a single authoritative location. If a future caller uses `model.forward()` without `forward_with_attention()` under accumulation, the cache is still correctly cleared per call. The risk is low but the dual-site pattern is worth noting as a maintenance fragility.

## Structural Refactor Audit — amortized_inference=True Hot Path

### Finding 1 (Easy Win): `forward()` and `forward_with_attention()` are partially duplicated — but less than feared

**Location:** `model.py:889-1026` (`forward`) and `model.py:1028-1406` (`forward_with_attention`).

Both methods share an identical prolog (prior bank cache clear, `_embed_and_prepare`, `_compute_logits`, `_apply_cross_head_perm`) that was already correctly factored into helpers. The divergence is real and meaningful: `forward` delegates the entire block stack to `self.transformer()` in one call (lines 985-999), while `forward_with_attention` manually iterates over `self.transformer.blocks` (lines 1088-1228) in order to collect per-layer `beta`/`kl`, apply `hierarchical_priors`, compute `aux_layer_loss`, collect `_layer_diagnostics`, and run `ImplicitEMGradient`. There is also a guard at lines 923-933 in `forward()` that raises `RuntimeError` when `implicit_em=True` and training — which is a code smell: the guard exists because `forward()` cannot be used for training when that flag is set, making the two methods semantically exclusive under a significant configuration. The pure duplication is approximately 20-30 lines (cache clear, embed, logits, perm). The training-time `ImplicitEMGradient` logic at lines 1244-1254 in `forward_with_attention` is the sole thing that cannot be in a shared helper without restructuring the block loop.

**Concrete fix:** Extract a `_run_block_stack(mu_q, sigma_q, phi, ...)` that accepts a `capture_per_layer: bool` flag and returns the tuple `(mu_q, sigma_q, phi, all_betas, all_kls, aux_losses)`. `forward()` calls it with `capture_per_layer=False` (fast path, reconstructs the current `self.transformer()` shortcut); `forward_with_attention()` calls it with `capture_per_layer=True`. This eliminates the 20-30 line prolog duplication and makes the `ImplicitEMGradient` application a post-call step in `forward_with_attention` only. The per-layer diagnostics collection stays inside the `capture_per_layer=True` branch. This is a pure restructure: no logic change, just relocation.

### Finding 2 (Easy Win): `amortized_inference=False` path is effectively dead code

**Location:** `variational_ffn.py:1315`, `1381`, `2579`; `block_config.py:105`; `train_publication.py:417`.

`amortized_inference` has exactly two behavioral branches. Branch 1 (`True`): `mu_p_current = mu_prior` (gradient flows through prior). Branch 2 (`False`): `mu_p_current = mu_prior.detach()`, and at line 1381, `phi_current = phi.detach()`. The entire codebase sets it `True` everywhere except a single Hebbian training config at `train_publication.py:417` — the pure Hebbian path that explicitly disables backprop entirely. Line 2579 also has the notably awkward `getattr(self, 'amortized_inference', False) is False` which uses `is False` instead of `== False` and wraps it in `getattr` defensively, suggesting this code has drifted.

There are zero test files that exercise `amortized_inference=False` independently (the Hebbian config is tested end-to-end, not the detach branch in isolation). The flag interacts with `detach_phi` but `detach_phi` is also only set `True` in the Hebbian config. The non-amortized path is not dead in the absolute sense but it is a maintenance liability: the branch at line 2579 is the worst example — a double-negative conditional on a flag that is almost never False.

**Concrete fix:** If the Hebbian config is the only consumer of `amortized_inference=False`, promote `detach_phi=True` to directly control the `phi.detach()` call at line 1382 (it already reads `self.detach_phi`) and remove the `amortized_inference` check at line 1381. The remaining check at line 1315 becomes a one-line docstring clarifying that `mu_p_current = mu_prior` always (since `implicit_em` is guarded separately). The awkward `getattr(...) is False` at line 2579 simplifies to just reading `self.amortized_inference`. This is a one-pass cleanup, not a refactor.

### Finding 3 (High Value, Medium Invasiveness): `_vfe_iteration` self-coupling gradient is triplicated across kernel variants

**Location:** `vfe_gradients.py:120-152` (`_compute_vfe_gradients_block_diagonal`), `vfe_gradients.py:558-581` (`_compute_vfe_gradients_block_diagonal_diag`), and `vfe_gradients.py:887-900` (`_fused_attention_and_vfe_gradients_block_diag`).

All three kernels compute the diagonal self-coupling gradient with identical structure: `grad_mu_self = alpha * delta_mu / sigma_p_safe`, `grad_sigma_self = alpha * 0.5 * (1/sigma_p - 1/sigma_q)`, then the product-rule correction block for learnable alpha (`alpha_c0 is not None and isinstance(alpha, torch.Tensor)`), then the debug writes to `_VFE_GRAD_DEBUG`. The full-covariance variant at lines 120-152 legitimately differs (uses `einsum` and `sigma_p_inv`, requires an extra dimension on `alpha`), but the diagonal variant at lines 558-569 and the fused variant at lines 887-895 are verbatim copies. The product-rule correction block alone is ~8 lines duplicated between the two diagonal kernels.

This duplication is not purely accidental: the fused kernel was split from the non-fused one for performance (one Omega pass instead of two), and they share the self-coupling computation because it does not touch Omega. However, the math is genuinely identical in the two diagonal cases.

**Concrete fix:** Extract `_compute_diagonal_self_coupling(mu_q, mu_p, sigma_q, sigma_p, alpha, alpha_c0, eps)` returning `(grad_mu_self, grad_sigma_self)`. Both `_compute_vfe_gradients_block_diagonal_diag` and `_fused_attention_and_vfe_gradients_block_diag` call it as their first step before diverging into their respective Omega-handling loops. The debug write block for `grad_mu_self`/`grad_sigma_self` can remain in each caller since debug instrumentation legitimately differs between fast and slow paths.

### Finding 4 (Medium Value, Low Invasiveness): `PriorBank.decode` three-branch structure is clean but mislabeled

**Location:** `prior_bank.py:549-572`.

The three-way conditional is structurally sound: (1) cache hit, (2) legacy gradient-checkpoint path, (3) inline + cache populate. The comment block at lines 523-548 accurately explains each path. What makes this smell is that path (2) and path (3) share identical post-processing (`mu_p, sigma_p = _prior_out[0], _prior_out[1]`) despite the fact that path (2) goes through a checkpoint lambda while path (3) optionally populates the cache. The two else branches (lines 556-572) could be collapsed into a single branch with `use_checkpoint = (self.gauge_fixed_priors and self.training and not self.cache_decode_priors)` as a local bool, making the cache-populate logic a simple postfix step. As-is, reading the code requires mentally tracking which `elif`/`else` you are in before seeing the shared postfix.

**Concrete fix:** Collapse paths (2) and (3) into one branch with an internal `if use_checkpoint:` guard, then unconditionally do `mu_p, sigma_p = _prior_out[0], _prior_out[1]`, then conditionally populate the cache. This makes the three conceptually distinct concerns (get priors, populate cache, compute KL) visible as sequential steps rather than parallel branches with duplicated postfix.

### Finding 5 (Medium Value, High Invasiveness): `_vfe_iteration` is the cyclomatic complexity center of mass

**Location:** `variational_ffn.py:1463-2430` (approximately 970 lines).

The method has roughly 3 major structural branches at the top level: (a) `multihead_vfe=True` path (lines 1608-1940, ~330 lines) which itself contains a 3-way dispatch (`rope_full_gauge` / `fused_mh` / fallback), (b) a single-beta path for `multihead_vfe=False` (lines ~1940-1970), and (c) shared post-gradient steps (natural gradient, clipping, sigma retraction, isotropic enforcement, diagnostics, phi evolution). Under `amortized_inference=True` with the default production config (`multihead_vfe=True`, `diagonal_covariance=True`, `rope_full_gauge=False`), the live path is the fused diagonal path at line 1714. The `rope_full_gauge` branch at line 1695 and the fallback full-covariance branch at line 1734 are dormant. Yet all three branches occupy ~200 lines inside the per-head loop, and they must be read through to find the active path.

The highest-value single decomposition: extract the per-head dispatch logic (lines 1608-1940, the `for h, d_h in enumerate(self.irrep_dims)` loop) into `_compute_multihead_gradients(...)` returning `(grad_mu, grad_sigma, beta_heads, _mh_cached_bep)`. This would reduce `_vfe_iteration` from ~970 lines to ~640 lines, make the per-head dispatch independently testable, and allow the three inner dispatch variants (rope_full, fused, fallback) to be seen without the surrounding iteration machinery. This is a pure extraction with no logic change.

### Finding 6 (Low Value, Zero Invasiveness): Config branching combinations — live vs tested

The theoretical combination space of the flags listed in the audit prompt is approximately 2^11 = 2048. The following combinations have zero test coverage and at least one structural reason to be silently broken or degenerate:

- `closed_form_e_step=True` + `update_phi_per_iteration=True`: The closed-form path at line 2530 captures `phi_current` but the phi-update-per-iteration path inside the iteration loop is skipped (`_n_iters = 0`). Phi never evolves. No warning is emitted.
- `implicit_em=True` + `gradient_checkpoint_vfe=True`: `implicit_em` detaches `mu_current` at the start of the E-step (`variational_ffn.py:1362`), which means gradient checkpointing saves activations that have no gradient anyway. The combination is not wrong but is wasteful and may confuse profiling.
