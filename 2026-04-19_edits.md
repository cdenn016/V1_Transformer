# 2026-04-19 Edits ‚Äî Surgical stride + random_offset_per_epoch rollout

## Context

Re-introduces `stride`, `random_offset_per_epoch`, `eval_stride`, and
`stride_base_seed` to the dataset/dataloader stack after the 2026-04-18 rollout
was reverted. The prior rollout produced a train-PPL drift at step 100 (781.7
vs 710.3 baseline) even with the feature disabled at its defaults. Post-mortem
traced it to a `num_sequences` formula rewrite ‚Äî legacy `max(1, N - T)` to
strided `(N - T) // stride + 1` ‚Äî that changed the length of the permutation
produced by `torch.randperm` inside PyTorch's `RandomSampler`. Same model
weights, different batch order, diverging loss trajectory.

The non-negotiable invariant this time: when `stride=None` (default), every
line of code executed is byte-identical to the pre-feature legacy path. No
equivalent-math rewrites, no "identical up to" claims ‚Äî the literal original
expression on the default path. The user confirmed the `710.3 @ step=100`
baseline is reproducible on the current machine before we started.

All nine commits land with the full test suite green (795 passed: 31 new
stride tests + 764 pre-existing). No existing tests required modification.

## Scope of changes

### Config layer (`transformer/training/config.py`)

Added four optional fields to the `TrainingConfig` dataclass after
`max_seq_len`:

- `stride: Optional[int] = None` ‚Äî window stride. `None` preserves legacy
  formula. `int` produces windows at offset + k¬∑stride.
- `random_offset_per_epoch: bool = False` ‚Äî when `stride > 1`, draw a
  per-epoch offset in `[0, stride)` from an isolated `torch.Generator`
  seeded by `stride_base_seed + epoch`. Forced `False` automatically when
  `stride is None` or `stride == 1`.
- `eval_stride: Optional[int] = None` ‚Äî separate stride for val/test
  loaders. Default `None` preserves dense eval-time coverage.
- `stride_base_seed: int = 0` ‚Äî seed for the isolated offset generator.

All four threaded through `TrainingConfig` primarily for introspection and
forward-compatibility. The trainer does not branch on them; all actual
windowing logic lives in the dataset classes.

### Dataset layer (`transformer/data/datasets.py`)

Four dataset classes ‚Äî `WikiText2Dataset` (transformers/BPE),
`WikiText2TiktokenDataset` (primary tiktoken path), `WikiText2CharDataset`
(char-level), `WikiText2ByteDataset` (byte-level) ‚Äî each received:

1. **New `__init__` kwargs**: `stride`, `random_offset_per_epoch`,
   `base_epoch_seed`, all with legacy-preserving defaults.
2. **Storage**: `self.stride`, `self.random_offset_per_epoch` (auto-gated off
   when stride is None or ‚â§1), `self.base_epoch_seed`, `self._epoch_offset=0`.
3. **`num_sequences` branch**:
   ```python
   if self.stride is None:
       # LEGACY PATH ‚Äî DO NOT MODIFY.
       self.num_sequences = max(1, len(self.tokens) - self.max_seq_len)
   else:
       N, T, S = len(self.tokens), self.max_seq_len, int(self.stride)
       k_max = max(0, (N - T - S) // S)
       self.num_sequences = max(1, k_max + 1)
   ```
   The `(N - T - S) // S` reserves `(S - 1)` tokens at the tail so every
   offset in `[0, S)` fits a full unpadded window. Consequence: `stride=1`
   gives `N - T` sequences (matches legacy count); `stride=T` gives
   `(N - 2T)//T + 1` full non-overlapping windows with room to shift.
4. **`__getitem__` branch**:
   ```python
   if self.stride is None:
       start_idx = idx  # LEGACY PATH
   else:
       start_idx = self._epoch_offset + idx * int(self.stride)
   ```
5. **`set_epoch(epoch)` method** ‚Äî identical body on each class. Early-returns
   when `random_offset_per_epoch` is False. Uses an isolated
   `torch.Generator()` (verified to not mutate the global RNG state).

### DataLoader factories (`transformer/data/datasets.py`)

Both `create_dataloaders` and `create_char_dataloaders` gained the same four
kwargs. Train datasets receive `stride=stride, random_offset_per_epoch=...,
base_epoch_seed=...`; val/test datasets receive `stride=eval_stride,
random_offset_per_epoch=False, base_epoch_seed=0`.

Added a hard safety assertion at the top of each factory: when any stride is
active, `num_workers > 0` raises `ValueError`. PyTorch pickles datasets to
workers once under `persistent_workers=True`, so per-epoch offset updates on
the main-process instance would not reach worker copies. This was the top
verifier-flagged bug. As defense in depth, `persistent_workers` is also
forced to `False` when stride is active.

### Call-site threading (`experiment_runner.py`, `resume_training.py`)

Threaded `config.get('stride', None)` and the three siblings through:

- `_create_dataloaders` (both char and BPE branches) at
  `transformer/training/experiment_runner.py:1707`.
- `TrainingConfig(...)` construction at
  `transformer/training/experiment_runner.py:1929` (primary FastTrainer build).
- `TrainingConfig(...)` construction at
  `transformer/resume_training.py:449` (resume path).
- `create_dataloaders` / `create_char_dataloaders` calls at
  `transformer/resume_training.py:352, 360` so resumed runs can opt into
  stride via the config dict.

The three preset helpers in `config.py` (`get_standard_config`,
`get_vfe_dynamic_config`, `get_pure_fep_config`) did not need edits ‚Äî they
use the dataclass default (all `None`/`False`/`0`).

### Training-loop `set_epoch` hooks

Module-level `_maybe_set_epoch(dataset, epoch)` helper in
`experiment_runner.py` (which `getattr`-guards dataset.set_epoch for
datasets that don't implement the method). Hooked at five sites:

1. `FastTrainer.train` main loop ‚Äî init at 1208, StopIteration at 1281.
2. `experiment_runner.py` pure_fep loop ‚Äî init at 2687, StopIteration at 2721
   (accumulated micro-batch path) AND 2748 (single-step path).
3. `transformer/training/train_fast.py` ‚Äî init at 374, StopIteration at 393.
4. `transformer/vfe/trainer.py` ‚Äî init at 569, StopIteration at 584.

`train_fast.py` and `vfe/trainer.py` use inline `getattr/callable` guards
rather than importing the helper (avoids cross-package coupling for a
two-line check).

At each epoch boundary we increment `_epoch_idx` and call
`_maybe_set_epoch`. Under `stride=None` the method is a no-op; under
`stride>1` and `random_offset_per_epoch=True`, it updates the window offset
via the isolated generator.

### Tests (`tests/transformer/test_dataset_stride.py`, new, 31 tests)

- `TestNumSequencesTable` (6): locks the `num_sequences` formula at known
  values across `(N, T, stride)` triples including degenerate `N==T`.
- `TestSetEpochContract` (16, parametrized over 4 dataset classes): proves

- `TestStrideNoneIdentity` (2): synthetic-dataset end-to-end check that

- `TestStrideTReproducibility` (2): same base_seed reproduces the full offset

- `TestWorstCaseOffsetBounds` (1): last idx √ó last offset stays inside the
  token tensor (guarantees no silent padding).
- `TestEpochZeroUnderOptionB` (1): different `base_epoch_seed` at epoch 0
  produces different offsets ‚Äî confirms Option B (no epoch-0 special case).
- `TestFactoryAssertion` (3): both factories raise `ValueError` on
  `stride + num_workers>0` before constructing any dataset.

## Design decisions locked in (from the planning pass)

- **Option B for epoch-0**: always draw the offset from the generator, even
  at epoch 0. `stride_base_seed` controls the first-epoch trajectory; cross-
  seed runs differ immediately. Rejected Option A ("force offset=0 at epoch
  0") because it would artificially correlate first-epoch data across seeds
  and render `stride_base_seed` inert at epoch 0.
- **Tail reservation `(S-1)`**: guarantees every offset in `[0, S)` maps to a
  full unpadded window. Wastes up to `S-1` tokens at the corpus end, but
  eliminates silent-padding data entering batches.
- **Factory assertion over silent fallback**: `stride + num_workers>0` hard-
  raises rather than coercing `num_workers=0`. Explicit failure is better
  than silent behavior change for a user who expected workers.
- **`stride=1` is NOT bit-identical to `stride=None`**: `stride=1` routes
  through the strided branch (different code path, same `__getitem__` output
  at offset 0 but different `if` evaluation). Use `stride=None` for bit-
  identical reproduction of pre-feature trajectories. Documented in the
  `TrainingConfig` field docstring.

## Second change set ‚Äî `phi_project_slk=True` restored under `use_prior_bank=True`

### Context

User report: `phi_project_slk=True` produced no visible effect when `use_prior_bank=True`. Three Explore agents and one independent verifier confirmed that `VFEPriorBank._apply_phi_det_control` runs as documented (`transformer/vfe/prior_bank.py:239-271`, called at encode `:287` and decode `:333`), but the positional encoder immediately undoes the projection. `VFEPositionalEncoding.pos_phi` was an unconstrained `nn.Parameter` of shape `(max_seq_len, n_gen)`. Since `tr([X,Y])=0`, every term in the BCH series past first order is traceless, so `tr(BCH(X,Y)) = tr(X)+tr(Y)` exactly. A traceless token phi composed with a non-traceless `pos_phi` yielded a composed phi with per-block trace equal to `tr(pos_phi) ‚âÝ 0`. The same defect existed in the legacy `GaugePositionalEncoding` (`transformer/core/embeddings.py`).

### Fix

Re-parameterize the positional Lie-algebra parameter onto the traceless subalgebra `sl(K_1) ‚äï ... ‚äï sl(K_H)` whenever `phi_project_slk=True`. Chosen over output-side projection because it eliminates gradient flow along the per-block trace directions structurally rather than zeroing it each forward.

New shared helper `build_slk_basis(generators, irrep_dims) -> (V_blocks, P)` in `transformer/core/vfe_utils.py`: returns per-block trace vectors and an orthonormal basis `P` of the traceless complement (via `eigh` of `I ‚àí Œ£_h VÃÇ_h VÃÇ_h^T`). Both positional encoders store `pos_phi_free` of shape `(max_seq_len, n_gen ‚àí H)` and expand as `pos_phi_free @ P.T` at forward time. `V_blocks @ P = 0` by construction, so the expansion lies in sl(K) for every representable value of `pos_phi_free`.

### Verification

- New test file: 9/9 passed.
- Regression sweep: `test_vfe_package.py` (31) + `test_embeddings.py` (9) + `test_prior_bank.py` (17) ‚Äî 57/57 passed.

### Backward compatibility

Old checkpoints written before this change store `pos_phi` (full `n_gen` width). When reloading with `phi_project_slk=True` the parameter name and shape both change to `pos_phi_free` (width `n_gen ‚àí H`), so state-dict loading of pre-existing checkpoints under the new flag will fail at the shape mismatch. Load with `phi_project_slk=False` (unchanged behavior) or retrain.

### Root-cause trace (for future readers)

The projection path at `prior_bank.py:287` and `:333` was never the bug. The non-traceless drift entered at `positional.py:50-52` / `embeddings.py:996` via an unconstrained `nn.Parameter`. Because BCH preserves additive traces exactly, projecting either side alone is insufficient ‚Äî both sides of the composition must be traceless.

## Third change set ‚Äî Real Yang-Mills curvature + skip-subplot when zero

### Context

End-of-publication-run emitted a matplotlib `UserWarning` from
`gauge_geometry_plots.py:158` (`Data has no positive values, and therefore
cannot be log-scaled`). Three Explore agents plus an independent verifier
traced it to `publication_metrics.py:1701` hardcoding
`energies=[0.0] * len(steps)` with comment "YM=0 for flat transport", which
was being fed unconditionally to a log-scaled axis. The comment was also
misleading: the framework's bilinear transport `Omega_ij = exp(phi_i) exp(-phi_j)`
is cocycle-flat by construction and yields YM = 0 exactly up to float noise
regardless of training state ‚Äî not "for flat transport" but always.

User requested: implement real YM curvature (zero-dimensional base manifold,
so holonomy-based, not continuum `F_{ŒºŒΩ}`) and skip the subplot when
energies are zero. Initial wiring used the existing `logm` path and ran 5+
minutes per diagnostic at publication scale (8000 complex eigendecompositions
of 128√ó128 matrices). Refactored to a Wilson/lattice action default which
drops cost to ~20 ms per call on realistic VFE_dynamic structure.

### Math

Base manifold is discrete (tokens are points), so curvature is loop
holonomy on the attention graph, not a continuous 2-form. For a triangle
`(i,j,k)` the holonomy is `C_{ijk} = Omega_{ij} ¬∑ Omega_{jk} ¬∑ Omega_{ki}`
and the Yang-Mills action admits two equivalent discrete forms:

- **Wilson** (default): `E_YM = (1/2) Œ£_t ||C_t ‚àí I||_F^2`. No
  eigendecomposition required; `O(T K^2)` per call. This is the standard
  lattice gauge-theory action.
- **logm** (opt-in): `E_YM = (1/2) Œ£_t ||log(C_t)||_F^2`. Requires complex
  `torch.linalg.eig` on every sampled triple; `O(T K^3)` and dominates
  runtime at publication K.

For bilinear (phi-path) transport the cocycle identity gives
`Omega_ij ¬∑ Omega_jk ¬∑ Omega_ki = omega_i ¬∑ omega_j^{-1} ¬∑ omega_j ¬∑ omega_k^{-1} ¬∑ omega_k ¬∑ omega_i^{-1} = I`
exactly. Reported YM energy is therefore zero up to float noise. A
non-trivial value requires a genuinely non-flat edge correction
(e.g. `connection.py` MLP mode with learned `delta_ij`).

### Design decisions

- **Wilson action as default**: on a zero-dimensional base manifold it is
  the standard lattice gauge-theory action, and avoids the `O(K^3)` complex
  eig that was causing the 5-minute diagnostic stall. `logm` remains
  available as an opt-in for the continuum correspondence.
- **`batch_limit=1` default**: matches the existing
  `compute_gauge_orbit_dimension` convention (first batch element only).
  Pass `batch_limit=None` to process the full batch.
- **Analytical inverse via `stable_matrix_exp_pair`**: solve-based
  inversion accumulates `O(K)` roundoff that surfaced above the physical
  noise floor at K=128, pushing the computed YM above the skip threshold
  on purely flat transport. Using `exp(-phi¬∑G)` directly drops the noise
  floor by ~4 orders of magnitude at publication scale.
- **`ym_tol = 1e-4` skip threshold in the plotter**: above the observed
  float-noise floor for bilinear transport (~2e-8 realistic,
  ~3e-4 pathological test), below any physically meaningful non-flat YM
  which scales like `phi_scale^2 ‚âà 2.5e-3`.

## Fourth change set ‚Äî Close the EM_CONFIG leak: legacy PriorBank + gauge_fixed_priors=False

### Context

User reported that after the second change set, `det(Œ©)` under their EM_CONFIG still matched the pre-fix distribution bit-identically (`mean=1.2366, median=1.0000, std=1.0300, min=0.0335, max=29.8822`). Three investigative agents plus an independent verifier traced the gap to two layers of `gauge_fixed_priors`-gated dead code that kept the projection silent in the user's configuration (`use_prior_bank=True`, `gauge_fixed_priors=False`, `phi_project_slk=True`, `irrep_spec=[('fund',2,10)]`, `em_mode='em_phi_p'`).

The second change set had fixed `VFEPositionalEncoding` and `GaugePositionalEncoding` to hold `pos_phi` on the traceless submanifold. What remained uncovered:

1. Legacy `transformer/core/prior_bank.py` had no `_apply_phi_det_control` at all ‚Äî `phi_project_slk` was not a kwarg, no `_phi_trace_vec` buffer was registered, and the token phi flowed raw into downstream transport.
2. `transformer/core/model.py:949` passed `generators` to the bank only when `gauge_fixed_priors=True`. With `False` the bank could not build its trace-vector buffer even if the projection code existed.
3. `transformer/core/model.py:306` set `_embed_irrep_dims=None` unless `gauge_fixed_priors=True`, so the per-head block structure never reached the bank. Empirically this reduced `max|s_h|` from 0.152 only to 0.088 ‚Äî the projection was collapsing two GL(10) blocks into one GL(20) and killing only the full-matrix trace, leaving per-head traces alive.

## Change set 5 ‚Äî BALD double-count, no-grad defense, F(t) monotone check

### Context

Audit of the VFE package and active-inference path surfaced four issues. Three parallel Explore investigators inspected the code, and a verifier reconciled a conflict in their findings using the primary-source comment at `transformer/vfe/efe.py:103-110`. The user chose canonical BALD MI semantics (both Pass-1 and Pass-2 of `active_inference.py`) and warning-only enforcement for the new F(t) monotone invariant.

### C1 ‚Äî BALD epistemic œÉ double-count (`transformer/core/active_inference.py`)

The Welford-style 2-pass epistemic gradient passed `sigma_f32` (Pass-1, line 267) and `sigma_var_s` (Pass-2, line 293) into `_readout` while ALSO sampling `z_s = Œº + ‚àöœÉ ¬∑ Œµ`. With `PriorBank.decode`, the œÉ argument feeds the KL-trace term `tr(Œ£_q / Œ£_p)` ‚Äî double-counting œÉ against the reparameterization path. The reference pattern at `efe.py:103-110` carries an explicit comment stating œÉ must be 0 in decode for canonical BALD MI.

Fix applied at both call sites: pass `torch.zeros_like(sigma)` to `_readout` so pÃÑ (Pass-1) and p_s (Pass-2) share the same point-conditional decoder. The author's design comment at lines 212-218 (which documented the œÉ-in-both-paths behavior as intentional EFE variant) was rewritten to reflect the new canonical BALD semantics and the full-covariance consequence: path-2 reparameterization does not differentiate through Cholesky, so with œÉ=0 in the readout, the full-cov epistemic term contributes zero gradient to Œ£. This matches `efe.py` (diagonal-only) and is an accepted limitation, not silent bias.

The pragmatic readout at line 197 (`logits = _readout(mu_var, sigma_var)`) is untouched: that term is H[p(v|Œº,œÉ)] ‚Äî entropy of the belief-decoded predictive ‚Äî not a BALD MI estimator, so œÉ in decode is semantically required.

### C2 ‚Äî `_update_phi` missing `torch.enable_grad()` (`transformer/vfe/e_step.py`)

`_update_phi` cloned `phi` with `requires_grad_(True)` and ran `torch.autograd.grad(alignment_loss, phi_for_grad)` without a defensive `torch.enable_grad()` wrapper. Under an outer `torch.no_grad()` (eval contexts), `requires_grad_(True)` is silently ignored and `alignment_loss.grad_fn` is None; the existing `if alignment_loss.grad_fn is not None:` guard at line 447 then silently returned œÜ unchanged rather than raising. Fix mirrors the defensive pattern in `_compute_rope_full_gauge_gradient_per_head` at `vfe_gradients.py:2106`: wrap the forward + autograd.grad block in `with torch.enable_grad():`.

### C4 ‚Äî F(t) monotone descent invariant (`transformer/vfe/e_step.py`)

E-step diagnostics logged gradient norms, Œ≤ stats, KL, œÜ norms ‚Äî but never the scalar free energy F(t). Bogacz 2017 requires F(t+1) ‚â§ F(t) for correct natural-gradient descent; without this signal, silent divergence corrupts training without surface symptoms. Added:

- Per-iteration F(t) computation (diagonal covariance only ‚Äî full-cov KL(q‚Äñp) needs `log|det|` and is skipped). F = Œ±¬∑Œ£_i KL(q_i‚Äñp_i) + Œª_align¬∑Œ£_ij Œ≤_ij¬∑KL_ij.
- Combined relative + absolute tolerance (`5%` of |F| or `1e-2`, whichever larger) to avoid flagging finite-step overshoot. A `RuntimeWarning` fires only on violations exceeding BOTH tolerances.
- `f_history`, `f_final`, and `f_monotone` fields added to `self._last_diagnostics` for downstream hard-gate unit tests.

Running the test suite after the fix surfaced real non-monotone excursions in `test_vfe_package.py::TestVFEModel` (F jumping ~5% per iteration). These are pre-existing, newly visible; no test failures induced.

### C3 ‚Äî FD coverage for `rope_full_gauge` path (`tests/transformer/test_rope_full_gauge_fd.py`)

Prior coverage was zero: `scripts/verify_vfe_gradients_fd.py` only exercised `_compute_vfe_gradients_block_diagonal_diag`, and integration tests in `test_per_head_gauge_connection.py` only checked forward/backward completion. Added `tests/transformer/test_rope_full_gauge_fd.py` with two tests:

1. `test_rope_full_gauge_grad_matches_f64_autograd_oracle` ‚Äî reconstructs the per-head F scalar (RoPE rotation on Œº and Œ£, Œ©-transport via matrix_exp pair, full-cov KL with slogdet, softmax Œ± and alignment terms) in float64, runs autograd to obtain the oracle gradient, then compares against the float32 analytic gradient from `_compute_rope_full_gauge_gradient_per_head`. PASS criterion: max relative error < 1e-2.
2. `test_rope_full_gauge_raises_under_no_grad` ‚Äî regression fixture that invokes the function under an outer `torch.no_grad()` and asserts gradients are finite and non-zero, keeping the defensive `enable_grad()` at `vfe_gradients.py:2106` from silently regressing.

Both pass. Full suite: 817 passed, 4 skipped.

### NF1 / NF2 ‚Äî not code-changed

NF1 (target-leakage under `no_grad` around `model.forward(targets)`) is theoretical: no current entry point wraps forward+loss in `torch.no_grad()` where phi would be affected. With C2 fixed, the failure mode is also now loud rather than silent. NF2 (C1 magnitude on training path) is bounded: `train_publication.py:171` sets `active_inference=False` by default, so C1 only affected training when opt-in enabled.

## Change set 9 ‚Äî Full-covariance NaN-at-step-104 instability fixes

### Problem

Training `scripts/run_ablation_suite.py` with `diagonal_covariance=False`, `gauge_param='phi'`, `em_mode='em_phi_p'`, and a user-override `M_sigma_p_lr=1e-6` NaNs at steps 104‚Äì115 out of 5000. The diagonal path is stable even at `M_sigma_p_lr=0.025`. A *smaller* prior LR making the full-cov path *less* stable is diagnostic ‚Äî it rules out every standard "small update = stable" explanation and implicates a frame-aware failure mode.

Three parallel investigation agents plus three verification agents converged on five structural root causes. Two earlier hypotheses (the `omega_det_penalty` and the V8 `_compute_omega_grad_direct` activation from commit ad545671) were refuted ‚Äî both are gated on `gauge_param='omega'` and do not fire under the user's `gauge_param='phi'` config.

### Verified root causes

1. **Ridge-vs-clamp floor asymmetry** (`transformer/core/variational_ffn.py:1486-1508`). Full-covariance E-step floor used `sigma_p + _floor ¬∑ I_K` (additive ridge); diagonal uses `clamp(min=_floor)`. ## Change set 11 ‚Äî `MahalanobisNorm` full-cov stability under `evolve_sigma=True`

### Problem

After change sets 9 + 10 landed (all 753 tests passing), the user re-ran `scripts/run_ablation_suite.py` under `M_sigma_p_lr=0.0`, `diagonal_covariance=False`, `evolve_sigma=True`, 200 steps. NaN still appeared starting at step 107, with:

```
eigh_jitter_escalate: 180 | eigh_svd_fallback: 180
loss_nan_skip: 90 | nan_replace: 90 | spd_retract_eigenclamp: 5745276
```

The user's diagnostic narrowed the source definitively: setting `evolve_sigma=False` (freeze Œ£_q at init) fixes the run, and switching `norm_type='layernorm'` (Œ£-independent) also fixes it. Only `mahalnorm + full-cov + evolve_sigma=True` NaNs. The NaN originates in `MahalanobisNorm`, not in the KL / E-step paths already fixed.

### Mechanism

`transformer/core/blocks.py:138-156` ‚Äî `MahalanobisNorm.forward` full-covariance branch:

```python
ridge = make_ridge(K, eps=1e-5, ...)  # eps¬∑I
sigma_f32 = sigma.float() + ridge
sig_inv_mu = torch.linalg.solve(sigma_f32, x_f32.unsqueeze(-1))
s2 = (x_f32 * sig_inv_mu).sum(...)
scale = torch.sqrt(K / (s2 + eps))
```

Under `evolve_sigma=True`, `retract_spd_torch` clamps Œ£_q eigenvalues only to its numerical `eps=1e-6` ‚Äî allowed to drift as low as 1e-6. The `eps=1e-5` ridge shifts eigenvalues by 1e-5 ‚Äî still `cond(Œ£) ‚âà 10‚Å∂`. Float32 `torch.linalg.solve` on such a matrix loses precision; `Œº·µÄŒ£‚Åª¬πŒº` picks up negative noise from cancellation; `sqrt(K / negative + eps) = NaN`.

The diagonal branch is safe because line 137 does `sigma.clamp(min=eps)`, bounding `1/œÉ_k` per-dimension. Full-cov has no equivalent eigenvalue floor.

### Why this is orthogonal to change sets 9 + 10

Change sets 9 + 10 addressed three SPD-inversion sites: the E-step Œ£_p floor (`variational_ffn.py:1486-1508`), the transported-Œ£_t eigclamp (`_kl_kernel_dense`), and the NaN-grad guard (`RiemannianAdamW.step`). MahalanobisNorm is a **fourth** site ‚Äî Œº normalisation occurs outside the E-step + KL pipeline, so none of those fixes reached it. The fix reuses the same `spd_eigfloor` primitive but applied at the norm layer.

### Fix

`transformer/core/blocks.py::MahalanobisNorm`:

- **`__init__`** (line 107): new `sigma_floor: Optional[float] = None` kwarg. Default `None` preserves the legacy ridge path (bit-identity).
- **`forward` full-cov branch**: when `self.sigma_floor` is set, replace the `eps¬∑I` ridge with `spd_eigfloor(sigma, floor=self.sigma_floor, exp_phi=exp_phi)` before the solve. Eigenvalue floor bounds `cond(Œ£) ‚â§ ‚àû/floor`, so solve stays well-conditioned. Strict GL(K)-covariant when `exp_phi` is provided (via `spd_eigfloor`'s whitening path added in change set 10).
- **Belt-and-suspenders**: the post-solve scale computation now uses `s2.clamp(min=self.eps)` before sqrt, so even if residual float32 cancellation produces negative `s2`, the output stays finite.

`transformer/core/blocks.py::_make_norm`: accepts new `sigma_floor` kwarg; passes to `MahalanobisNorm`. Ignored for other norm types.

Call-site updates (lines 287, 419, 862): all three `_make_norm(...)` invocations now pass `sigma_floor=getattr(cfg, 'e_step_sigma_floor', None)`, reusing the existing `BlockConfig.e_step_sigma_floor` field (default 0.01). **No new config field.**
