# 2026-04-18 вЂ” Deep VFE Package Audit Fixes

## Context

Five specialist agents (code-reviewer, data-engineer, performance-engineer,
general-purpose-as-ml-engineer, debugger) ran parallel audits of
`transformer/vfe/`. Findings were consolidated and triaged into
CRITICAL / HIGH / MEDIUM / LOW by severity. This commit lands the surgical
fixes; structural perf refactors are deferred and noted at bottom.

All 715 pre-existing tests pass after these changes.

---

## CRITICAL (fixed)

### C1 вЂ” `norm_type='layernorm'` silent no-op

`train_vfe.py` selected `'layernorm'` but `block.py`/`model.py` only branched
on `'mahalnorm' | 'rmsnorm'`; the unknown string fell through to
`self.norm = None`, meaning training ran with no layer normalization
whatsoever and no warning.

- `transformer/vfe/block.py`: added explicit `'none'` branch plus
  `else: raise ValueError(...)` for unrecognized strings.
- `transformer/vfe/model.py`: same fix on `final_norm`.
- `transformer/vfe/train_vfe.py`: changed default to `'rmsnorm'`.

### C2 вЂ” CE `ignore_index=-1` but dataset pads with `-100`

`model.py:137` and `trainer.py:326` used `-1`. The dataset pads targets with
`-100`, so either CE crashed under strict PyTorch builds or silently
corrupted loss and val-PPL whenever tail padding fired.

- `transformer/vfe/model.py`: `ignore_index=-1` в†’ `-100`.
- `transformer/vfe/trainer.py`: `(target_ids != -1)` в†’ `(target_ids != -100)`.

### C3 вЂ” `_update_phi` runs attention with wrong transport kernel

The main E-step at `e_step.py:286` passes
`exact_diagonal_transport=self.exact_diagonal_transport`. The
`_update_phi` path at `e_step.py:416` omitted it, defaulting to `False`.
Under the common config (`diagonal_covariance=True`,
`exact_diagonal_transport=True`) this means the phi gradient descends a
different KL than the mu/sigma gradients вЂ” phi converges to a gauge frame
misaligned with the objective being minimized.

- `transformer/vfe/e_step.py`: threaded
  `exact_diagonal_transport=self.exact_diagonal_transport` into the
  `_update_phi` call to `compute_kl_attention`.

### C4 вЂ” `sigma_p` inverse recomputed every E-step iteration

`vfe_gradients.py` recomputes `_safe_spd_inv(sigma_p)` and `slogdet(sigma_p)`
on every inner iteration even though `sigma_p` is frozen at the embedding
value for the full-cov path. Deferred: hoisting requires adding an optional
precomputed argument to `compute_vfe_gradients_gpu` which has three call
sites across the vfe, legacy ffn, and deq paths. Listed as a structural
perf optimization to track separately.

## HIGH (fixed)

### H5 вЂ” `prior_handoff_phi` was a silent no-op

`stack.py:107` wrote `priors.phi` conditionally on the flag but
`e_step.py:198` initializes `phi = beliefs.phi` and never reads
`priors.phi`. The flag did nothing.

- `transformer/vfe/stack.py`: replaced the conditional write with
  `new_prior_phi = initial_priors.phi` unconditionally plus a comment
  explaining phi flows via `beliefs`, not `priors`.
- `transformer/vfe/config.py`: added `DeprecationWarning` in
  `__post_init__` when the flag is set; doc-comment updated.
- `transformer/vfe/train_vfe.py`: changed default from `True` to `False`.

### H6 вЂ” No seed in entry point

`train_vfe.py` never called `torch.manual_seed` / `numpy.random.seed`;
runs were non-reproducible.

- `transformer/vfe/train_vfe.py`: added `SEED = 1234` constant and explicit
  `random.seed` / `np.random.seed` / `torch.manual_seed` /
  `torch.cuda.manual_seed_all` calls at the top of `__main__`.

### H7 вЂ” `allclose` generator skew-check forces GPU sync every iteration

`attention.py:42` called `torch.allclose(generators, -generators.T, atol=1e-6)`
inside `compute_gauge_transport`, which fires on every E-step iteration.
`generators` is an immutable registered buffer.

- `transformer/vfe/attention.py`: added `_SKEW_CACHE` keyed by
  `id(generators)` and an optional `skew_symmetric: bool` kwarg to
  `compute_gauge_transport`. The `allclose` now fires once per buffer
  lifetime, not once per iteration.

### H10 вЂ” CSV "loss component" columns hard-coded to zero

`trainer.py` wrote `train_loss_belief_align`, `train_loss_self_consistency`,
`train_loss_model_coupling`, `train_loss_aux_layer_ce` all set to `0`.
Publication plots reading those columns showed flat-zero curves, which is
misleading.

- `transformer/vfe/trainer.py`: removed the placeholder zero columns. The
  VFE objective is minimized implicitly via the E-step inner loop, so the
  decomposed components aren't directly available without additional
  instrumentation.

### H11 вЂ” `initial_priors` aliases belief tensors, enabling in-place mutation

`model.py:120` set `initial_priors = beliefs`. The stack then clamps
`new_prior_sigma[..., diag_idx, diag_idx]` in place (`stack.py:102`). When
the frozen branch (`rho_sigma == 0`) reuses `initial_priors.sigma`
directly, in-place writes by downstream code would leak into the posterior
through the alias.

- `transformer/vfe/model.py`: `initial_priors` now constructed explicitly
  with cloned `sigma`/`phi`:
  ```python
  initial_priors = BeliefState(
      mu=beliefs.mu,
      sigma=beliefs.sigma.clone(),
      phi=beliefs.phi.clone(),
  )
  ```

### H19 вЂ” Full-cov sigma handoff clamps only diagonal, doesn't enforce SPD

`stack.py:97-104` built the blended prior sigma as
`(1-ПЃ)В·ОЈ_embed + ПЃВ·ОЈ_post` and clamped only the diagonal. If the
posterior's off-diagonals are large the result could be indefinite.

- `transformer/vfe/stack.py`: added `0.5 * (ОЈ + ОЈ.T)` symmetrization
  before the diagonal clamp.

## MEDIUM (fixed)

### M12 вЂ” `rope_full_gauge` silently promoted

`config.py:148` auto-switched `rope_full_gauge=False в†’ True` whenever
`diagonal_covariance=False` and `use_rope=True`. A checkpoint's config
would round-trip as `False` but the runtime would execute the `True`
branch.

- `transformer/vfe/config.py`: replaced the silent promotion with an
  explicit `ValueError` requiring the user to set the flag.
- `tests/transformer/test_vfe_package.py`: updated
  `test_full_cov_allowed` to pass `rope_full_gauge=True` explicitly.

### M13 вЂ” `rope_full_gauge` branch gated on inference mode, not grad mode

`e_step.py:234` checked `not torch.is_inference_mode_enabled()`. Under
`@torch.no_grad()` blocks in `generate()` and `evaluate()` the branch
still fires, even though no gradients are needed.

- `transformer/vfe/e_step.py`: added `and torch.is_grad_enabled()` to
  the guard.

### M14 вЂ” `lambda_soft` entry-point override contradicts schema default

`config.py:37` default was `1.0`; `train_vfe.py:50` overrode with `0`.
`lambda_soft` scales the `betaВ·KL.detach()` term in phi gradient вЂ”
materially different gradient flow.

- `transformer/vfe/train_vfe.py`: removed the override so the live config
  matches the documented default.

### M15 вЂ” `prior_handoff_sigma=0.01` override contradicts CLAUDE.md

CLAUDE.md documents the vfe package profile as "sigma_p frozen at the
embedding value when `prior_handoff_sigma=0` (the default)." The entry
point overrode to `0.01`, silently running the blend branch.

- `transformer/vfe/train_vfe.py`: reverted to `0.0`.

### M16 вЂ” `softplus(raw_c0)` allocated every iteration

`e_step.py:231` recomputed `F_fn.softplus(self.raw_c0)` inside the E-step
loop. `raw_c0` is a leaf parameter unchanged by the inner loop.

- `transformer/vfe/e_step.py`: hoisted `alpha_c0_full_hoisted =
  F_fn.softplus(self.raw_c0)` to before the loop; the loop reuses the
  hoisted value via `alpha_c0_full = alpha_c0_full_hoisted`.

### M18 вЂ” `_append_extra` method doesn't exist on tracker

`trainer.py:526` guarded with `hasattr(tracker, '_append_extra')` but the
method exists nowhere in the tree вЂ” the VFE dynamics columns silently
dropped.

- `transformer/vfe/trainer.py`: added fallback to
  `tracker.log_step(step, **vfe_row)` with a warn-once diagnostic on
  failure.

# 2026-04-18 (afternoon) вЂ” `gauge_param='omega'` path operability

## Context

User requested a trace of the `gauge_param='omega'` path for
`em_mode='em_phi_p'`, noting the omega path is "not yet fully operable."
Three parallel agents (two tracers + independent code-reviewer verifier)
consolidated the defect list. DEQ fixed-point omega support is explicitly
deferred per user direction.

## CRITICAL вЂ” em_phi_p Г— omega contract violations (fixed)

`em_phi_p` means the gauge parameter is an M-step variable (Оё), frozen
during the E-step. The phi path enforces this via an entry-time
`phi.detach()` and an exit-time detach guard. The omega path did neither.

### V1 вЂ” E-step entry: detach omega under M_phi_p

`transformer/core/variational_ffn.py:1535-1541`. Added an analog of the
existing `if self.em_phi_mode == 'M_phi_p': phi_current = phi.detach()`
for omega. Without this, KL expressions inside every E-step iteration
accumulated semi-gradient into `omega_embed.weight` through beta and the
sandwich transport вЂ” turning em_phi_p into a de-facto straight-through
hybrid on the omega path.

### V2 вЂ” Post-loop gauge evolution guard

`transformer/core/variational_ffn.py:2528-2543`. The post-loop
`update_phi and not update_phi_per_iteration` block lacked the
`em_phi_mode != 'M_phi_p'` guard that the per-iter block at ~line 2214
already has. With `update_phi_per_iteration=False` + `em_phi_p`, this
block silently mutated phi/omega during the E-step, violating the freeze
contract.

### V3 вЂ” Closed-form gauge evolution guard

`transformer/core/vfe_closed_form.py:592-604`. Same missing guard in the
closed-form path. With `closed_form_e_step=True` + `em_phi_p`, phi/omega
were evolved regardless of EM mode.

### V4 вЂ” EM-boundary detach for omega

`transformer/core/variational_ffn.py:2598-2609`. Under `_em_active`,
mu/sigma were detached (always) and phi was detached (E_phi_q only).
Omega was never detached. The attached `_last_evolved_omega` was then
handed to the next block, compounding the E-step autograd graph across
L layers. Added `if gauge_param=='omega' and omega_current is not None:
omega_current = omega_current.detach()` to close the boundary.

## HIGH вЂ” Surrounding omega completeness (fixed)

### V5 вЂ” `omega_trust_region` wired from TrainingConfig to FFN

`TrainingConfig.omega_trust_region=0.3` existed but was never set on
the FFN. Every read site used `getattr(self, 'omega_trust_region', 0.3)`
and silently fell back to the default. Fixed by adding the field to
`BlockConfig` and propagating:

- `transformer/core/block_config.py`: added `omega_trust_region: float = 0.3`
  field and `config.get('omega_trust_region', 0.3)` in `from_config`.
- `transformer/core/blocks.py`: passes `cfg.omega_trust_region` into
  `VariationalFFNDynamic`.
- `transformer/core/variational_ffn.py`: accepts `omega_trust_region`
  kwarg and stores on `self.omega_trust_region`. Existing `getattr`
  fallbacks at iter/post-loop/closed-form retract sites still resolve
  to the wired value.

### V6 вЂ” `det_omega` analysis for omega path

`transformer/analysis/gauge_geometry.py:compute_gauge_invariants`. The
formula `det(Omega) = exp(tr(phi_i - phi_j))` is phi-only; on the omega
path it reads from a dummy zero phi tensor and reports `det(Omega)=1`
uniformly. Added optional `omega: Optional[Tensor]` argument; when
provided, `det_per_token = det(Omega_i)` is computed directly and
`det_omega_direct = det_i / det_j` overrides the phi-based result.

- `transformer/analysis/gauge_geometry.py`: added `Optional` import,
  new `omega=None` kwarg, direct-Omega branch, and final override inside
  the return dict.
- `transformer/analysis/publication_metrics.py:1619`: caller now passes
  `attn_info.get('omega', None)` into `compute_gauge_invariants`.
- `transformer/core/model.py:1493-1497`: `attention_info` now carries
  `'omega': omega` (last-layer evolved gauge frame) when
  `gauge_param=='omega'`, else `None`.

### V7 вЂ” PriorBank dummy `phi_embed` no longer drifts

`transformer/core/prior_bank.py:240-256`. Under `gauge_param='omega'`
+ `gauge_fixed_priors=False` the code instantiated a learnable
`nn.Embedding` initialized to zeros as an API stub. AdamW's momentum
+ weight decay caused these zero weights to drift (silent waste of
`vocab_size * phi_dim` parameters; asymmetric with the
`GaugeTokenEmbedding` omega branch which uses a non-learnable buffer).
Fixed by setting `requires_grad_(False)` on the stub weight вЂ”
`training/optimizer.py:672-673` already skips frozen params.

## V8 вЂ” Full-covariance omega gradient

`transformer/core/variational_ffn.py:_compute_omega_grad_direct`. The
function's 936-line early return `if sigma_current is None or not
is_diagonal: return None` silently disabled в€‚F/в€‚О© for the full-covariance
regime. Phi had analytic + autograd support already (see
`_compute_phi_grad:1166-1169`).

- Relaxed the gate to `if sigma_current is None: return None` only.
- Per-block sigma slicing now mirrors the phi path:
  - diagonal: `sigma_h = sigma_current[:, :, bs:be].detach()`
  - full-cov: `sigma_h = sigma_current[:, :, bs:be, bs:be].detach()`
- Downstream `compute_attention_weights(..., diagonal_covariance=False,
  sigma_q=sigma_h, ...)` already supported full-cov KL for both
  `gauge_param` branches, so no kernel changes were needed.
- New test `test_compute_omega_grad_direct_full_covariance` verifies the
  gradient is non-None, finite, correctly-shaped, and non-trivially zero
  on a full-cov sigma input.

## V9 вЂ” log|det(Omega)| penalty (Fix4 resolved)

Direct-Omega parameterization has no exp-map guarantee that О© в€€ GL(K).
Without regularization, О© can drift toward singular (det в†’ 0) or blow up
(det в†’ в€ћ), breaking the sandwich transport and the cocycle identity.
Penalty: О» В· mean_{i,h} (log|det О©_{i,h}|)ВІ. The minimum is |det О©_h| = 1,
so this drives the gauge toward the unit-determinant sheet while
permitting det < 0 (reflections are still representable).

- `transformer/training/config.py`: added field
  `omega_det_penalty: float = 0.0`.
- `transformer/train.py:compute_free_energy_loss`: added kwarg
  `omega_det_penalty`; penalty computed per-head via
  `torch.linalg.slogdet` on the per-block slice of the INITIAL (layer-0,
  embedding-lookup) omega, summed over blocks with mean-over-tokens;
  added to `total_loss` and reported in metrics under `loss/omega_det`.
- `transformer/core/model.py`: `attention_info` now exposes
  `'omega_initial'` вЂ” the per-token omega from `_embed_and_prepare`,
  attached to `omega_embed` so the penalty gradient reaches the
  embedding parameters. (The existing `'omega'` key still carries the
  last-layer EVOLVED omega, detached under em_phi_p, for diagnostics.)
- Wired through production training entry points:
  - `transformer/training/experiment_runner.py:733`
  - `transformer/training/train_fast.py:237`
  - `transformer/resume_training.py:459`
- Evaluation call sites use the default 0.0 (no penalty during eval).
- New tests: `test_omega_det_penalty_zero_at_identity`,
  `test_omega_det_penalty_positive_off_identity`,
  `test_omega_det_penalty_invariant_to_reflection`.

## V10 вЂ” `gauge_fixed_priors=True` + `gauge_param='omega'`

Under `gauge_fixed_priors=True` the model assumes all per-token priors
are gauge transforms of a shared base prior: `Ој_v = A_v @ Ој_0`,
`ОЈ_v = A_v ОЈ_0 A_vбµЂ`. The phi path obtains `A_v = exp(П†_v В· G)`; the
omega path was silently falling through to `A_v = exp(dummy_zero В· G) = I`
so every token received the same prior вЂ” a guaranteed collapse.

- `transformer/core/prior_bank.py:__init__`: the gauge-fixed branch now
  creates `omega_embed` (near-identity init) when `gauge_param='omega'`,
  plus a frozen zero `phi_embed` for API arity compatibility.
- `transformer/core/prior_bank.py:_get_prior_for_tokens`: adds an
  omega-first branch that transports `base_prior_mu` and `base_sigma`
  per-block using О©_v directly (mirrors the phi-path block-diagonal
  logic but skips the exp map). Returns `(mu_p, sigma_p, phi_dummy, omega)`.
- `transformer/core/embeddings.py:GaugeTokenEmbedding.forward`: same
  treatment вЂ” a new omega branch inside `if self.gauge_fixed_priors:`
  transports the base prior via `omega` blocks; the phi-based branches
  are gated behind `_skip_phi_gauge_transport` so the omega path does
  not consume the dummy zero phi.
- New test `test_omega_em_phi_p_with_gauge_fixed_priors` asserts that
  two different token IDs produce distinct `mu_prior` (catches the
  silent-collapse regression directly).

## V11 вЂ” Positional encoding on the omega path

`GaugePositionalEncoding.compose(phi, ...)` adds positional П† via BCH
so that `О©_i = exp((П†_i^tok + П†_i^pos)В·G)` has a position-dependent
gauge transport. On the omega path the dummy zero phi consumed the
positional signal, leaving omega positionless at the gauge level.

Added `GaugePositionalEncoding.compose_omega(omega, num_agents, device)`:
right-multiplies each token's О© by `R(i) = exp(П†_pos(i)В·G)`, giving
`О©_i^combined = О©_i^tok В· R(i)`. Under this definition
`О©_{ij} = О©_i^tok В· R(i) В· R(-j) В· (О©_j^tok)^{-1}` so the positional
contribution depends on relative position only вЂ” shift-invariance
matches the phi-path property.

- `transformer/core/embeddings.py`: new `compose_omega` method with
  short-circuit for `mode='none'` and safe fallback (no-op) when
  `self.generators is None`.
- `transformer/core/model.py:_embed_and_prepare`: calls `compose_omega`
  alongside the existing phi `compose` when `gauge_param='omega'`.

## V12 вЂ” Cross-head permutation for omega

`_apply_cross_head_perm` permutes the K-axis of Ој (and both K-axes of
full-cov ОЈ) so coupled heads are contiguous. Under `gauge_param='omega'`
the omega tensor has two K-axes and must also be sandwich-permuted,
otherwise attention consumes un-permuted omega blocks while VFE operates
on permuted Ој/Пѓ вЂ” a silent coordinate mismatch.

- `transformer/core/model.py:_apply_cross_head_perm`: extended to accept
  an optional `omega` kwarg and return a 3-tuple when provided. The
  sandwich is `omega[..., perm, :][..., :, perm]`.
- `transformer/core/model.py:_embed_and_prepare`: the forward-perm call
  now passes `omega=omega` and unpacks a 3-tuple when the omega path is
  active. Inverse-perm sites (before vocab projection) still use the
  2-tuple form because omega is purely internal and does not reach the
  logits projection.

## V13 вЂ” Invalidate stale `cached_head_transports` on evolve

The model precomputes `cached_head_transports` from the INITIAL embedding
omega. Once a block evolves omega via `_retract_omega` (em_modes other
than `em_phi_p`), the cache is stale: subsequent blocks would apply
attention transport built from the old omega while their FFN sees the
new one.

- `transformer/core/model.py:forward_with_attention`: after
  `omega = evolved_omega`, set `cached_head_transports = None` so the
  next block rebuilds from fresh omega. `blocks.py:586-611` already
  handles the rebuild when the incoming cache is None.
- `transformer/core/blocks.py:GaugeTransformerStack.forward`: mirror
  invalidation inside the stack's own layer loop.

Latent under `em_phi_p` (evolve is skipped by V2/V3) but ensures
correctness under `em_phi_q`, `straight_through`, and `ift_phi`.

## V14 вЂ” `omega_embed` weight decay = 0.0

`omega_embed` was placed in the generic `embed_wd` group at
`optimizer.py:766`, decaying every weight toward zero вЂ” i.e., toward
the SINGULAR matrix, exactly the direction the V9 log|det(О©)| penalty
is designed to prevent. Set `weight_decay=0.0` on the dedicated
`omega_embed` param group and rely on `omega_det_penalty` for
determinant control.

## V15 вЂ” End-to-end omega Г— em_phi_p forward+backward test

`tests/transformer/test_omega_gradient.py::test_omega_em_phi_p_end_to_end_forward_backward`
builds a full `GaugeTransformerLM` with `gauge_param='omega'` and
`em_mode='em_phi_p'`, runs forward+backward, and asserts:

- logits are finite of the correct shape,
- `attn_info['omega_initial']` is attached (M-step gradient path) while
  `attn_info['omega']` (evolved) is detached (em_phi_p contract),
- the free-energy loss is finite,
- `omega_embed.weight.grad` is non-None, finite, and non-trivially zero,
- the V9 `loss/omega_det` metric is recorded when the penalty is active.

Combined with `test_omega_em_phi_p_with_gauge_fixed_priors` (V10), this
closes the "no end-to-end omega forward test" gap in the TODO list.

### F7 вЂ” `plot_sigma_bhattacharyya_mds()` + `bhattacharyya_distance_matrix()`

Gaussian-aware ОЈ visualization. Unlike `plot_sigma_clustering`'s flat-PCA
panel (a), this figure uses a metric that respects the Gaussian geometry:
two tokens are close iff their full *distributions* p(x | token) overlap,
consuming both Ој and ОЈ jointly.

**New utility `bhattacharyya_distance_matrix(mu, sigma)`** вЂ” returns the
(n, n) symmetric non-negative Bhattacharyya distance matrix. Handles both
diagonal ОЈ (fully vectorized) and full-cov ОЈ (row-wise batched Cholesky).
Fallback path switches to `slogdet` + `linalg.solve` when Cholesky fails
on a degenerate pair. Symmetrizes, clamps to non-negative, zeros the
diagonal.

Verified numerically:
- 1D closed form: `D_B = (1/8)(Ој_1в€’Ој_2)ВІ / ПѓМ„ВІ + (1/2) ln(ПѓМ„ВІ / sqrt(ПѓВІ_1В·ПѓВІ_2))`
  matches the implementation to 1e-5 tolerance.
- Isotropic full-cov `ОЈ_i = s_iВ·I` matches diagonal `[s_i]*K` to 2.6e-6.
- D is exactly zero on diagonal, symmetric, non-negative.

**New figure `plot_sigma_bhattacharyya_mds(mu, sigma, вЂ¦)`** вЂ” 2-panel:

- (a) **Metric MDS** embedding via `sklearn.manifold.MDS(dissimilarity='precomputed')`
      from the pairwise D_B matrix, colored by category with the shared
      envelope overlay. Title reports the final stress value.
- (b) **Pairwise distance distribution** split into within-category vs
      between-category. Separation between the two distributions is a
      direct diagnostic of whether token categories cluster under the
      Bhattacharyya metric. Title reports the between/within mean ratio.

sklearn's `FutureWarning`s (`dissimilarity`/`init` renames coming in
1.10) are suppressed locally via `warnings.catch_warnings` to keep
training-run stdout clean.

**Wiring**: `analyze_gauge_semantics` calls the new figure when both
`mu_embed` and `sigma_embed` are available. Default `n_tokens=300` for

### F8 вЂ” Guard all PCA call sites against zero-variance inputs

User reported the sklearn warning:
```

  explained_variance_ratio_ = explained_variance_ / total_var
```

This fires from `sklearn.decomposition.PCA` when the input has zero total
variance. In this codebase it fires whenever:
- ОЈ is **shared** (single `(K,)` buffer broadcast to all tokens) вЂ” every
  token has identical covariance в†’ `reshape(V, K)` has zero variance.
- О© is **identically the identity** under `gauge_mode='trivial'` вЂ”
  `reshape(V, KВІ)` is constant.
- П† is at **initialization** (all zeros) вЂ” never diverged.
- Any early-training step before the embeddings differentiate.

1. Returns `(None, None, status)` for degenerate inputs (zero variance,

2. Runs PCA inside a `warnings.filterwarnings('ignore', RuntimeWarning)`
   context as a belt-and-suspenders against residual numerical warnings.
3. Pads output to the requested `n_components` with zero-variance columns

## R1 вЂ” `resume_training.py` emits SO(3)-format `irrep_spec` for GL(K) checkpoints

### Symptom

Resuming any GL(K) multi-head checkpoint (e.g.
`1_ffn_VFE_dynamic/checkpoint_step_149999.pt`, embed_dim=80, 4 heads Г— 20)
failed during `GaugeTransformerLM(config)` construction with:

```
ValueError: GL(K) multi-head: n_heads(1) Г— d_head(20) = 20 must equal
embed_dim=80.
```

### Root cause

`infer_config_from_state_dict` reads per-head generator shapes
(`head_generators.{h}.gen`) and always emits SO(3)-style irrep entries
of the form `[('в„“в„“', 1, dim), вЂ¦]` вЂ” one tuple per head. For a GL(K)
multi-head model, the correct format is a single tuple
`[('fund', n_heads, d_head)]` where the second element is the head
count. `attention.py` (GL(K) branch) reads `irrep_spec[0]` and unpacks
`(label, n_heads, d_head)`, so the SO(3)-style emission was parsed as
`n_heads=1, d_head=20`, triggering the mismatch against embed_dim=80.

Additionally, the config-merge order in `resume_training()`
(`json < ckpt < inferred`) means the inferred (incorrect) `irrep_spec`
clobbered the correct value already present in
`experiment_config.json`, and `gauge_group` was never inferred at all вЂ”
falling back to the `'SO3'` default applied in `model_defaults`.

### Fix

`transformer/resume_training.py` вЂ” rewrite the irrep inference block to
detect GL(K) vs SO(3) from two signals:

1. **Head-dim parity.** SO(3) irreps require odd dimensions (`2в„“+1`);
   even dims в‡’ must be GL(K).
2. **Global generator count cross-check.** SO(3) yields
   `n_gen = 3 Г— n_heads` total generators; GL(K) block-diagonal
   multi-head yields `n_gen = n_heads Г— d_headВІ`. When head dims are
   uniform and the global `generators` buffer is present, the count
   unambiguously distinguishes the two.

When GL(K) is detected, emit
`irrep_spec=[['fund', n_heads, d_head]]`, `gauge_group='GLK'`, and
`gauge_dim=d_head`. When non-uniform head dims appear alongside a GL(K)
signal (indicating cross-head coupling with super-blocks), raise with a
clear message directing the user to supply `experiment_config.json` вЂ”
`cross_couplings` cannot be reconstructed from state_dict alone.

Also scoped the inference loop to `blocks.0` only (all blocks share the
same head structure, so collecting across every layer was redundant).

### Verification

```
1_ffn_VFE_dynamic/checkpoint_step_149999.pt:
  embed_dim: 80, irrep_spec: [['fund', 4, 20]], gauge_group: GLK,
  gauge_dim: 20  вњ“  (matches experiment_config.json)

141_K=20_GL(10)_N=128_batch=16/checkpoint_step_49999.pt:
  embed_dim: 20, irrep_spec: [['fund', 2, 10]], gauge_group: GLK,
  gauge_dim: 10  вњ“
```
