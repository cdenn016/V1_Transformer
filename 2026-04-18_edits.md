# 2026-04-18 — Deep VFE Package Audit Fixes

## Context

Five specialist agents (code-reviewer, data-engineer, performance-engineer,
general-purpose-as-ml-engineer, debugger) ran parallel audits of
`transformer/vfe/`. Findings were consolidated and triaged into
CRITICAL / HIGH / MEDIUM / LOW by severity. This commit lands the surgical
fixes; structural perf refactors are deferred and noted at bottom.

All 715 pre-existing tests pass after these changes.

---

## CRITICAL (fixed)

### C1 — `norm_type='layernorm'` silent no-op

`train_vfe.py` selected `'layernorm'` but `block.py`/`model.py` only branched
on `'mahalnorm' | 'rmsnorm'`; the unknown string fell through to
`self.norm = None`, meaning training ran with no layer normalization
whatsoever and no warning.

- `transformer/vfe/block.py`: added explicit `'none'` branch plus
  `else: raise ValueError(...)` for unrecognized strings.
- `transformer/vfe/model.py`: same fix on `final_norm`.
- `transformer/vfe/train_vfe.py`: changed default to `'rmsnorm'`.

### C2 — CE `ignore_index=-1` but dataset pads with `-100`

`model.py:137` and `trainer.py:326` used `-1`. The dataset pads targets with
`-100`, so either CE crashed under strict PyTorch builds or silently
corrupted loss and val-PPL whenever tail padding fired.

- `transformer/vfe/model.py`: `ignore_index=-1` → `-100`.
- `transformer/vfe/trainer.py`: `(target_ids != -1)` → `(target_ids != -100)`.

### C3 — `_update_phi` runs attention with wrong transport kernel

The main E-step at `e_step.py:286` passes
`exact_diagonal_transport=self.exact_diagonal_transport`. The
`_update_phi` path at `e_step.py:416` omitted it, defaulting to `False`.
Under the common config (`diagonal_covariance=True`,
`exact_diagonal_transport=True`) this means the phi gradient descends a
different KL than the mu/sigma gradients — phi converges to a gauge frame
misaligned with the objective being minimized.

- `transformer/vfe/e_step.py`: threaded
  `exact_diagonal_transport=self.exact_diagonal_transport` into the
  `_update_phi` call to `compute_kl_attention`.

### C4 — `sigma_p` inverse recomputed every E-step iteration

`vfe_gradients.py` recomputes `_safe_spd_inv(sigma_p)` and `slogdet(sigma_p)`
on every inner iteration even though `sigma_p` is frozen at the embedding
value for the full-cov path. Deferred: hoisting requires adding an optional
precomputed argument to `compute_vfe_gradients_gpu` which has three call
sites across the vfe, legacy ffn, and deq paths. Listed as a structural
perf optimization to track separately.

---

## HIGH (fixed)

### H5 — `prior_handoff_phi` was a silent no-op

`stack.py:107` wrote `priors.phi` conditionally on the flag but
`e_step.py:198` initializes `phi = beliefs.phi` and never reads
`priors.phi`. The flag did nothing.

- `transformer/vfe/stack.py`: replaced the conditional write with
  `new_prior_phi = initial_priors.phi` unconditionally plus a comment
  explaining phi flows via `beliefs`, not `priors`.
- `transformer/vfe/config.py`: added `DeprecationWarning` in
  `__post_init__` when the flag is set; doc-comment updated.
- `transformer/vfe/train_vfe.py`: changed default from `True` to `False`.

### H6 — No seed in entry point

`train_vfe.py` never called `torch.manual_seed` / `numpy.random.seed`;
runs were non-reproducible.

- `transformer/vfe/train_vfe.py`: added `SEED = 1234` constant and explicit
  `random.seed` / `np.random.seed` / `torch.manual_seed` /
  `torch.cuda.manual_seed_all` calls at the top of `__main__`.

### H7 — `allclose` generator skew-check forces GPU sync every iteration

`attention.py:42` called `torch.allclose(generators, -generators.T, atol=1e-6)`
inside `compute_gauge_transport`, which fires on every E-step iteration.
`generators` is an immutable registered buffer.

- `transformer/vfe/attention.py`: added `_SKEW_CACHE` keyed by
  `id(generators)` and an optional `skew_symmetric: bool` kwarg to
  `compute_gauge_transport`. The `allclose` now fires once per buffer
  lifetime, not once per iteration.

### H10 — CSV "loss component" columns hard-coded to zero

`trainer.py` wrote `train_loss_belief_align`, `train_loss_self_consistency`,
`train_loss_model_coupling`, `train_loss_aux_layer_ce` all set to `0`.
Publication plots reading those columns showed flat-zero curves, which is
misleading.

- `transformer/vfe/trainer.py`: removed the placeholder zero columns. The
  VFE objective is minimized implicitly via the E-step inner loop, so the
  decomposed components aren't directly available without additional
  instrumentation.

### H11 — `initial_priors` aliases belief tensors, enabling in-place mutation

`model.py:120` set `initial_priors = beliefs`. The stack then clamps
`new_prior_sigma[..., diag_idx, diag_idx]` in place (`stack.py:102`). When
the frozen branch (`rho_sigma == 0`) reuses `initial_priors.sigma`
directly, in-place writes by downstream code would leak into the posterior
through the alias.

- `transformer/vfe/model.py`: `initial_priors` now constructed explicitly
  with cloned `sigma`/`phi`:
  ```python
  initial_priors = BeliefState(
      mu=beliefs.mu,
      sigma=beliefs.sigma.clone(),
      phi=beliefs.phi.clone(),
  )
  ```

### H19 — Full-cov sigma handoff clamps only diagonal, doesn't enforce SPD

`stack.py:97-104` built the blended prior sigma as
`(1-ρ)·Σ_embed + ρ·Σ_post` and clamped only the diagonal. If the
posterior's off-diagonals are large the result could be indefinite.

- `transformer/vfe/stack.py`: added `0.5 * (Σ + Σ.T)` symmetrization
  before the diagonal clamp.

---

## MEDIUM (fixed)

### M12 — `rope_full_gauge` silently promoted

`config.py:148` auto-switched `rope_full_gauge=False → True` whenever
`diagonal_covariance=False` and `use_rope=True`. A checkpoint's config
would round-trip as `False` but the runtime would execute the `True`
branch.

- `transformer/vfe/config.py`: replaced the silent promotion with an
  explicit `ValueError` requiring the user to set the flag.
- `tests/transformer/test_vfe_package.py`: updated
  `test_full_cov_allowed` to pass `rope_full_gauge=True` explicitly.

### M13 — `rope_full_gauge` branch gated on inference mode, not grad mode

`e_step.py:234` checked `not torch.is_inference_mode_enabled()`. Under
`@torch.no_grad()` blocks in `generate()` and `evaluate()` the branch
still fires, even though no gradients are needed.

- `transformer/vfe/e_step.py`: added `and torch.is_grad_enabled()` to
  the guard.

### M14 — `lambda_soft` entry-point override contradicts schema default

`config.py:37` default was `1.0`; `train_vfe.py:50` overrode with `0`.
`lambda_soft` scales the `beta·KL.detach()` term in phi gradient —
materially different gradient flow.

- `transformer/vfe/train_vfe.py`: removed the override so the live config
  matches the documented default.

### M15 — `prior_handoff_sigma=0.01` override contradicts CLAUDE.md

CLAUDE.md documents the vfe package profile as "sigma_p frozen at the
embedding value when `prior_handoff_sigma=0` (the default)." The entry
point overrode to `0.01`, silently running the blend branch.

- `transformer/vfe/train_vfe.py`: reverted to `0.0`.

### M16 — `softplus(raw_c0)` allocated every iteration

`e_step.py:231` recomputed `F_fn.softplus(self.raw_c0)` inside the E-step
loop. `raw_c0` is a leaf parameter unchanged by the inner loop.

- `transformer/vfe/e_step.py`: hoisted `alpha_c0_full_hoisted =
  F_fn.softplus(self.raw_c0)` to before the loop; the loop reuses the
  hoisted value via `alpha_c0_full = alpha_c0_full_hoisted`.

### M18 — `_append_extra` method doesn't exist on tracker

`trainer.py:526` guarded with `hasattr(tracker, '_append_extra')` but the
method exists nowhere in the tree — the VFE dynamics columns silently
dropped.

- `transformer/vfe/trainer.py`: added fallback to
  `tracker.log_step(step, **vfe_row)` with a warn-once diagnostic on
  failure.

---

## LOW (fixed)

### L20 — `n_e_steps=0` silently produces identity layers

If someone sets `n_e_steps=0` the E-step loop never executes and the
layer returns the unchanged input.

- `transformer/vfe/e_step.py`: `VFEEStep.__init__` now raises `ValueError`
  for `n_e_steps < 1`.

### L21 — `torch.tensor` without `dtype=`

`model.py:206` (`torch.tensor([[next_id]], device=...)` — no `dtype`)
and `efe.py:193` (`torch.tensor(0.0, device=device)` — no `dtype`)
could fail under mixed precision training.

- `transformer/vfe/model.py`: added `dtype=ids.dtype`.
- `transformer/vfe/efe.py`: added `dtype=risks[0].dtype` with a `float32`
  fallback.

### L22 — Diagnostics overwrite per-layer on every iteration

`_collect_e_step_diagnostics` looped through all layers but wrote only
the last layer's diag.

- `transformer/vfe/trainer.py`: now averages numeric metrics across layers
  and exposes the final-layer values under a `_final` suffix.

---

## Deferred (structural — listed for follow-up)

- **C4** (sigma_p inverse hoist): requires extending
  `compute_vfe_gradients_gpu` signature to accept a precomputed
  `sigma_p_inv` and threading it through three call sites (vfe/e_step.py,
  legacy core/variational_ffn.py, core/vfe_deq.py). Target: future perf pass.
- **Perf #3** (`_update_phi` double attention forward): the phi-gradient
  path rebuilds the full attention computation from `phi_for_grad` because
  the outer cache was built from the outer `phi`. A rewrite that shares
  the computation graph would halve per-iteration cost but is not
  surgical.
- **Perf #5** (NaN check GPU syncs): 4-6 `isnan().any()` calls per iter
  in `vfe_gradients.py`. Replacement with unconditional `nan_to_num`
  requires regression testing of training stability (a prior revert is
  commented near `vfe_gradients.py:350-354`).
- **L23** (`generators` buffer triplicated): each of VFEModel, VFEPriorBank,
  VFEEStep registers the same tensor. Small memory waste, no correctness
  issue.

# 2026-04-18 (afternoon) — `gauge_param='omega'` path operability

## Context

User requested a trace of the `gauge_param='omega'` path for
`em_mode='em_phi_p'`, noting the omega path is "not yet fully operable."
Three parallel agents (two tracers + independent code-reviewer verifier)
consolidated the defect list. DEQ fixed-point omega support is explicitly
deferred per user direction.

## CRITICAL — em_phi_p × omega contract violations (fixed)

`em_phi_p` means the gauge parameter is an M-step variable (θ), frozen
during the E-step. The phi path enforces this via an entry-time
`phi.detach()` and an exit-time detach guard. The omega path did neither.

### V1 — E-step entry: detach omega under M_phi_p

`transformer/core/variational_ffn.py:1535-1541`. Added an analog of the
existing `if self.em_phi_mode == 'M_phi_p': phi_current = phi.detach()`
for omega. Without this, KL expressions inside every E-step iteration
accumulated semi-gradient into `omega_embed.weight` through beta and the
sandwich transport — turning em_phi_p into a de-facto straight-through
hybrid on the omega path.

### V2 — Post-loop gauge evolution guard

`transformer/core/variational_ffn.py:2528-2543`. The post-loop
`update_phi and not update_phi_per_iteration` block lacked the
`em_phi_mode != 'M_phi_p'` guard that the per-iter block at ~line 2214
already has. With `update_phi_per_iteration=False` + `em_phi_p`, this
block silently mutated phi/omega during the E-step, violating the freeze
contract.

### V3 — Closed-form gauge evolution guard

`transformer/core/vfe_closed_form.py:592-604`. Same missing guard in the
closed-form path. With `closed_form_e_step=True` + `em_phi_p`, phi/omega
were evolved regardless of EM mode.

### V4 — EM-boundary detach for omega

`transformer/core/variational_ffn.py:2598-2609`. Under `_em_active`,
mu/sigma were detached (always) and phi was detached (E_phi_q only).
Omega was never detached. The attached `_last_evolved_omega` was then
handed to the next block, compounding the E-step autograd graph across
L layers. Added `if gauge_param=='omega' and omega_current is not None:
omega_current = omega_current.detach()` to close the boundary.

## HIGH — Surrounding omega completeness (fixed)

### V5 — `omega_trust_region` wired from TrainingConfig to FFN

`TrainingConfig.omega_trust_region=0.3` existed but was never set on
the FFN. Every read site used `getattr(self, 'omega_trust_region', 0.3)`
and silently fell back to the default. Fixed by adding the field to
`BlockConfig` and propagating:

- `transformer/core/block_config.py`: added `omega_trust_region: float = 0.3`
  field and `config.get('omega_trust_region', 0.3)` in `from_config`.
- `transformer/core/blocks.py`: passes `cfg.omega_trust_region` into
  `VariationalFFNDynamic`.
- `transformer/core/variational_ffn.py`: accepts `omega_trust_region`
  kwarg and stores on `self.omega_trust_region`. Existing `getattr`
  fallbacks at iter/post-loop/closed-form retract sites still resolve
  to the wired value.

### V6 — `det_omega` analysis for omega path

`transformer/analysis/gauge_geometry.py:compute_gauge_invariants`. The
formula `det(Omega) = exp(tr(phi_i - phi_j))` is phi-only; on the omega
path it reads from a dummy zero phi tensor and reports `det(Omega)=1`
uniformly. Added optional `omega: Optional[Tensor]` argument; when
provided, `det_per_token = det(Omega_i)` is computed directly and
`det_omega_direct = det_i / det_j` overrides the phi-based result.

- `transformer/analysis/gauge_geometry.py`: added `Optional` import,
  new `omega=None` kwarg, direct-Omega branch, and final override inside
  the return dict.
- `transformer/analysis/publication_metrics.py:1619`: caller now passes
  `attn_info.get('omega', None)` into `compute_gauge_invariants`.
- `transformer/core/model.py:1493-1497`: `attention_info` now carries
  `'omega': omega` (last-layer evolved gauge frame) when
  `gauge_param=='omega'`, else `None`.

### V7 — PriorBank dummy `phi_embed` no longer drifts

`transformer/core/prior_bank.py:240-256`. Under `gauge_param='omega'`
+ `gauge_fixed_priors=False` the code instantiated a learnable
`nn.Embedding` initialized to zeros as an API stub. AdamW's momentum
+ weight decay caused these zero weights to drift (silent waste of
`vocab_size * phi_dim` parameters; asymmetric with the
`GaugeTokenEmbedding` omega branch which uses a non-learnable buffer).
Fixed by setting `requires_grad_(False)` on the stub weight —
`training/optimizer.py:672-673` already skips frozen params.

## DEFERRED / TODO

Per user direction: DEQ omega support is deferred. Other incomplete
items from the audit, left as explicit TODOs:

- **Full-covariance partial F / partial Omega.** `variational_ffn.py:_compute_omega_grad_direct`
  still `return None` when `not is_diagonal`. Phi has full-cov analytic
  and autograd paths. Non-trivial to implement; not needed for the
  diagonal-covariance publication config.
- **DEQ omega fixed-point.** `_make_deq_step_fn*` enumerates `(mu, sigma, phi)`
  only. `DEQFixedPoint(Full).apply` will not accelerate or implicitly-
  differentiate through an omega E-step. Deferred per user.
- **`gauge_fixed_priors=True` + omega.** Currently the omega branch in
  `prior_bank.py` and `embeddings.py` only fires in the non-gauge-fixed
  branch. Gauge-fixed priors under omega parameterization are not
  implemented.
- **Position encoding on omega path.** `GaugePositionalEncoding` composes
  onto the dummy `phi_base` in omega mode, folding positional information
  into an unused tensor. Omega path is effectively positionless at the
  gauge-transport level (RoPE on mu is the only positional signal).
- **Cross-head permutation for omega.** `model._apply_cross_head_perm`
  permutes mu/sigma but not omega. Latent under `use_cross_head_perm=False`
  (the default) but will fire silently when enabled with
  `gauge_param='omega'`.
- **Stale `cached_head_transports`.** Precomputed once at model-level
  from the original embedding omega and never refreshed after cross-layer
  omega evolution. Latent under `em_phi_p` (omega does not evolve once
  V1-V4 are in place) but will fire under other em_modes.
- **`log|det(Omega)|` penalty (TODO Fix4).** `transport_ops.py:441-446`
  proposes an `omega_det_penalty` hyperparameter to keep Omega invertible
  without an exp-map guarantee. Not wired.
- **`omega_embed` weight-decay direction.** Optimizer places `omega_embed`
  in the generic `embed_wd` group, decaying toward zero (singular).
  Should either be `wd=0` or a decay-to-identity penalty.
- **End-to-end omega forward test.** `tests/transformer/test_omega_gradient.py`
  covers unit-level transport/retraction/init but no full-model forward
  under `gauge_param='omega'` and no combination with `em_phi_p`.
