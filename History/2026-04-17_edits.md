# 2026-04-17 Edits

## Phase K — Fix: `get_bayesian_alpha` mixed diagonal / full-cov σ shapes

### Failure mode

Production run with `EM_CONFIG` (full-cov mode, `diagonal_covariance=False`, `use_prior_bank=True`) crashed:

```
File transformer/core/variational_ffn.py:830 in get_bayesian_alpha
    (1.0 - alpha_div) * sigma_q_diag + alpha_div * sigma_p_diag
RuntimeError: The size of tensor a (20) must match the size of tensor b (64) at non-singleton dimension 2
```

### Root cause

`get_bayesian_alpha` used a single `is_diagonal = (sigma_p.dim() == 3)` flag to decide both tensors' shape handling. In full-cov mode with `PriorBank`:

- `sigma_p` arrives as `(B, N, K)` — PriorBank encodes diagonal prior variances per token.
- `sigma_q` arrives as `(B, N, K, K)` — the E-step evolved the posterior to full covariance.

The single `is_diagonal` flag picked the diagonal branch based on `sigma_p` and left `sigma_q` untouched, so the blend line tried to add `(B, N, 20) + (B, N, 20, 20)` and broadcast-failed.

Phase J (2026-04-16) introduced the new Rényi α-divergence branch that exercises this path more heavily. The pre-existing KL branch had the same latent vulnerability — the old `sigma_p_inv = _safe_spd_inv(sigma_p)` in the `else` branch also assumed both were full. Neither was hit under diagonal-only training configs.

### Fix

`transformer/core/variational_ffn.py:get_bayesian_alpha` — detect shape per tensor and handle all four combinations robustly:

```python
is_p_diag = (sigma_p.dim() == 3)
is_q_diag = (sigma_q.dim() == 3)
sigma_p_diag = (sigma_p if is_p_diag else sigma_p.diagonal(dim1=-2, dim2=-1)).clamp(min=eps)
sigma_q_diag = (sigma_q if is_q_diag else sigma_q.diagonal(dim1=-2, dim2=-1)).clamp(min=eps)
```

- **Rényi branch (`α_div ≠ 1`):** uses per-dim diagonals uniformly — already a diagonal proxy, unchanged in behavior for the pure cases.
- **KL branch (`α_div = 1`):** three sub-branches preserving precision where possible:
  - Both diagonal → closed-form per-dim KL (unchanged).
  - Both full → original matrix-inverse path preserving off-diagonal correlations (unchanged).
  - Mixed → diagonal proxy on both (new path, previously would have crashed).

Removed the dead `is_diagonal = (sigma_p.dim() == 3)` variable.

### Tests

Added 4 parametrized regression tests in `tests/transformer/test_bayesian_alpha.py::test_mixed_diagonal_and_full_sigma` — covers all four shape combinations `(p_shape × q_shape)` ∈ `{diag, full}²` across both KL and Rényi branches. Asserts finiteness, positivity, correct output shape, and agreement with the pure-diagonal reference to floating-point tolerance.

### Verification

- `pytest tests/` → **671 passed, 4 skipped** (was 667 pre-fix; +4 new parametrized tests).
- The four production shape combinations (prior diag + post full, prior full + post diag, both diag, both full) each verified finite and correct via a scripted smoke check before the test suite.

### Files touched

`transformer/core/variational_ffn.py`, `tests/transformer/test_bayesian_alpha.py`, `2026-04-17_edits.md` (new).

## Phase L — Cross-Head GL(K) Coupling Audit & Fixes

### Audit findings

A user-supplied audit identified six concerns about the multi-head GL(K) cross-head coupling subsystem. Three independent verifier sub-agents (read-only `Explore` agents) confirmed five and refuted one (the `per_head_flatness_gate` sizing bug had already been removed on 2026-04-16, see Phase F in `History/2026-04-16_edits.md`). The five live findings:

1. **Lie subalgebra non-closure.** `generate_glK_cross_head_generators` at `math_utils/generators.py:912` builds a basis comprising every diagonal `gl(d_head)` block plus only the explicitly listed off-diagonal `(a,b)` blocks. The basis is generically not closed under the matrix commutator. `glK_bracket_torch:2057` computes `A1 @ A2 - A2 @ A1` and then re-projects onto the basis via `extract_glK_coords_torch:2092` (Frobenius inner product). For chains like `[(0,1), (1,2)]`, `[E^{(0,1)}, E^{(1,2)}]` lives in an `E^{(0,2)}` block absent from the basis, so the projection silently drops it. `glK_compose_bch_torch:2140` compounds this through nested brackets up to degree-5 BCH.
2. **`use_block_diagonal_kl=False` + `cross_couplings` is a silent coordinate-frame mismatch.** `model.py:242` set `ffn_irrep_dims=None` whenever the flag is off, but the cross-head builder unconditionally reordered generators into super-block coordinates, and `attention.py:1421` then fell back to `[d_head]*n_heads`. No guard.
3. **`cross_couplings` not deduplicated.** The loop at `generators.py:980` accepted duplicates blindly; pairs are directional, so dedup must remove only exact `(a,b)==(a,b)` duplicates and leave `(b,a)` alone.
4. **Super-block κ semantics.** `attention.py:1458` declared `log_kappa_per_head` with shape `len(self.irrep_dims)`. In the coupled path that equals `len(super_block_dims)`, so κ is per-super-block, not per original head. Behavior was correct; only naming and logging were misleading.
5. **README doc lag.** `README.md:210-218` described multi-head GL(K) only as the block-diagonal subgroup `GL(d_head)^H` and did not mention `cross_couplings`. The TeX manuscript covers it; the README was the lagging artifact.

A bonus issue: the redundant `cross_head_perm is not None` guard at `blocks.py:277` was harmless but obscured intent (the canonical condition is whether `ffn_irrep_dims` was forwarded by `model.py`).

### Edits

* `math_utils/generators.py`
  * Added `_dedup_cross_couplings` (private helper; preserves order, removes exact tuple duplicates, warns once with the dropped indices).
  * Wired the helper into `generate_glK_cross_head_generators` and `merge_coupled_heads` so both paths see the canonical list.
  * Added `validate_generator_closure(generators, tol=1e-6) -> dict` returning `{closed, max_residual, n_pairs, n_offending_pairs, offending_pairs}`. Reuses the same Frobenius-inner-product projection as `extract_glK_coords_torch`. Cost is O(n_gen² · K³); intended as a one-shot startup check.
  * Added `close_under_brackets(generators, max_iter=10, max_dim=K², tol=1e-6) -> (closed, info)` implementing iterative Lie-closure construction with thin-SVD deflation (chosen over classical Gram-Schmidt for numerical stability when sparse-block residuals are nearly but not exactly mutually orthogonal). Caps at `max_dim` and warns when closure walks toward full `gl(K)`.
  * Strengthened the docstrings of `glK_bracket_torch`, `extract_glK_coords_torch`, and `glK_compose_bch_torch` with `.. warning::` blocks documenting the projected-approximation regime when `n_gen != K**2` and pointing to the new validation/closure utilities.
  * Note: a repo-wide grep confirmed `glK_bracket_torch` / `glK_compose_bch_torch` / `extract_glK_coords_torch` have no callers outside `math_utils/generators.py` itself. They are dead on the live forward path today, so the non-closure issue is currently a documentation/honesty fix and a forward-proofing measure for any future caller, not a runtime bug.
* `transformer/core/model.py`
  * Imported `_dedup_cross_couplings`, `validate_generator_closure`, `close_under_brackets` from `math_utils.generators`.
  * After reading `cross_couplings` from config, applied the dedup helper.
  * Added a hard-error `ValueError` when `cross_couplings` is non-empty and `use_block_diagonal_kl=False` (the silent coordinate-frame mismatch).
  * Added `validate_cross_head_closure: bool = True` config flag (default on). Runs `validate_generator_closure` on the post-reorder basis and emits a `logger.warning` quoting the worst offending pairs and pointing to the auto-close opt-in.
  * Added `auto_close_cross_head_basis: bool = False` config flag. When set, runs `close_under_brackets` on the post-reorder basis, replaces `self.generators`, and updates `self.phi_dim`. Emits a follow-up warning when closure adds generators that may span the user-supplied super-block partition (since `self._super_block_dims` and `_cross_head_perm` still reflect the pre-closure structure).
* `transformer/core/blocks.py`
  * Replaced the redundant `cfg.cross_head_perm is not None` guard with `cfg.ffn_irrep_dims is not None` at `:277`. Forwards the override whenever `model.py` chose to provide one. Verified via `block_config.py:282, 471` that `ffn_irrep_dims` is a populated `BlockConfig` field.
* `transformer/core/attention.py`
  * Added a clarifying comment block above `log_kappa_per_head = nn.Parameter(...)` documenting that the parameter is per-super-block in the coupled path (not per original head). The parameter name is preserved for `state_dict` compatibility — no rename.
  * Sharpened the existing `[GL(K) cross-head] super-blocks=...` `logger.info` to mention that per-block parameters are per-super-block, not per-original-head, and to record the original head count.
* `README.md`
  * Replaced the `### Multi-Head Attention as Block-Diagonal GL(K)` subsection with two: `### Block-Diagonal Multi-Head (Default)` (uncoupled `GL(d_head)^H`, accurate as written) and a new `### Cross-Head Coupling (Sparse gl(K) Subspace)` documenting the algebraic structure, the non-closure caveat, the projected-approximation behavior of BCH, the super-block κ semantics, the hard incompatibility with `use_block_diagonal_kl=False`, and the dedup behavior. Avoided the banned-words list from `CLAUDE.md`.

### Edits

#### bug_003 — alpha_c0 product-rule correction in `VFEEStep`

When `E_learnable_alpha=True` (the default per `transformer/vfe/train_vfe.py:48`), the effective `alpha = c0 / (b0 + KL)` is a function of `(μ, σ)` via the per-dim KL. Differentiating `α·KL` wrt `θ` therefore carries a product-rule term `−(α²/c0)·KL·(dKL/dθ)`. `compute_vfe_gradients_gpu` already implements the correction — gated on `alpha_c0 is not None` (`transformer/core/vfe_gradients.py:1610, 1628, 1663`) — and legacy callers `transformer/core/variational_ffn.py:1843, 1876` and `transformer/core/vfe_deq.py:363, 508` correctly pass `alpha_c0=c0_h`. But the new VFE-package call sites omitted the kwarg.

* `transformer/vfe/e_step.py` — at the top of the inner E-step loop, compute `alpha_c0_full = F.softplus(self.raw_c0)` whenever `E_learnable_alpha=True`. Pass `alpha_c0=alpha_c0_full` to the standard `compute_vfe_gradients_gpu` call (formerly at line 287). Pass `alpha_c0=alpha_c0_full[block_start:block_end]` to the per-head `_compute_rope_full_gauge_gradient_per_head` call (formerly at line 239).
* `transformer/core/vfe_gradients.py` — extended `_compute_rope_full_gauge_gradient_per_head` (line 2025) signature with `alpha_c0: Optional[torch.Tensor] = None`. Added a phantom autograd term `−0.5·(α²/c0)·KL²` (with `α` and `c0` detached) inside `F_self` for both the diagonal and full-covariance branches. Differentiating this term reproduces exactly the missing `−(α²/c0)·KL·(dKL/dθ)` correction under `torch.autograd.grad`, matching the analytic form used by the legacy path.

#### bug_004 — `partition_generators_by_block` regression for SO(N) multi-irrep

`transformer/core/blocks.py:431` unconditionally called `partition_generators_by_block` for the `non_flat_transport=True` branch. The function (`transformer/core/connection.py:147-237`) raises `ValueError` at lines 225-230 when any generator's per-block Frobenius mass falls below `localization_threshold=0.999`. `generate_multi_irrep_soN_generators` writes the same SO(N) generators into every multiplicity copy of each non-scalar irrep, so per-block mass is exactly `1/mult` — failing the threshold whenever `mult ≥ 2`. Pre-refactor (commit c9987d87) used a single global `GaugeConnection(d_head=cfg.embed_dim, ...)` and worked.

* `transformer/core/blocks.py` (around line 431) — wrapped the partition call in `try` / `except ValueError`. On failure, fall back to a single global `GaugeConnection` over the full `embed_dim`, emit a `RuntimeWarning` explaining why, and tag the block with `self._connection_mode = 'global'` (vs `'per_head'`) for diagnostics. Output shape `(B, N, N, n_gen_total)` matches between the two paths so `compute_transport_operators` is unchanged.

#### bug_005 — `gauge_covariant_ridge` silently degrades under non-flat transport

`aggregate_messages` at `transformer/core/attention.py:1159-1169` populates `exp_phi_legacy` only when the cache carries BOTH `'Omega'` AND `'exp_phi'`. Two non-flat producers emit Omega-only caches. With `gauge_covariant_ridge=True`, the downstream branches at `attention.py:1210-1219, 1248-1254` then silently fall back to `eps·I` instead of the advertised `eps·(g·gᵀ)`. The flag becomes a no-op exactly where gauge covariance matters most.

* `transformer/core/blocks.py` (around line 559) — captured `exp_phi_full = transport['exp_phi']` (already returned by `compute_transport_operators` per `transport_ops.py:397-401`) and added `'exp_phi': exp_phi_full[..., dim_start:dim_start+d, dim_start:dim_start+d]` alongside `'Omega'` in each per-head cache entry.
* `transformer/core/variational_ffn.py` (around line 1714, 1793) — the multihead-helper code path that constructs `_head_ct = {'Omega': ...}` is dead today (`_nonflat_omega` is always `None` per the wiring at line 1985), but the cache producer is wired in. Hardened it: added `_nonflat_exp_phi` parameter to the helper signature; require both tensors together (`raise ValueError` if `_nonflat_omega` is set without `_nonflat_exp_phi`); populate `'exp_phi'` from the slice. Future enablement now cannot reintroduce the silent degradation.

#### device-mismatch hotfix

* `transformer/core/model.py:1240` — `torch.stack([_mu_in_norm, _mu_out_norm, _delta_norm, _phi_norm, _mu_attn_norm, _mu_ffn_norm, _sigma_mean, _mu_pos_std])` mixed CUDA tensors (real norms) with CPU `torch.tensor(0.0)` fallbacks created when `_mu_attn`, `_mu_ffn`, or `sigma_q` was `None`. ## Phase N — Tri-state `rope_full_gauge` ('off' | 'vfe_only' | 'both') + RoPE chain-rule hard-error

### Motivation

The user asked whether `rope_full_gauge` should apply σ → RΣRᵀ in BOTH the attention sublayer and the FFN VFE iteration. Up to now the flag was a `bool` wired only into the FFN's per-head VFE gradient helper (`_compute_rope_full_gauge_gradient_per_head` in `transformer/core/vfe_gradients.py:2025`); the attention sublayer hardcoded the standard-transformer pattern of rotating only μ and leaving σ raw — which `transformer/core/attention.py:301-322` itself acknowledges "departs from that prescription" and is "neither a textbook KL between rotated Gaussians nor the manuscript's full gauge interpretation."

The asymmetry is real: under strict GL(K) covariance the attention β values and FFN gradient updates derive from inconsistent geometries. Per `CLAUDE.md` ("there should ALWAYS exist a theoretically/mathematically pure path under appropriate toggles"), the codebase needed an opt-in fully-consistent path. Rather than break backwards compatibility or force the cost on everyone, this phase introduces a tri-state flag.

A second, smaller issue surfaced in the same review: `transformer/core/vfe_gradients.py:1568-1576` used to *warn* when `use_rope=True` was passed without `irrep_dims` and continue with a known-biased softmax-coupling gradient (raw-μ instead of `R^T·∂KL_RoPE/∂(R μ)`). That is now a hard error.

### Edits

* `transformer/core/block_config.py`
  * Added `RopeFullGaugeMode = Literal['off', 'vfe_only', 'both']` and `_coerce_rope_full_gauge(value) -> RopeFullGaugeMode`. Coercion: `True → 'vfe_only'`, `False → 'off'`, `None → 'off'`, valid string passes through, invalid string raises `ValueError`, invalid type raises `TypeError`.
  * Field annotation widened to `Union[bool, RopeFullGaugeMode]` with default `'off'`. The `__post_init__` calls the coercer so the canonical string always lands on the dataclass.
  * `BlockConfig.from_config` runs the value from `config.get('rope_full_gauge', 'off')` through the coercer too. The diagonal/isotropic-covariance warning fires for any mode `!= 'off'` (was: `if self.rope_full_gauge`).
* `transformer/core/variational_ffn.py`
  * `self._rope_full_gauge_vfe` initialized to the string `'off'` (was: bool `False`). The dispatch at the helper firing point checks `getattr(self, '_rope_full_gauge_vfe', 'off') in ('vfe_only', 'both')`. Backwards compatible because the upstream coercer maps legacy `True` → `'vfe_only'`.
* `transformer/core/blocks.py`
  * Block init now reads `_rope_mode = getattr(cfg, 'rope_full_gauge', 'off')` once and forwards to BOTH sublayers: `self.ffn._rope_full_gauge_vfe = _rope_mode` (FFN side, fires for `'vfe_only'` or `'both'`) and `self.attention.rope_full_gauge_mode = _rope_mode` (attention side, fires for `'both'`).
* `transformer/core/attention.py`
  * Imported `_apply_rope_to_covariance` from `transport_ops`.
  * Added `self.rope_full_gauge_mode = 'off'` to `IrrepMultiHeadAttention.__init__` with a docstring explaining the tri-state semantics and the diagonal-covariance constraint.
  * In the per-head forward path, after the existing `mu_q_for_attn = _apply_rope(mu_q, base=self.rope_base)` call, added a new branch:
    * If `use_rope and rope_full_gauge_mode == 'both'`: validate `not self.diagonal_covariance` (raise `ValueError` with a clear message otherwise), lift σ via `torch.diag_embed` if it arrived as `(B, N, K)`, then apply `_apply_rope_to_covariance(sigma_q, base=self.rope_base)` to get the rotated full-cov σ. The rotated σ is used for `_split_irreps_sigma`; the raw σ remains untouched for any value-side aggregation.
* `transformer/core/vfe_gradients.py`
  * The `if use_rope:` warning at line 1568-1576 (no `irrep_dims` chain-rule fix) is replaced with `raise ValueError(...)` explaining why the path is unsupported and pointing to the two recoveries (`irrep_dims=...` or `use_rope=False`). Callers can no longer silently train on biased descent directions.

### Out of scope (deliberate)

* The `'both'` σ rotation for attention applies the same `_apply_rope_to_covariance` machinery the FFN helper already uses. The implementation requires `diagonal_covariance=False` (the rotated covariance has off-diagonal terms by construction); the diagonal case has no well-defined `R Σ Rᵀ` representation and is rejected with a clear error rather than silently approximated.
* No β-recomputation via autograd in attention (as the FFN helper does). The σ rotation flows through the existing per-head KL machinery, which already handles full-cov σ. Adding a separate autograd-based attention β path would duplicate the FFN helper unnecessarily; if a user needs strict gradient-level consistency they can also enable `evolve_sigma=True` so the FFN helper's autograd dominates.
* No rename of `_rope_full_gauge_vfe` (state-dict naming is irrelevant since it's not a parameter, but downstream code reads the attribute by this exact name across `blocks.py` and the FFN dispatch — keep it stable).
