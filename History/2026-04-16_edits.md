## Phase E — Determinant control for GL(K)

Diagnostic context: user reported `det(Ω): mean=15.6, median=1.0, std=133, min=0.0, max=26923` on `K=90`, `gauge_group=GLK`, 6 × GL(15) blocks, `phi_max_norm=15`, `em_mode=em_phi_p`. Median ≈ 1 means most tokens are well-behaved; one token has drifted in the trace direction. Root cause: GL(K) has a 1D unbounded determinant subgroup (`v_a = tr(G_a)`) that L2-norm clamping does not constrain. With `K=15` per block and `‖φ‖ ≤ 15`, theoretical `|tr(φ·G)| ≤ √15·15 ≈ 58`, hence `det(Ω) ∈ [exp(-116), exp(116)]`.

Two new opt-in toggles added to `BlockConfig` and `VFEConfig` (defaults preserve existing behavior):

- **`phi_project_slk: bool = False`** — hard project φ → sl(K) (traceless subalgebra). det(Ω) ≡ 1 by construction. Drops one degree of freedom per block (the determinant, which the loss never used to do useful work).
- **`phi_trace_clamp: Optional[float] = None`** — soft cap `|tr(φ·G)| ≤ T` per token. det(Ω_ij) ∈ [exp(-2T), exp(2T)]. e.g., `T=0.35` → det ∈ [0.5, 2].

If both set, `phi_project_slk` wins.

**Implementation**:
- `transformer/core/vfe_utils.py:_retract_phi` — added `project_slk`, `trace_clamp` params; new helper `_apply_det_control` runs after every retraction (no-op when `is_glk=False` since SO(N) has tr(G)=0 already). All 4 return points unified.
- `transformer/core/vfe_utils.py:_apply_det_control` — new function. Computes `v_a = tr(G_a)` and either projects out the trace direction or rescales it to fit the clamp.
- `transformer/core/variational_ffn.py:VariationalFFN.__init__` — accepts and stores both toggles.
- `transformer/core/blocks.py:~297` — passes `cfg.phi_project_slk` and `cfg.phi_trace_clamp` to FFN.
- `transformer/core/vfe_closed_form.py:~602` and `transformer/core/variational_ffn.py:~2210, ~2552` — pass the toggles into `_retract_phi`.
- `transformer/vfe/e_step.py:~98, ~431` — VFE E-step path reads cfg toggles and forwards to `_retract_phi`.
- `transformer/vfe/prior_bank.py:PriorBank.__init__` — caches trace vector `v_a = tr(G_a)` as a buffer; `encode()` and the cached-decode path apply `_apply_phi_det_control` post-`phi_embed` lookup. **This is the load-bearing fix for `em_phi_p` mode**: when φ is updated only by the M-step backprop optimizer (not by `_retract_phi`), the parameter can drift outside sl(K), but the model never sees the trace component because every read of `phi_embed.weight` is projected on the way out.
- `transformer/core/block_config.py` and `transformer/vfe/config.py` — new fields with documentation.
- `transformer/train_publication.py` and `transformer/vfe/train_vfe.py` — config dicts now expose both toggles (default off).

For the user's specific symptom (one outlier token, `em_phi_p`, GL(K)): set `'phi_project_slk': True` in the active config dict. det(Ω) should snap to ≈1.0 across all tokens within one forward pass.

## Phase F — Ultrareview follow-ups

A `/ultrareview` pass after Phase E surfaced three additional issues. All verified by direct read and fixed.

**Bug 002 (CRITICAL — introduced in Phase E)** — `_apply_det_control` and `_apply_phi_det_control` were projecting only the **single summed trace direction** `v = Σ_h v_h`, not per-block. For multi-block gauge groups (the user's config has 6 × GL(15) blocks; `train_publication.py:713` uses 4 × GL(10); `train_vfe.py` uses 2 × GL(10)) the SL(K) projection was geometrically incomplete: compensating signs across blocks could drive per-block determinants to `exp(±50+)` while satisfying the single summed constraint `Σ_h tr(φ·G_h) = 0`. Empirical verification with deliberate compensating-sign injection on a 2 × GL(6) bank confirmed `det(Ω_h)` blew up to `[1.6e-24, 6.4e23]` per block under the old single-direction projection.
- **Fix**: build `V_blocks ∈ R^{H × n_gen}` where `V_blocks[h, a] = tr(G_a |_block_h)`. Project per-block: `phi - einsum('...h,hg->...g', (phi @ V.T) / ||v_h||², V)`. Block-diagonal generators have disjoint support per block so the per-block trace vectors are mutually orthogonal — single matmul handles all blocks.
- **Files**: `transformer/core/vfe_utils.py:_apply_det_control` (added `irrep_dims` param), `transformer/vfe/prior_bank.py:_apply_phi_det_control` (uses cached per-block `_phi_trace_vec` of shape `(H, n_gen)`).
- **Wiring**: added `irrep_dims=...` kwarg to `_retract_phi` and threaded through all 4 callers (`vfe/e_step.py:431`, `core/variational_ffn.py:2210, 2552`, `core/vfe_closed_form.py:602`).
- **Verification**: post-fix, the same compensating-sign attack on 2 × GL(6) gives `det(Ω_h) = 1.000` exactly per block with `phi_project_slk=True` and `det(Ω_h) ∈ [0.497, 2.014]` per block with `phi_trace_clamp=0.35`.

**Bug 003 (nit)** — `transformer/vfe/prior_bank.py:__init__` called `generators.diagonal(dim1=-2, dim2=-1)` BEFORE the numpy→torch conversion guard. PyTorch-only kwargs `dim1`/`dim2` would raise `TypeError` on a numpy `generators` input under either det-control toggle. No production caller currently triggers it (`VFEModel` pre-converts), but the ordering was inconsistent with the guard's intent.
- **Fix**: moved the `isinstance(generators, torch.Tensor)` conversion block ABOVE the buffer/det-control setup so all subsequent code can rely on tensor methods.

**Bug 008** — `transformer/vfe/prior_bank.py:84-86` skew-symmetric detection used the same single-generator check (`torch.allclose(generators[0], -generators[0].T)`) that Phase B fixed in `vfe/attention.py:41`. For mixed generator banks where `generators[0]` happens to be skew, the bank was wrongly classified as `_generators_are_skew=True` and `fused_block_matrix_exp_pairs` silently used the SO(N)-optimized exp path on non-skew generators, corrupting the prior bank's transports.
- **Fix**: replaced with `torch.allclose(generators, -generators.transpose(-1, -2), atol=1e-6)` across all generators, matching the attention.py fix.

### Change Set B — α-divergence × learnable-α schedule consistency

Pre-change inconsistency: `VariationalFFNDynamic.get_bayesian_alpha` computed `α_k = c₀_k / (b₀_k + KL_k)` using KL regardless of `alpha_divergence`. When `alpha_divergence != 1.0` the self-coupling gradient used Rényi `D_α`, and the product-rule correction `∂(α·div)/∂θ = α·∂div/∂θ + (∂α/∂θ)·div` was explicitly disabled in three Rényi branches with the comment *"Product-rule correction is not applied for alpha_div != 1.0 because the KL used to define the learnable alpha schedule is not D_alpha"*. This left the user's production config (`alpha_divergence=0.2, E_learnable_alpha=True`) with a schedule that responded to one divergence and a gradient that computed another.

Chose the pure fix: redefine the schedule so the Rényi branch uses `D_α,k` per-dim, matching the gradient. Product-rule correction re-enabled because `α_k = c₀_k / (b₀_k + D_α,k)` yields `∂α_k/∂θ = -α_k²/c₀ · ∂D_α,k/∂θ` identically to the KL case.

**Changes:**
1. **`transformer/core/variational_ffn.py:get_bayesian_alpha`** — rewrote the body to branch on `self.alpha_divergence`. KL branch unchanged. Rényi branch computes per-dim `D_α,k = (1/2)[α·δμ_k²/σ̃_k + ((1-α)log σ_q,k + α log σ_p,k - log σ̃_k)/(α-1)]` with `σ̃_k = (1-α)σ_q,k + α σ_p,k`. Full-cov uses the diagonal-of-full proxy consistent with the existing KL-branch heuristic.
2. **`transformer/core/vfe_gradients.py`** — re-enabled the product-rule correction in three Rényi branches (full-cov self-coupling, diagonal self-coupling, fused-path diagonal). ### CS-4 — `rope_full_gauge` diagonal/isotropic guard

Added `__post_init__` warning in `block_config.py` (near line 205) when `rope_full_gauge=True` meets `diagonal_covariance=True` or `isotropic_covariance=True`. Diagnostic explains the two wasteful regimes:
- **Isotropic**: `R·(σ²I)·R^T = σ²I` is invariant under rotation — `rope_full_gauge=True` pays `O(K²)` cost for a result identical to `rope_full_gauge=False`.
- **Diagonal non-isotropic**: lifts `σ` to full `(B, N, K, K)` — genuine gauge-consistent transport but expensive. Default path (`rope_full_gauge=False`) is the deliberate design choice.

Soft warning, not `ValueError` — the legacy path at `vfe_gradients.py:_compute_rope_full_gauge_gradient_per_head` lifts diagonal `σ` internally, so it's not a correctness bug. `vfe/` remains stricter (`ValueError`).

### CS-5 — RAdamW pullback metric option (pure path, opt-in)

**Motivation.** CLAUDE.md: *"There should ALWAYS exist a theoretically/mathematically pure path under appropriate toggles."* The Cartan-modified Killing form is a position-independent approximation; the mathematically exact gauge-natural metric is the pullback of the Frobenius inner product on GL(K) through the exponential map, `G_{ab}(φ) = ⟨Ψ(ad_φ) T_a, Ψ(ad_φ) T_b⟩_gram`. The E-step already supports `phi_natural_gradient='pullback'`; the M-step optimizer was Killing-only. Added a matching opt-in.

**Changes:**

1. **`gauge_preconditioner.py`** — refactored `apply_pullback_natural_gradient` to call a new public helper `build_pullback_metric_tensor(phi, generators, structure_constants, gram, series_order)`. Same math as before; now reusable by the optimizer's clipping path. At `φ=0`: `G = gram` (Frobenius). Regularized with `+ 1e-6·I` for numerical stability.

2. **`training/optimizer.py:RiemannianAdamW`** — added `metric: str = 'killing'` parameter plus `generators`, `structure_constants`, `gram`, `pullback_series_order`. Construction auto-builds `structure_constants` and `gram` from `generators` when `metric='pullback'`. Raises `ValueError` on missing generators or invalid metric string.

3. **`_precondition_phi` branches** on `self._metric`:
   - `'killing'` (default): unchanged — `grad ← grad @ killing_inv`.
   - `'pullback'`: `grad ← apply_pullback_natural_gradient(grad, p.data, ...)` per param. Per-token `(n_gen, n_gen)` solve.

4. **`_riemannian_clip`** branches on `self._metric` for phi/omega groups:
   - `'killing'`: unchanged — group-level `‖ξ‖_K = sqrt(Σ_v ξᵀ K ξ)` with fixed Killing metric.
   - `'pullback'`: **per-token** clipping — builds `G(φ_v)` per token via `build_pullback_metric_tensor`, computes `‖ξ_v‖_{G(φ_v)} = sqrt(ξ_vᵀ G(φ_v) ξ_v)` per row, scales each row independently to the trust radius `δ`. Genuinely gauge-natural clip that matches the gauge-natural precondition. Pays `O(V · n_gen² · series_order)` per step.

5. **`create_optimizer`** reads `phi_optimizer_metric` and `pullback_series_order` from the config; forwards generators/structure/gram to `RiemannianAdamW` on the pullback path; logs `"RAdamW metric: 'pullback' (per-token (n_gen,n_gen) build + solve; gauge-natural, expensive)"` when active.

6. **`training/config.py`** — two new fields: `phi_optimizer_metric: str = 'killing'` and `pullback_series_order: int = 6`. Plumbed through `experiment_runner.py:TrainingConfig(...)` construction with `config.get` fallbacks.
