## scripts/run_ablation_suite.py — add `e_step_sigma_floor` sweep

Added a new single-parameter sweep to `SWEEPS` with explicit values
`[0.01, 0.05, 0.1, 0.25]` and baseline `0.01`. Motivation: 0.01 caps
E-step precision at 100, which can drive overconfidence in the q
posterior; the right value is coupled to K, data statistics, and the

## Add `ce_label_smoothing` config flag (legacy core path only)

Adds optional label smoothing to the observation-likelihood term
`-E_q[log p(o|x)]` in the free-energy loss. Default `0.0` keeps existing
runs byte-identical. Only the legacy core path
(`transformer/train.py::compute_free_energy_loss`) is plumbed; the VFE
package (`transformer/vfe/model.py:147`) and the hand-rolled pure_vfe
analytic CE (`transformer/pure_vfe/learning.py:201,205,652`) are
deliberately out of scope per scoping decision recorded in
`C:\Users\chris and christine\.claude\plans\ultrathink-and-deploy-agents-twinkling-newt.md`.

### PPL is preserved un-smoothed

Inside `compute_free_energy_loss` (`transformer/train.py:363-381`),
`ce_loss_raw` continues to be the un-smoothed
`F.cross_entropy(logits, targets)` exactly as before. When
`ce_label_smoothing > 0.0`, a *second* `F.cross_entropy` call is made
with `label_smoothing=ε` and stored as `ce_loss` for the loss path. When
`ce_label_smoothing == 0.0` the second call is skipped and
`ce_loss = ce_loss_raw`, giving a zero-overhead bit-identical fast path
that does not perturb the autograd graph. `metrics['loss/ce_raw']`,
PPL = `exp(loss/ce_raw)`, and BPC reporting therefore remain directly
comparable to all prior runs even with smoothing on.

### Why VFE-side smoothing was rejected

VFE's `αKL(q||p)` and `λ_β · Σβ_ij KL(q_i || Ω_ij q_j)` already provide
entropy regularization on the *belief* side. Smoothing the
observation-side CE on top is somewhat redundant in the gauge framework,
but empirically often still helps — the user accepted the redundancy
caveat and chose to enable it as an opt-in sweep parameter rather than
hardcode.

### `sigma_ce_scale` interaction

`sigma_ce_scale` is implemented at `transformer/core/prior_bank.py:816-817`
and `:921-932` as `sigma_p_safe = sigma_p_clamped.detach() + _s *
(sigma_p_clamped - sigma_p_clamped.detach())`. This is a gradient
*magnitude* scaling that is agnostic to the target distribution shape;
label smoothing changes the gradient `softmax(logits) - smoothed_target`
only in its values, which still flow through the same scaled channel
without any plumbing change. No special handling required.

## Follow-up fix: dict→TrainingConfig boundary was dropping the new flag

Ultrareview (`bug_001`) caught a real silent no-op in the original
`ce_label_smoothing` plumbing. `run_single_experiment`
(`transformer/training/experiment_runner.py:1935`) and `resume_training`
(`transformer/resume_training.py:457`) build `TrainingConfig(...)` from
explicit named kwargs. `ce_label_smoothing` was not in either kwargs
list, so `cfg['ce_label_smoothing']` was silently dropped and the
dataclass default `0.0` was assigned regardless of input. The downstream
`getattr(self.config, 'ce_label_smoothing', 0.0)` reads at
`experiment_runner.py:751` and `train_fast.py:259` therefore always saw
`0.0` — meaning setting `'ce_label_smoothing': 0.1` in `EM_CONFIG` did
nothing, and uncommenting `'ce_label_smoothing'` in `SWEEP_ORDER` would
have produced four numerically identical runs across `[0.0, 0.05, 0.1,
0.2]`.

The same structural omission existed (latently) for `normalize_ce_by_dim`
in both files and for `detach_beta_m_step` in `experiment_runner.py`.
These were latent because the dataclass defaults (`True`, `True`) happen
to match the active config dict values, but any sweep that varied them
would silently no-op for the same reason.

### Fix

`transformer/training/experiment_runner.py:2010-2016` — added a "CE loss
handling" block right after `sigma_ce_scale` that threads all three keys
through:

```python
        normalize_ce_by_dim=config.get('normalize_ce_by_dim', True),
        ce_label_smoothing=config.get('ce_label_smoothing', 0.0),
        detach_beta_m_step=config.get('detach_beta_m_step', True),
```

`transformer/resume_training.py:497-498` — added the two missing keys
next to the existing `detach_beta_m_step` line:

```python
        normalize_ce_by_dim=config.get('normalize_ce_by_dim', True),
        ce_label_smoothing=config.get('ce_label_smoothing', 0.0),
```

### Verification

Smoke test confirmed `TrainingConfig` now accepts and round-trips
`ce_label_smoothing=0.1`, `normalize_ce_by_dim=False`, and
`detach_beta_m_step=False` through the constructor. With the fix in
place, `'ce_label_smoothing': 0.1` in `EM_CONFIG` and the
`SWEEPS['ce_label_smoothing']` ablation now reach
`compute_free_energy_loss` at runtime.
