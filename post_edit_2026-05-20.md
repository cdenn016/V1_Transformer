### P2 — block-diagonal full-cov decode (investigation only, no code change)

Investigation written to `docs/debates/2026-05-20-vfe-use-prior-bank-decoder/06_P2_investigation.md`. Findings:

- The diagonal projection at `prior_bank.py:494` discards information **only** in the combination `gauge_fixed_priors=True` AND `diagonal_covariance=False`. The other three mode combinations are structurally exact: in `diagonal_covariance=True` paths the `torch.diagonal(...)` call is a no-op on a `(K,)` tensor; in the `(gauge_fixed_priors=False, diagonal_covariance=False)` combination the per-token σ is diagonal-embedded into a `(K,K)` tensor at `_diag_to_full_block`, so all off-diagonal entries are zero by construction.
- In the affected combination, sigma_p is **block-diagonal** with dense within-block content `A_h diag(s_h) A_h^T`. The exact full-cov KL decomposes additively across blocks, with per-block cost `O(d_h³)` (block inverse + log-det) plus `O(d_h²)` per position for the trace and quadratic. For the user's active scale (K=100, 5 blocks of d_h=20, V=10000, B=16, N=128), exact-block decode is ~20× the diagonal cost: ~4·10¹⁰ FLOPs/forward vs ~2·10⁹.
- Approximation error magnitude: roughly `-0.5·d_h·log(1-ρ²)` per block where ρ is the within-block correlation. At `phi_scale=0.001` (active init), within-block correlation is small and the approximation is tight; the gap grows as φ_v moves away from zero during training.
- Recommendation: add an opt-in `exact_full_cov_decode` flag to `VFEConfig`, default `False`, gated to `(gauge_fixed_priors=True, diagonal_covariance=False)`. Per the CLAUDE.md "pure path under appropriate toggles" doctrine. Not implemented in this session; user opted for investigation only.

The investigation also catalogued super-block sizing under `cross_couplings` (the `d_h³` cost is paid in the largest super-block, not the average) as a separate cost note for any future implementation.

## Session — pullback metric code repair in `transformer/core/gauge_preconditioner.py`

### Trigger

User request after the §C debate verdict and the codebase audit (which found the pullback metric bug propagated into `gauge_preconditioner.py:552-554` but the preconditioner was explicitly disabled at the config level):

> fix the 'pullback' code

### Process

The §C debate verdict's optional code-mode follow-up identified the `build_pullback_metric_tensor` function as containing the same wrong formula as supplementary line 606 (now corrected in commit `e4481f7c`). The function was dormant (the `pullback` preconditioner is gated by `vfe/config.py:591-600` raising `ValueError`), but a separate code path through `transformer/training/optimizer.py::RiemannianAdamW(metric='pullback')` consumes the same `build_pullback_metric_tensor`, so the bug was latent in the outer optimizer.

### Investigation

Tracing the chain `psi[a,c]` → "c-th coordinate of `Ψ(ad_φ)(T_a)`" by verifying with a 2x2 explicit case (phi = c·T_2 in a basis with `[T_1, T_2] = T_2`):
- `ad_φ_std = [[0, 0], [-c, 0]]` (standard operator matrix, rows=output).
- Code's `ad_X = [[0, -c], [0, 0]]` (matches `(ad_φ_std)^T`).
- Therefore `psi = Ψ(ad_X) = Ψ(ad_φ_std)^T`, giving `psi[a, c] = (Ψ(ad_φ))^c_a`.

The legacy code `G = psi^T @ gram @ psi` was the WRONG matrix product ordering for this convention; it should have been `psi @ H @ psi^T` where `H[c,d] = tr(T_c^T · T_d · M)` with `M = exp(φ) exp(φ)^T`. The legacy tests passed despite this because (a) at φ=0 both orderings reduce to gram, and (b) no test verified against the canonical Frobenius pullback at non-zero φ.

### Fixes applied to `transformer/core/gauge_preconditioner.py`

1. **`apply_pullback_natural_gradient` docstring (line 541+)**: replaced the wrong dexp form with the canonical right-trivialized `D_φ exp[T_a] = Ψ(ad_φ)(T_a) · exp(φ)` per [Hall 2015 §2.7 / Theorem 5.4], and added the correct pullback metric `G_ab(φ) = tr(Ψ(ad_φ)(T_a)^T · Ψ(ad_φ)(T_b) · exp(φ) exp(φ)^T)`.

2. **`build_pullback_metric_tensor` (line 586+)**: implemented the corrected metric. Three changes:
   - Compute `phi_mat = Σ_a φ^a T_a` and `M = exp(phi_mat) @ exp(phi_mat)^T` in matrix space (per token).
   - Build φ-dependent Gram-like tensor `H_{cd}(φ) = tr(T_c^T · T_d · M) = einsum('cil,dij,...jl->...cd', generators, generators, M)`.
   - Use the correct matrix-product ordering `G = psi @ H @ psi^T` (replacing the legacy `psi^T @ gram @ psi`).
   - Documented the convention `psi[a, c]` = c-th coord of `Ψ(ad_φ)(T_a)` inline so future readers don't fall into the same trap.

3. **Module-level docstring (lines 34-46)**: updated to reference the corrected metric and the operator identity `Ψ_R(z) = e^z Ψ_L(z)`.
