# Edits — 2026-04-08

## Claude Code settings.json + hook scripts repair

Diagnosed and fixed eight defects in the project's Claude Code configuration
and supporting hook scripts. The net effect is that the two intended
constraint-enforcement hooks (no-nn and pre-commit pytest) now actually run,
where previously both were silent no-ops.

### Files touched

- `.claude/settings.json` — rewritten
- `scripts/hooks/check_no_nn.py` — stdin ingestion fix + file-type guard
- `scripts/hooks/test_on_commit.py` — new file

### Defects in `.claude/settings.json`

1. **Trailing comma after the `hooks` object** — invalid JSON. A malformed
   settings.json silently disables every permission and hook in the file,
   so nothing was active prior to this edit.
2. **Bash permission rules used the wrong wildcard form.** Entries like
   `"Bash(python *)"`, `"Bash(git status*)"`, etc. Corrected to the
   documented `prefix:*` form: `"Bash(python:*)"`, `"Bash(git status:*)"`,
   and so on. `"Bash(find:*)"` was already correct.
3. **`matcher: "Write"` missed `Edit`.** Most in-place changes route
   through `Edit`, not `Write`, so the no-nn gate never saw incremental
   modifications. Changed to `"Write|Edit"`.
4. **Hooks were wired as `PostToolUse`.** Both were enforcement gates
   (block bad writes; block bad commits), but `PostToolUse` fires after
   the tool has already succeeded, so neither could block anything.
   Moved both under `PreToolUse`. This matches the docstring intent of
   `check_no_nn.py` ("Pre-tool hook") which had been mis-wired since
   creation.
5. **`$TOOL_INPUT` env var does not exist.** The original inline Bash
   hook did `echo "$TOOL_INPUT" | python -c "..."`. Claude Code hooks
   receive their payload on **stdin as JSON**, not via environment
   variables, so the variable expanded to the empty string, the Python
   parse raised, `2>/dev/null` swallowed it, the `if` was always false,
   and pytest **never ran**.
6. **Even with stdin, the Python dereferenced the wrong level.** The
   hook payload is
   `{"tool_name": ..., "tool_input": {"command": ...}}`; the original
   code did `d.get('command', '')` on the root, which would always be
   empty.
7. **Substring match on the commit needle self-triggered.** My first
   fix was an inline `python -c` one-liner. Because that one-liner's
   source contained the literal commit-detection string, and because
   Claude Code passes the caller's full bash string as
   `tool_input.command`, the detector matched its own source and
   blocked every bash call. Resolved by promoting the logic to a
   dedicated script (`scripts/hooks/test_on_commit.py`) that uses
   `shlex` tokenization and walks command boundaries so quoted
   occurrences never false-positive.

### Defects in `scripts/hooks/check_no_nn.py`

The script was self-described as a pre-tool hook and claimed to read
`TOOL_INPUT` from the environment. That environment variable does not
exist — the Claude Code hook contract is JSON on stdin. The script
therefore always took the `if not tool_input_raw: return 0` branch and
silently allowed every write. Fixed by reading
`json.load(sys.stdin)` and extracting `tool_input` from the hook
payload. Removed the now-unused `import os`. Docstring updated to
describe the real interface and the exit-2 blocking convention.

8. **No file-type guard.** The checker ran its regex against any file
   path, so any Markdown or config file that *discussed* the forbidden
   symbols (e.g. this edit log, or `CLAUDE.md` itself when written by
   the hook) would be blocked. Added an early return for paths that
   do not end in `.py`.

### New file: `scripts/hooks/test_on_commit.py`

Pre-tool hook for `Bash`. Parses the candidate bash command with
`shlex.split`, walks tokens while respecting shell command separators
(`;`, `&&`, `||`, `|`, `&`), and returns true only when the commit
verb appears as an actual command head — not inside a quoted argument
and not as a substring of some other token. If so, runs
`python -m pytest -x --tb=short -q` and maps a non-zero pytest exit to
exit code 2, which Claude Code treats as a blocking denial on
`PreToolUse`. Otherwise exits 0.

### Validation performed

- `json.load` on the new `settings.json` succeeds; structure matches
  the documented hooks schema.
- `check_no_nn.py` gate matrix:
  - Python file containing `class X(nn.Module)` → exit 2 (blocked).
  - Python file with clean gauge-covariant content → exit 0.
  - Markdown file mentioning the banned symbols in prose → exit 0.
  - File under `tests/` containing the banned pattern → exit 0 (exempt).
- `test_on_commit.py` detector matrix:
  - `ls -la` → allow.
  - `echo "git commit hello"` (quoted substring) → allow.
  - `git commit -m hi` → detected.
  - `git add . && git commit -m hi` → detected.
  - `echo "git commit"` → allow.
  - `git status` → allow.
- Hot-reload confirmed: after the final edit, subsequent bash tool
  calls in this session ran without the self-trigger error that the
  broken inline one-liner produced, and this very edit log landed
  through the live `Write|Edit` hook.

### Why the hooks had never enforced anything before

Each layer was independently broken. The JSON was unparseable so no
hook ran at all; the matcher missed `Edit`; the event was wrong for
blocking; the inline Python read a nonexistent env var and accessed
the wrong dict level; and the sidecar script also read the wrong env
var. Any one of these alone would have silenced the hook. All five
together meant the "no neural networks" constraint and the "tests
gate commits" policy were both uniformly unenforced despite being
present in config.

## Deep Audit Implementation — REVERTED

A three-round deep audit (Rounds 1/2/3) was run earlier in this session and
a substantial set of code changes was applied across PR-1 (constraint
compliance), PR-2 (performance), PR-3 (GL(K) safety warnings), PR-4 (research
quality), and PR-5 (numerical stability + Cholesky substitution). User
training runs with `use_prior_bank=True, diagonal_covariance=True` showed
catastrophic divergence (loss 8→21, PPL→4.85e8 by step 600), and the
`use_prior_bank=False` path also trained worse than the pre-audit baseline.

Root cause of the blow-up: PR-2.2 replaced conditional `if isnan().any()`
NaN guards in the fused attention + gradient hot path
(`_fused_attention_and_vfe_gradients_block_diag`) with unconditional
`torch.nan_to_num(sigma_j_transported, nan=1.0)`. The default
`posinf=None`/`neginf=None` arguments on `torch.nan_to_num` silently
replace `±inf` with the dtype's max/min finite value (`±3.4e38` for
float32). Any `+inf` in `sigma_j_transported` — which can arise from
ill-conditioned `Omega^2 * sigma` under GL(K) transport with large phi —
then produced `log(3.4e38) ≈ 88` in the KL, corrupting the softmax and
exploding gradients. Rule: **avoid `torch.nan_to_num` in the VFE hot path**;
if unavoidable, always pass `posinf=float('inf'), neginf=float('-inf')`
explicitly.

At user request, **all audit changes were fully reverted** to restore the

## amortized_inference=True audit + Cholesky numerical-stability refactor

Batch of six fixes applied in response to a deep audit of the codebase under
`amortized_inference=True`, with Cholesky decomposition used wherever
possible for numerical stability. The Cholesky factor `L` itself is NOT
gauge covariant (the sandwich product `Ω @ Σ @ Ω.T` does not commute with
Cholesky), but `Σ^{-1} = L^{-T} L^{-1}` IS gauge covariant as a precision,
so the final result of every refactored inversion remains gauge-invariant
identically to the old `torch.linalg.inv` path. `L` is never stored,
transported, or compared across frames — only the final inverse is returned.

### C6: `transformer/core/vfe_utils.py` `_safe_spd_inv` (line 142)

Replaced `torch.linalg.inv(M_reg)` with a Cholesky primary path
(`torch.linalg.cholesky_ex` + `torch.cholesky_inverse`), with escalating
jitter (3 rounds of 10×) on Cholesky failure, falling back to `pinv` as
last resort. Added explicit symmetrization `0.5 * (M + M.T)` before
Cholesky to handle roundoff asymmetry in sandwich-product inputs. This
primitive is the hub for every SPD inversion in the codebase — the change
propagates to 10+ call sites in `vfe_gradients.py` and `variational_ffn.py`
without touching them.

### C5: `transformer/core/attention.py` precision aggregation (lines 1088, 1097, 1195, 1201)

Replaced four raw `torch.linalg.inv(…) / torch.linalg.pinv(…)` fallback
chains in the precision-aggregation paths (both tiled-float32 path and
legacy path) with single calls to `_safe_spd_inv`. Added `from
transformer.core.vfe_utils import _safe_spd_inv` to the imports.  Every
input is a symmetrized, eps-regularized sandwich product or a precision
sum, all SPD by construction. The aggregation is gauge-covariant as a
precision: under a global gauge `g`, `Σ → g Σ g.T` implies `Σ^{-1} →
g^{-T} Σ^{-1} g^{-1}`, `β` is gauge-invariant (derived from KL), and the
aggregated precision transforms identically.

Note on an earlier audit finding: an agent initially flagged this
precision aggregation as gauge-breaking because "the inverse of the
sandwich is not the sandwich of the inverse". This claim is false —
`(Ω Σ Ω.T)^{-1} = Ω^{-T} Σ^{-1} Ω^{-1}`, which is exactly the precision
transform, so aggregating `β_ij · inv(Ω_ij Σ_j Ω_ij.T)` IS gauge-covariant
as a precision. The agent was rejected and the code is mathematically
correct; this refactor is a pure numerical-stability upgrade.

### C4: `transformer/core/vfe_closed_form.py` Picard loop (line 484)

Replaced `torch.linalg.inv(Sigma_j_t)` inside `_picard_refine` with
`_safe_spd_inv`, and added a symmetrization line before the eps
regularization. The Picard refinement loop calls this inversion per
iteration, so roundoff amplification was a real concern. Added
`_safe_spd_inv` to the `from vfe_utils import` line.

### C3: `transformer/core/vfe_gradients.py` rope_full_gauge path (line 1860)

Replaced `torch.linalg.inv(sigma_t)` with `_safe_spd_inv(sigma_t, eps=eps)`
in `_compute_rope_full_gauge_gradient_per_head`. `_safe_spd_inv` was
already imported at line 48. This path is experimental
(`rope_full_gauge=True` is off by default) but the old raw inv had no
Cholesky fallback and could produce NaN on near-singular GL(K) transports.

### C2: `transformer/core/variational_ffn.py` `_detach_e_step` (line 2523)

Changed the hardcoded `_detach_e_step = True` to
`_detach_e_step = not self.amortized_inference`. The old hardcode broke
the iteration-to-iteration gradient coupling in amortized mode: gradients
still flowed to `mu_p` through the first-order analytical formula, but
higher-order couplings (where the previous iterate's `mu` carries a
dependency on `mu_p`) were truncated. With the fix, `amortized_inference=
True` now correctly preserves the full iteration gradient chain through
all `n_iterations` E-step steps. `sigma_p` remains detached unconditionally
elsewhere per the hard-constraint: "sigma_p is an M-step parameter — the
E-step reads it but must not write gradients to it."

### C1: `transformer/core/kl_computation.py` `_cholesky_with_fallback` (line 165)

The previous implementation, on total Cholesky failure after 5
regularization rounds, returned `chol(I + eps*I)` — silently producing
`KL ≈ 0` for the whole batch (trace ≈ K, Mahal ≈ 0, logdet ≈ 0) and
poisoning the M-step with a zero coupling signal. Fixed by raising
`RuntimeError` inside `_cholesky_with_fallback` and changing the outer
`try/except` at line 195 in `_kl_kernel_dense` to return a NaN-filled KL
tensor with a `RuntimeWarning` instead of re-raising. The NaN propagates
loudly to the training loss (triggering a NaN loss and early
termination), which is the correct behavior for truly pathological
inputs — much better than the silent zero-KL corruption.

## Round 2 deep audit — cross-layer and round-up fixes

Continuing the audit after round 1, dispatched five parallel Explore
agents to cover `variational_ffn._vfe_iteration`, `active_inference.py`,
`transport_ops.py` + `model.py` + `blocks.py` omega inversions,
`vfe_gradients._fused_attention_and_vfe_gradients_block_diag`, and the
multi-layer hierarchical cascade.  Verified every critical claim against
the source before committing fixes — three agent findings were wrong and
I rejected them (precision aggregation gauge covariance in round 1,
plus the phi ternary rewrite and the rope_full_gauge detach in round 2,
which were both semantically correct but visually confusing).

### R2-C1: `transformer/core/blocks.py:779` hierarchical_priors cascade detach

The multi-layer hierarchical cascade unconditionally detached
`mu_q` before assigning it to `mu_prior` for the next layer, with the
comment "preserves proper EM: the prior is an E-step constant". This
defeats `amortized_inference=True` by severing cross-layer gradient
flow — the whole point of amortized inference is gradient propagation
through the entire forward pass, including layer boundaries.  Fixed by
conditioning the detach on `self.amortized_inference`, mirroring the
pattern at `variational_ffn.py:1315–1318` (the mu_p branch inside a
single layer's E-step).  Added `self.amortized_inference` as a stack
attribute in `__init__`.  End-to-end validated with a 2-layer model:
all 20 trainable params receive nonzero gradients after
`loss.backward()`.

### R2-C2: `blocks.py:456` and `model.py:438` — omega_h_inv no-safety

Two sites cached `omega_h_inv = torch.linalg.inv(omega_h)` on GL+(K)
blocks without ridge regularization or fallback.  Unlike
`transport_ops.py:462` (which uses `_ridge=1e-6` + try/except + pinv),
these sites trusted the upstream pipeline.  A near-singular phi would
silently produce NaN, poisoning the entire forward pass.  Fixed by
adding ridge regularization (`_omega_ridge = 1e-6`), try/except, and
pinv fallback to match the transport_ops pattern.  Cholesky does not
apply here because omega is GL+, not SPD.

### R2-C3: `gauge_preconditioner.py:285` raw SPD inversion

The Killing/Cartan form natural gradient preconditioner computed
`inv_metric = torch.linalg.inv(metric)` on an SPD metric without
Cholesky or fallback.  Replaced with `_safe_spd_inv(metric, eps=1e-8)`.
The metric is constructed from outer products of phi gradients; SPD
by construction but numerically rank-deficient at init or after large
phi updates.  Added `from transformer.core.vfe_utils import _safe_spd_inv`
to the imports.

### R2-C4: `vfe_gradients.py:978, 1007` — hardcoded clamp(min=1e-4)

The fused block-diagonal kernel floored transported and source diagonal
variances at a hardcoded `1e-4`, inconsistent with the `eps` parameter
(typically `1e-6`) passed to the function and used by the self-coupling
path.  Biased small variances upward by up to two orders of magnitude in
normal training.  Fixed by computing `_sigma_floor = max(float(eps), 1e-7)`
once and using it at both clamp sites.  The `1e-7` lower bound protects
against the `1/sigma_j_transported` division while letting callers
tighten the floor down to sqrt(float32 eps).

### R2-C5: `vfe_gradients.py:1554` — missing sandwich symmetrization

The full-covariance belief alignment path built
`sigma_j_transported = Ω @ Σ_j @ Ω^T` via einsum and fed the result
directly to `_safe_spd_inv` and `torch.linalg.cholesky`, without
explicit symmetrization.  `_safe_spd_inv` now symmetrizes internally
(from R1-C6), so the `_safe_spd_inv` consumer is safe, but the
`torch.linalg.cholesky(sigma_j_reg)` at line 1585 still hit needless
`eigvalsh` fallback trips from O(eps_machine) float32 asymmetry.  Added
`sigma_j_transported = 0.5 * (sigma_j_transported + sigma_j_transported.transpose(-1, -2))`
after the einsum, matching the fused block-diag path convention.

### R2-C6: `vfe_gradients.py:1704` — natural gradient sigma symmetrization

The full-covariance natural gradient computed
`nat_grad_sigma = 2 * Σ @ ∇_Σ @ Σ` via einsum.  Mathematically
symmetric iff `∇_Σ` is symmetric (which holds analytically), but
vulnerable to roundoff drift.  Added explicit symmetrization before
return to guarantee the M-step sigma update keeps `Σ` on the SPD
manifold.

### R2-C7: `variational_ffn.py:2588–2590` — confusing phi detach ternary

The hoisted `_build_block_exp_pairs` call used
`phi_current if not (getattr(self, 'amortized_inference', False) is False
and self.detach_phi) else phi_current.detach()`.  Truth-table analysis
confirmed the logic was CORRECT (matches `line 1381–1384`: detach phi
iff `not amortized_inference AND detach_phi`), but the double negative
via `is False` comparison was visually confusing.  Rewrote as
`_amortized = getattr(self, 'amortized_inference', True)` /
`_honor_detach_phi = (not _amortized) and self.detach_phi` /
`_phi_for_cache = phi_current.detach() if _honor_detach_phi else phi_current`.
Pure readability fix; no behavior change.

### R2-C8: `variational_ffn.py.__init__` — amortized+implicit_em conflict warning

When both `amortized_inference=True` and `implicit_em=True`, the
`implicit_em` path at line 1362 unconditionally detaches `mu_current`
and `sigma_current`, silently overriding the attached `mu_p_current`
path and rendering `amortized_inference` a no-op.  Added a loud
`RuntimeWarning` in `__init__` so users who inadvertently set both
flags see the silent override.  Also logged via `logger.warning`.
Behavior unchanged for existing single-flag configs.

**71 test PPL on WikiText-103 K=90 GL(15) in one epoch**.  The
current codebase reaches only **83 PPL in 2 epochs** on the same
architecture.  User confirmed that rounds 1–2 audit fixes are NOT
responsible, and that config-parameter drift is irrelevant — only

architectural addition relative to the 71-PPL baseline:

- A/B-1: `learnable_head_kappa=False` (match OLD fixed scalar kappa)
- A/B-2: `optimizer_type='adamw'` (match OLD plain AdamW, no
  Riemannian preconditioning)
- A/B-3: `amortized_inference=False` (match OLD hardcoded
  `_detach_e_step=True` behavior)
- A/B-4: all three combined

All four failed to recover PPL.  This isolated the regression to
code *outside* any existing config flag.

### Archaeological diff (TransformerOld vs current)

With `TransformerOld/` restored in the working tree, direct file diff
surfaced the root cause: the 2026-04-07 deep-audit **Fix #1** + **Fix
#20** refactor of the mu sublayer residuals in
`GaugeTransformerBlock.forward`.  Both sublayers were changed from
plain additive to delta-subtraction form:

```python
# OLD (TransformerOld, 71-PPL baseline)
# model.py lines 725, 751, 892, 922 — forward_with_attention inline block
if block.use_residual:
    mu_q = mu_q + mu_attn          # attention sublayer — plain additive
    ...
    mu_q = mu_q + mu_ffn           # FFN sublayer — plain additive

# CURRENT (before this commit)
# blocks.py lines 547, 636 — GaugeTransformerBlock.forward
if self.use_residual:
    mu_q = mu_q + (mu_attn - mu_normalized)   # attention — delta extraction
    ...
    mu_q = mu_q + (mu_ffn - mu_normalized)    # FFN — delta extraction
```

The audit's own assessment, quoted from `docs/session_2026_04_07_
deep_audit_fixes.md:143-145`:

> "This is arguably the single most consequential fix of the entire
> audit because it actually affects training dynamics, whereas the
> round-1 fix only affected inference."

Echoed in `session_2026_04_07_deep_audit_round4.md:211-214`:

> "The most consequential fix was round 3 Fix #20 (attention residual
> bug live in the training forward path), which was silently
> degrading training dynamics at every layer..."

Fix #20 was the only audit change (a) explicitly identified as
affecting training dynamics, (b) applied to both mu sublayer residuals
in the training hot path, and (c) with the exact mechanism (changing
gradient flow through the residual stream) that would shift
one-epoch PPL by ~12 points.  The user's 71-PPL run predates the
fix.  None of the Phase-A A/B tests touched this code path, which is
why they all failed.

### Why the audit was wrong for this configuration

The audit's rationale (Fix #1 docstring, `session_2026_04_07_deep_
audit_fixes.md:10-23`) was that `mu_attn = Σ_j β_ij · Ω_ij ·
mu_normalized[j]` is an *aggregation* of the normalized input, not a
zero-centred correction; when self-attention dominates, `mu_attn[i]
≈ mu_normalized[i]`, so the plain residual dumps `mu_normalized`
into the residual stream each layer.  This argument is correct **in
the abstract** for deep unnormalised stacks with dominant
self-attention.

For the single-layer LayerNorm'd K=90 GL(15) config where the 71-PPL
result was achieved, the argument does not apply:

1. **Single layer**: `n_layers=1`, so there is no cross-layer
   compounding and no "explosion across layers" can manifest.
2. **LayerNorm is already applied**: `self.norm1(mu_q)` re-scales
   each sublayer input, mitigating any residual-stream growth the
   audit was worried about.
3. **Gradient-signal strength**: the delta form's output gradient is
   `I + (∂mu_sub/∂mu_q) − (∂mu_normalized/∂mu_q)`.  The LayerNorm
   Jacobian subtraction cancels part of the identity path from
   loss → mu_q.  The additive form has `I + (∂mu_sub/∂mu_q)` with no
   cancellation, preserving the full direct gradient signal to the
   embedding layer.

The fix is mathematically defensible but empirically harmful for the
user's actual architecture.

### Fix

Introduced a `residual_type` config flag with default `'additive'`
(matches the 71-PPL baseline).  The audit's delta-extraction form is
preserved as an opt-in via `residual_type='delta'` for configurations
where the audit's pathology actually applies (deep unnormalised
multi-layer stacks with dominant self-attention).

**`transformer/core/block_config.py`**:

1. Added `residual_type: str = 'additive'` to the `BlockConfig`
   dataclass alongside `use_residual`, with docstring pointing at
   the audit history and the 2026-04-08 flip.
2. Added `residual_type=config.get('residual_type', 'additive')` to
   `BlockConfig.from_config`.

**`transformer/core/blocks.py`**:

1. `GaugeTransformerBlock.__init__` now stores
   `self.residual_type = getattr(cfg, 'residual_type', 'additive')`
   right after `self.use_residual`.
2. Attention sublayer residual (formerly line 547) now branches on
   `self.residual_type`: the 'additive' branch computes
   `mu_q = mu_q + mu_attn` (the 71-PPL baseline form), and the
   'delta' branch computes `mu_q = mu_q + (mu_attn - mu_normalized)`
   (the audit Fix #1 form).
3. FFN sublayer residual (formerly line 636) receives the same
   branch.  Default `'additive'` matches OLD semantics at both
   sites; `'delta'` reproduces the audit fix.
4. The audit-era comment blocks at both residual sites were
   rewritten to (a) name the flag, (b) explain that the default is
   now additive as of 2026-04-08, and (c) preserve the audit
   rationale as the reason the 'delta' branch exists.

### What is NOT done in this change

- No regression test (user will verify PPL recovery independently)
- No end-to-end K=90 GL(15) training smoke
- No rollback of rounds 1–2 audit fixes
- No changes to sigma residuals (already replacement semantics in

## Round 4: Optimizer wiring bugs — `base_mu` routing + `_get_sigma` fallback

### Context

While searching for additional architectural causes of the 71→83
PPL regression beyond the Round-3 residual flag, the user traced the
parameter-group creation path and found that `base_mu` — the primary
prior mean parameter when `gauge_fixed_priors=True` — does not match
any of the optimizer's routing substrings.  Investigation confirmed
the bug and surfaced a compounding second bug in the RiemannianAdamW
mu preconditioner that disables Fisher preconditioning on the same
parameter even if routing were fixed.

Both bugs are specific to the baseline config
(`gauge_fixed_priors=True`, `use_prior_bank=False`,
`optimizer_type='riemannian_adam'`).

### Bug 1 — `token_embed.base_mu` routed to the `ffn` group

`GaugeTokenEmbedding.__init__` at `transformer/core/embeddings.py:227`
stores the prior mean as

```python
self.base_mu = nn.Parameter(torch.randn(embed_dim) * init_std)
```

so the full parameter name seen by `model.named_parameters()` is
`token_embed.base_mu`.

The mu-matching branch in `create_param_groups`
(`transformer/training/optimizer.py:509`, pre-fix) was

```python
elif 'mu_embed' in name or 'mu_prior' in name or 'prior_mu' in name:
    mu_params.append(param)
```

None of `mu_embed`, `mu_prior`, `prior_mu` are substrings of
`token_embed.base_mu`.  Every subsequent branch also fails (no
`sigma`/`log_sigma`/`omega`/`phi`/`pos_encoding`/`log_kappa`/
`attention`/`out_proj`/`norm`/`bias`/...), so the parameter falls
through to the final `else` and lands in the `ffn` group at
`M_vfe_hyperparam_lr` with `non_embed_weight_decay`.

Consequence: `M_mu_p_lr` and `embed_weight_decay` are silently
dropped for the baseline prior mean.  In the current ablation run
both LRs happen to equal `0.005` and both WDs equal `0.01`, so the
damage is silent *in this exact config* — but any sweep over
`M_mu_p_lr` or `embed_weight_decay` measures zero effect on
`base_mu`.

### Bug 2 — `_get_sigma()` cannot see `base_log_sigma_diag`

Even if Bug 1 were fixed, the mu-group Fisher preconditioner in
`RiemannianAdamW._precondition_mu` (`optimizer.py:128`) would still
be a no-op.  `_get_sigma()` (`optimizer.py:159`, pre-fix) was

```python
def _get_sigma(self) -> Optional[torch.Tensor]:
    embed = getattr(self._model, 'token_embed', None)
    if embed is None:
        return None
    if hasattr(embed, 'log_sigma_diag') and isinstance(embed.log_sigma_diag, nn.Parameter):
        return torch.exp(embed.log_sigma_diag.data).clamp(min=1e-6, max=10.0)
    return None
```

Under `gauge_fixed_priors=True` the log-variance attribute is named
`base_log_sigma_diag` (`embeddings.py:251`), not `log_sigma_diag`, so
`hasattr` fails and `_get_sigma()` returns `None`.  The mu
preconditioner then early-returns at `optimizer.py:139` and never
applies `∇_nat μ = Σ · ∇_E μ`.  The corresponding Riemannian
trust-region clip at `optimizer.py:213` (`elif 'mu' in name and
sigma is not None`) also degrades to the Euclidean `else` branch for
the same reason.

Compounded effect: on the baseline, `base_mu` receives neither the
mu LR nor the Fisher rescaling.  High-variance embedding dimensions
take steps roughly `σ²` smaller than the natural-gradient design
intent — a direction error, not just an LR scale.

### Verification that the sigma side is NOT broken

- `token_embed.base_log_sigma_diag` contains the `'log_sigma'`
  substring and is correctly routed to the `sigma_embed` group at
  `M_sigma_p_lr`.
- The `_precondition_sigma` factor of `2.0` at `optimizer.py:157` is
  mathematically correct for log-variance parameterization.  Traced
  `exp(base_log_sigma_diag)` into the diagonal sandwich product at
  `embeddings.py:434,466`, where it appears as `(A ** 2) @ σ` — i.e.
  treated as variance (diagonal of Σ), not standard deviation.  For
  log-variance `η = log(σ²)`, Fisher `F_ηη = 1/2`, so natural
  gradient `= 2 · ∇_E η`.  No bug.
- `token_embed.phi_embed.weight` → `phi_embed` group ✓.
- `pos_encoding.pos_phi` (if `mode='learned'`) → `phi_embed` group
  via the `'pos_encoding'` substring ✓.
- `prior_bank.base_prior_mu` / `base_log_prior_sigma` would route
  correctly via `'prior_mu'` / `'log_prior'`, but are irrelevant here
  (`use_prior_bank=False`).
- `sign_embed` group is created empty when `learnable_reflection
  =False` — dead code path, not a bug.

### Fix

**File**: `transformer/training/optimizer.py`

1. **Mu routing** (`optimizer.py:509`): extended the pattern match so
   `token_embed.base_mu` routes into `mu_params`:

   ```python
   elif (
       'mu_embed' in name
       or 'mu_prior' in name
       or 'prior_mu' in name
       or name.endswith('base_mu')
   ):
       mu_params.append(param)
   ```

   Used `name.endswith('base_mu')` rather than `'base_mu' in name`
   to anchor the match on the parameter tail and avoid accidental
   substring collisions with any future nested module path.

2. **`_get_sigma` fallback** (`optimizer.py:159`): added a second
   `hasattr` branch for the `gauge_fixed_priors=True` layout so the
   Fisher preconditioner can find the base prior variance:

   ```python
   if hasattr(embed, 'log_sigma_diag') and isinstance(embed.log_sigma_diag, nn.Parameter):
       return torch.exp(embed.log_sigma_diag.data).clamp(min=1e-6, max=10.0)
   if hasattr(embed, 'base_log_sigma_diag') and isinstance(embed.base_log_sigma_diag, nn.Parameter):
       return torch.exp(embed.base_log_sigma_diag.data).clamp(min=1e-6, max=10.0)
   ```

   Shapes are compatible: `base_mu` and `base_log_sigma_diag` are
   both `(K,)`, so the `p.grad.shape == sigma.shape` gate at
   `optimizer.py:145` matches and the elementwise
   `p.grad = p.grad * sigma` broadcasts exactly to
   `∇_nat μ_k = σ²_k · ∇_E μ_k`.

### What is NOT done in this change

- No change to `create_param_groups` for any other parameter — only

- No change to the sigma or phi preconditioners — both verified
  correct.
- No regression test (user will verify PPL recovery independently).
- No rollback of Rounds 1–3.
- No touch of `RiemannianAdamW.step`, `_riemannian_clip`, or
  `NaturalGradientOptimizer`.
